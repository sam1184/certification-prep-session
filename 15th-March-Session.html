🧠 Session 2 Keyword Map — Memorise Before Question 1

Step 1 — Spot the database access pattern first.
Is it key-value? → DynamoDB. Document? → DocumentDB. Cassandra? → Keyspaces. Relational + managed? → RDS/Aurora. In-memory cache? → ElastiCache. The engine choice question is almost always hidden in the workload description, not stated directly.
Step 2 — For observability, ask: trace vs log vs metric vs compliance.
Distributed trace (why is my request slow?) → X-Ray. Config change (what changed?) → AWS Config. Operational event (is my service degraded?) → Health Dashboard. Time-series metrics at scale → Prometheus + Grafana. ONE wrong pick here costs you the question.
Step 3 — Eliminate by hard constraint, not by preference. "Variable workload, no capacity planning" → Aurora Serverless. "Millisecond cache, session data" → ElastiCache. "Lambda over 15 minutes" → impossible, use Fargate or Batch. These constraints are the question — everything else is noise.

The four Auto Scaling policy types — know which problem each solves:

Option A (move everything to Glacier Flexible immediately) eliminates Standard-IA for recent files. Option D (Glacier Instant Retrieval for 90+ days) costs more than Glacier Flexible — it provides millisecond retrieval that you don't need for 90+ day files. You're paying for speed you won't use. This is the canonical AWS three-tier VPC architecture. R1 — Public subnets: have a route 0.0.0.0/0 → IGW in their route table + instances have public IPs. Inbound internet traffic reaches them. R2 — Private + NAT subnets: route table has 0.0.0.0/0 → NAT Gateway. NAT translates outbound traffic; there is no path for inbound internet-initiated connections — R2 satisfied. R3 — Isolated private subnets: route table has only the local VPC CIDR route. No IGW, no NAT. Completely isolated from the internet in both directions — R3 satisfied.
Option B (all tiers in public subnets) is the most dangerous distractor. A Security Group CAN block inbound traffic — but the instances still have public IPs and the subnet still has a route to IGW. An attacker with a misconfigured SG or a new rule added by a developer can expose the database directly. "No internet connectivity whatsoever" requires removing the route, not just blocking the port. Option C (same private subnet for app + DB with NACLs) violates defence-in-depth — the app and DB tiers share network space. NACLs are stateless and more error-prone. Shared subnets for different security tiers is an anti-pattern. Option D (private subnets + IGW attached) is a common misconception. Attaching an IGW to the VPC is not enough — there must ALSO be a route to the IGW in the subnet route table AND a public IP for internet access. Private subnets with no route to IGW remain private even with IGW attached to the VPC.
Public vs Private subnet — the precise definition:
A public subnet = subnet whose route table has a route 0.0.0.0/0 → Internet Gateway. That's it. Not "has a public IP" — the route table is what matters.
A private subnet = subnet with NO route to IGW. It may have a NAT Gateway route for outbound-only, or no internet route at all.
NAT Gateway = outbound-only internet for private subnets. It is ALWAYS placed in a PUBLIC subnet. Traffic flows: Private instance → NAT (in public subnet) → IGW → internet. Internet cannot initiate back through NAT.
The exam tests this with: "outbound but not inbound" = NAT Gateway in a private subnet's route table.
Three-tier VPC in the real world: Every regulated AWS workload (banking, healthcare, government) uses this exact pattern. The database tier's isolated private subnet is the key control: even if an attacker compromises an app server, they cannot exfiltrate data via an outbound internet call from the database subnet — there is no route. This architectural isolation is a core requirement of PCI-DSS, HIPAA, and SOC 2 Type II.
