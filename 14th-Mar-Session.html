Amazon RDS
Managed Relational Database · MySQL · PostgreSQL · Oracle · SQL Server · MariaDB
Database HIGH EXAM WEIGHT

Amazon RDS (Relational Database Service) is AWS's managed SQL database service. Think of it as "we run the database engine for you." You get automated backups, patching, Multi-AZ failover, and read replicas — but you still pick the instance size and storage. RDS supports 6 engines: MySQL, PostgreSQL, MariaDB, Oracle, SQL Server, and Aurora (which we'll cover separately).

Multi-AZ
High Availability
Synchronous standby in different AZ — automatic failover in 1-2 min
Read Replicas
Scale Reads
Up to 15 async replicas — can be cross-region for DR
35 Days
Backup Retention
Automated daily snapshots + transaction logs for point-in-time recovery
🚨 #1 Exam Trap — Multi-AZ vs Read Replicas: Students constantly confuse these. Multi-AZ = High Availability (standby is NOT readable, only for failover). Read Replicas = Performance (readable copies for scaling reads). You can have both! Multi-AZ with 5 read replicas is valid.

RDS Engine Decision Matrix

EngineUse WhenExam Keyword
MySQL / MariaDBOpen-source, WordPress, general OLTP apps"cost-effective", "open-source"
PostgreSQLAdvanced features, JSON support, geospatial (PostGIS)"complex queries", "JSON"
Oracle / SQL ServerEnterprise apps that require these specific engines"license included", "Windows auth"
AuroraBest performance, cloud-native features (see Aurora section)"5x faster", "serverless", "global"
🏢 Real-World Use Case
E-Commerce Site — Black Friday Traffic Spike

An online retailer runs their product catalog on RDS MySQL Multi-AZ. During Black Friday, read traffic spikes 10x (everyone browsing products). They spin up 5 Read Replicas and point their read-heavy queries (product searches, recommendations) to the replicas. Write traffic (checkouts, orders) still hits the primary. When the primary instance accidentally fails at peak traffic, Multi-AZ automatically fails over to the standby in 90 seconds — users see a brief slowdown but no downtime. After the sale, they delete 4 read replicas to save cost.

🎯 SAA-C03 Exam Tips:
• "Minimize downtime for database maintenance" → RDS Multi-AZ (patches standby first, then failover)
• "Disaster recovery in another region" → Cross-region Read Replica (promote to master if primary region fails)
• "Scale read performance without changing app code" → Read Replicas
• "Restore database to 5 minutes ago" → Point-in-time restore (enabled by default, uses backups + transaction logs)
• RDS encrypts data at rest with KMS — can't enable encryption on existing unencrypted DB (must create encrypted snapshot and restore)
Amazon Aurora
Cloud-Native Relational DB · MySQL & PostgreSQL Compatible · 5x Performance
Database HIGH EXAM WEIGHT PERFORMANCE

Amazon Aurora is AWS's proprietary relational database engine built for the cloud. It's MySQL-compatible or PostgreSQL-compatible (you pick), but the storage layer is custom AWS magic: data is automatically replicated 6 copies across 3 AZs, self-healing, and can scale up to 128 TB. It's 5x faster than standard MySQL on RDS because of distributed storage and custom caching.

6 Copies
Storage Replication
2 copies per AZ across 3 AZs — survives loss of entire AZ + 1 copy
15 Replicas
Aurora Replicas
Low-latency (<10ms lag), auto-failover, shared storage (not copied like RDS)
128 TB
Max Storage
Auto-scaling storage in 10GB increments — no pre-provisioning
Aurora vs RDS — The Key Differences:
Storage: RDS uses EBS (tied to instance). Aurora uses custom distributed storage (shared across all replicas).
Replicas: RDS Read Replicas have their own storage copy (async replication). Aurora Replicas share the same storage (no replication lag for reads, only writes).
Failover: RDS Multi-AZ takes 1-2 min (DNS change). Aurora failover to replica takes <30 seconds (just promotes a replica).
Cost: Aurora is 20% more expensive than RDS MySQL/PostgreSQL, but often cheaper at scale due to less operational overhead.

🚀 Real-World Use Case
SaaS Platform — Multi-Tenant Database with Global DR

A SaaS company serves 10,000 enterprise customers from a single Aurora PostgreSQL cluster. They use Aurora Global Database with primary in us-east-1 and secondary in eu-west-1. European customers read from the eu-west-1 replica (low latency). If us-east-1 fails, they can promote eu-west-1 to primary in under 1 minute. They use Backtrack when a customer accidentally runs a bad SQL migration — instead of restoring from backup (30+ min), they backtrack to 5 minutes ago in 30 seconds. They create staging environments using Aurora Cloning — a perfect copy of 2TB production database is ready in 10 minutes without doubling storage costs.

🎯 SAA-C03 Exam Tips:
• "Best performance for MySQL/PostgreSQL" → Aurora (5x faster)
• "Disaster recovery in another region with <1 second lag" → Aurora Global Database
• "Undo accidental data deletion without restore" → Aurora Backtrack
• "Create test environment from production in minutes" → Aurora Cloning
• Aurora automatically fails over to a replica if primary fails — no Multi-AZ flag needed (built-in HA)
Amazon Aurora Serverless
Auto-Scaling Database · Pay-Per-Second · No Instance Management
Database Serverless Auto-Scaling

Aurora Serverless is Aurora without the "pick an instance size" step. Instead of db.r5.2xlarge, you specify min/max ACUs (Aurora Capacity Units). The database automatically scales up when load increases, scales down (even to 0) when idle, and you pay only for the capacity you use per second. There are two versions: v1 (older, more limitations) and v2 (current, recommended).

💡 Real-World Use Case
Startup Analytics Dashboard — Weekend Silence, Weekday Chaos

A startup runs an internal analytics dashboard used by 200 employees. Usage pattern: 8am-6pm weekdays = high traffic, nights/weekends = zero traffic. With provisioned Aurora, they'd pay for a db.r5.large 24/7 (~$200/month). With Aurora Serverless v1, they set min=0.5 ACU, max=16 ACU. During work hours, it scales to 8-12 ACUs. At night and weekends, it pauses completely — $0 cost. Monthly cost drops to ~$60. When a data science team runs a heavy query at 2am, it auto-resumes and scales up in seconds.

🚨 Exam Trap — "Scale to Zero" Language: The exam may ask "which database can scale to zero when not in use?" The answer is Aurora Serverless v1. However, if the question mentions "production workload" or "instant scaling", prefer Aurora Serverless v2 (even though it can't pause). Read carefully!
🎯 SAA-C03 Exam Tips:
• "Infrequent database access, minimize cost when idle" → Aurora Serverless v1 (scale to zero)
• "Variable production workload, avoid connection drops during scaling" → Aurora Serverless v2
• "Development database used only during business hours" → Aurora Serverless v1 (pauses at night)
• Serverless Aurora still provides all core Aurora features: 6-way replication, automated backups, Multi-AZ
Amazon DynamoDB
Serverless NoSQL · Key-Value & Document · Single-Digit Millisecond Latency
Database HIGH EXAM WEIGHT NoSQL

DynamoDB is AWS's fully managed NoSQL database. It's serverless (no servers to provision), scales automatically to any workload, and provides single-digit millisecond latency at any scale. Data is stored as items (like rows) in tables. Each item must have a partition key (and optionally a sort key). DynamoDB is ideal for key-value lookups, session stores, shopping carts, mobile backends, IoT, and gaming leaderboards.

🚨 #1 Exam Trap — DynamoDB is NOT a Relational Database: Students try to apply SQL thinking. DynamoDB has NO JOINS, NO SQL queries, NO foreign keys. If question says "complex JOIN queries across tables" or "ACID transactions across tables" → NOT DynamoDB → use Aurora/RDS. DynamoDB is for simple key-value lookups or range scans on a single table.
🎮 Real-World Use Case
Mobile Gaming — Player Profiles & Leaderboards

A mobile game has 50 million players. They store player profiles in DynamoDB: Partition Key = PlayerID. Lookups like "get player stats" are <5ms. For the global leaderboard, they use a Global Secondary Index with Partition Key = "LeaderboardName", Sort Key = "Score" — this lets them query "top 100 players" efficiently. During a viral event, traffic spikes 100x in 10 minutes. Because DynamoDB is in On-Demand mode, it auto-scales from 1,000 reads/sec to 100,000 reads/sec with zero manual intervention. They use DynamoDB Global Tables to replicate data across us-east-1, eu-west-1, ap-southeast-1 for low-latency access worldwide.

🎯 SAA-C03 Exam Tips:
• "NoSQL database with single-digit millisecond latency" → DynamoDB
• "Unpredictable spiky traffic, auto-scale without provisioning" → DynamoDB On-Demand
• "Query on non-key attribute" → Global Secondary Index (GSI)
• "Multi-region active-active database" → DynamoDB Global Tables
• "Session store for web app" → DynamoDB (or ElastiCache — DynamoDB for durability, ElastiCache for speed)
• DynamoDB Streams = change data capture — triggers Lambda on every insert/update/delete (exam loves this pattern)
Amazon DocumentDB
MongoDB-Compatible · Managed Document Database · JSON Storage
Database NoSQL Document

Amazon DocumentDB is AWS's MongoDB-compatible document database. If you have an app built on MongoDB and want to migrate to AWS without rewriting code, DocumentDB is the answer. It's NOT actual MongoDB — it's AWS's implementation of the MongoDB API. It supports the same query language, drivers, and tools, so existing MongoDB apps work with minimal changes.

MongoDB API
Compatibility
Drop-in replacement for MongoDB 3.6, 4.0, 5.0 — use existing drivers/tools
15 Replicas
Read Scaling
Like Aurora — shared storage, low-latency failover
When to Use DocumentDB vs DynamoDB:
DocumentDB: You need MongoDB. Your app already uses MongoDB. You have complex nested JSON documents. You query by multiple fields within documents.
DynamoDB: You need extreme scale (millions of requests/sec). You want serverless (no instance management). Your access patterns are simple key-value lookups.
📱 Real-World Use Case
Content Management System — Flexible Schema Migration

A media company runs a CMS storing articles as MongoDB documents. Each article has flexible schema: some have videos, some have image galleries, some have polls. They migrate from self-managed MongoDB on EC2 to DocumentDB. Migration is easy: they use AWS DMS (Database Migration Service) to replicate data from MongoDB to DocumentDB with near-zero downtime. They change the connection string in their app from MongoDB endpoint to DocumentDB endpoint — code works unchanged. They get automatic backups, Multi-AZ, and read replicas without managing MongoDB sharding/replication.

🎯 SAA-C03 Exam Tips:
• "Migrate MongoDB application to AWS" → DocumentDB
• "MongoDB-compatible managed database" → DocumentDB
• "Store JSON documents with flexible schema" → DocumentDB (or DynamoDB — depends on scale/access pattern)
• DocumentDB architecture mirrors Aurora: 6-way replicated storage, up to 15 read replicas, automatic failover
Amazon ElastiCache
In-Memory Cache · Redis & Memcached · Sub-Millisecond Latency
Database HIGH EXAM WEIGHT Performance

Amazon ElastiCache is AWS's managed in-memory caching service. It supports two engines: Redis and Memcached. Use it to cache frequently-accessed data (database query results, session data, API responses) in RAM for sub-millisecond latency. This dramatically reduces load on your database and speeds up your app.

🚨 Exam Trap — Redis vs Memcached: The exam will give you subtle hints. "Need persistence/backup" → Redis. "High availability/Multi-AZ" → Redis. "Simple object cache, multi-threaded" → Memcached. "Leaderboard / sorted set" → Redis (Memcached can't do sorted sets). "Session store that survives cache restart" → Redis (persists to disk).
⚡ Real-World Use Case
E-Commerce — Product Catalog Cache

An e-commerce site has 1 million products in RDS MySQL. Product detail pages are hit millions of times per day. Without caching, every page view = 1 database query = slow performance + high RDS load. They add ElastiCache for Redis. First request: app checks Redis, misses, queries RDS, writes result to Redis with 5-minute TTL. Next 10,000 requests in 5 minutes: app gets data from Redis in <1ms — zero RDS queries. During flash sale, traffic spikes 50x. RDS query load stays flat because 90% of reads are served from Redis. They use Redis Cluster mode for horizontal scaling (sharding) across multiple nodes. Because it's Redis (not Memcached), cache data persists to disk — if a node reboots, data isn't lost.

🎯 SAA-C03 Exam Tips:
• "Reduce database load with caching" → ElastiCache
• "Need sorted leaderboard / ranking" → ElastiCache for Redis (sorted sets)
• "Session store with Multi-AZ failover" → ElastiCache for Redis
• "Simple cache, no persistence needed" → ElastiCache for Memcached
• Caching pattern: Lazy Loading (cache on miss) vs Write-Through (cache on write) — know the tradeoffs
Amazon Keyspaces (for Apache Cassandra)
Cassandra-Compatible · Serverless · Wide-Column NoSQL
Database NoSQL Cassandra

Amazon Keyspaces is AWS's managed Apache Cassandra-compatible database service. If you have apps built on Cassandra (wide-column NoSQL database), Keyspaces lets you run them on AWS without managing Cassandra clusters. It's serverless, auto-scales, and uses the Cassandra Query Language (CQL). Think of it as "Cassandra without the operational nightmare."

CQL
Cassandra Compatible
Use existing Cassandra drivers, tools, and CQL queries — minimal code changes
Serverless

When to Use Keyspaces: You're migrating a Cassandra application to AWS. You need wide-column data model (partition key + clustering columns). You have time-series data at massive scale. You don't want to manage Cassandra ring topology, gossip protocol, compaction, repairs.
📡 Real-World Use Case
IoT Platform — Sensor Data at Scale

An IoT company collects data from 10 million sensors (temperature, humidity, pressure) every 30 seconds. They need to store time-series data and query by sensor ID + time range. They use Amazon Keyspaces with partition key = SensorID, clustering column = Timestamp. This allows efficient queries like "get all readings for sensor ABC from last 7 days." Because Keyspaces is serverless, they don't provision capacity — it auto-scales as sensor count grows from 10M to 50M. They migrate from self-managed Cassandra on EC2, reusing their existing CQL queries with zero code changes.

🎯 SAA-C03 Exam Tips:
• "Migrate Cassandra application to AWS" → Amazon Keyspaces
• "Wide-column NoSQL database" → Amazon Keyspaces
• "Time-series data at scale with CQL" → Amazon Keyspaces
• Keyspaces provides automatic Multi-AZ replication, encryption, backups — fully managed

AWS Lambda
Serverless Compute · Event-Driven · Pay-Per-Invocation
Compute CRITICAL EXAM SERVICE Serverless

AWS Lambda is the cornerstone of serverless computing on AWS. You upload your code (function), Lambda runs it in response to events (API call, file upload, database change, schedule), and you pay only for compute time consumed (billed per millisecond). No servers to manage, automatic scaling, built-in high availability. Lambda is the #1 integration point in AWS — it connects everything.

10 GB

🚨 #1 Exam Trap — Lambda Limitations: Lambda is NOT for everything. Cannot run >15 minutes → use ECS/Fargate or Step Functions. Cannot use GPU → use EC2. Needs persistent filesystem → use ECS with EFS. Needs always-on WebSocket server → use EC2/Fargate. Know when NOT to use Lambda!
🖼️ Real-World Use Case
Image Processing Pipeline — Thumbnail Generation

A photo-sharing app stores user uploads in S3. When a user uploads an image, S3 triggers a Lambda function. The function: (1) downloads the image from S3, (2) generates 3 thumbnail sizes using ImageMagick, (3) uploads thumbnails back to S3, (4) writes metadata to DynamoDB. This happens in <2 seconds for most images. No servers to manage. If 1,000 users upload photos simultaneously, Lambda scales to 1,000 concurrent executions automatically. During low-traffic periods, cost is near $0 (no idle servers). They use Lambda Layers to share the ImageMagick library across functions.

🎯 SAA-C03 Exam Tips:
• "Run code in response to S3 upload" → Lambda with S3 trigger
• "Process DynamoDB changes in real-time" → Lambda with DynamoDB Streams
• "Serverless API backend" → API Gateway + Lambda
• "Schedule function to run every hour" → EventBridge (CloudWatch Events) + Lambda
• Lambda execution role = IAM role that grants Lambda permissions to access AWS services (S3, DynamoDB, etc.)
• Lambda@Edge = run Lambda at CloudFront edge locations — manipulate requests/responses globally

You define CPU/memory requirements, Fargate provisions the infrastructure, runs the container, and you pay only for the resources your container uses.

🔄 Real-World Use Case
Batch Processing — Video Transcoding

A video platform receives uploaded videos in S3. Each video needs to be transcoded to 3 formats (1080p, 720p, 480p). The transcoding process takes 5-30 minutes per video (too long for Lambda's 15-min limit). They use Fargate with ECS. When S3 receives a video, it sends a message to SQS. An ECS Service running on Fargate polls SQS, picks up the message, spins up a container with FFmpeg, transcodes the video, uploads outputs to S3. Each container runs for 10-30 minutes, then terminates. They use Fargate Spot (70% cheaper) for non-urgent jobs. During peak upload hours, ECS auto-scales Fargate tasks from 5 to 50 to handle the queue. No EC2 instances to manage.

🎯 SAA-C03 Exam Tips:
• "Run containers without managing servers" → Fargate
• "Long-running containerized microservices" → Fargate with ECS
• "Use Kubernetes on AWS without managing nodes" → Fargate with EKS
• "Batch job that runs for 2 hours" → Fargate (Lambda maxes at 15 min)
• Fargate pricing = vCPU/hour + GB RAM/hour (more expensive than EC2 for always-on workloads, cheaper for sporadic)

The most common use is EC2 Auto Scaling (add/remove instances), but Auto Scaling also works for ECS tasks, DynamoDB tables, Aurora replicas, and more. The goal: maintain performance during peaks, save money during low traffic.

🚨 Exam Trap — Scale-Out vs Scale-Up: Scale-Out (horizontal) = add more instances (Auto Scaling). Scale-Up (vertical) = upgrade instance type (requires stop → change type → start — causes downtime). 📈 Real-World Use Case
News Website — Traffic Surges During Breaking News

A news site runs on EC2 behind an ALB. Typical traffic: 1,000 req/sec (handled by 5 m5.large instances). During breaking news (e.g., election results), traffic spikes to 50,000 req/sec. They configure EC2 Auto Scaling with Target Tracking policy: CPU 70%. When traffic surges, CPU hits 90% → Auto Scaling launches 45 more instances over 10 minutes → CPU drops back to 70%. When the news cycle calms down (3 hours later), CPU drops to 40% → Auto Scaling terminates excess instances down to 8. They use Scheduled Scaling to pre-warm 10 instances at 6am (before morning rush). This saved 65% on compute cost vs. running 50 instances 24/7.

🎯 SAA-C03 Exam Tips:
• "Automatically adjust EC2 capacity based on demand" → Auto Scaling Group (ASG)
• "Maintain CPU at 70%" → Target Tracking Scaling Policy
• "Scale at 8am every weekday" → Scheduled Scaling
• "Proactively scale before predicted traffic spike" → Predictive Scaling
• Auto Scaling works with Launch Templates (modern) or Launch Configurations (legacy)
• Health checks: EC2 status check or ELB health check — ASG replaces unhealthy instances automatically
• Cooldown period = time to wait after scaling action before next one (prevents thrashing)

📊 Block 3 — Monitoring & Observability

⏱️ Time: 00:55 – 01:20 5 Services · Visibility into your AWS environment
AWS X-Ray
Distributed Tracing · Performance Analysis · Debug Microservices
Monitoring DevOps

AWS X-Ray helps you trace requests as they flow through your distributed application. When a user hits your API, the request might touch API Gateway → Lambda → DynamoDB → Lambda → S3. X-Ray shows you the entire journey: which service took how long, where errors occurred, where latency spikes happened. It creates a service map (visual graph of your architecture) and trace timelines (waterfall of each request).

Service Map
Visual Architecture
Auto-generated graph of how your services call each other
Trace Analysis
Request Tracking
See every hop a request makes — pinpoint slow/failing components
How X-Ray Works: You instrument your code (add X-Ray SDK to Lambda, EC2 app, ECS task). When a request comes in, X-Ray assigns it a trace ID. As the request flows through services, each service reports segments (sub-parts of the trace) to X-Ray. X-Ray assembles these segments into a complete trace and displays timing, status, metadata.
🐛 Real-World Use Case
Debugging Slow API — Finding the Bottleneck

A company's checkout API suddenly slows from 200ms to 3 seconds. The flow: API Gateway → Lambda (validate cart) → DynamoDB (fetch prices) → Lambda (calculate total) → RDS (save order). Using X-Ray, they see the service map and traces. Discovery: DynamoDB query is taking 2.8 seconds (normally 20ms). They drill into X-Ray annotations — the query is doing a table scan (missing index). They add a GSI to DynamoDB → query drops to 15ms → API is fast again. Without X-Ray, they would've wasted days guessing which component was slow.

🎯 SAA-C03 Exam Tips:
• "Debug performance in microservices architecture" → X-Ray
• "Visualize request flow across Lambda, API Gateway, DynamoDB" → X-Ray Service Map
• "Find which service in a chain is causing latency" → X-Ray Trace Analysis
• X-Ray integrates with Lambda, ECS, Elastic Beanstalk, API Gateway, EC2 (requires X-Ray daemon)

🤖 Real-World Use Case
Automated Backup Script — Daily S3 Sync

A company needs to back up their application logs from 50 EC2 instances to S3 every night. They write a bash script using AWS CLI:

#!/bin/bash
# Compress logs
tar -czf /tmp/logs-$(date +%Y%m%d).tar.gz /var/log/app/
# Upload to S3
aws s3 cp /tmp/logs-$(date +%Y%m%d).tar.gz s3://company-backups/logs/
# Delete local copy
rm /tmp/logs-$(date +%Y%m%d).tar.gz
They schedule this script with cron to run at 2am daily. The EC2 instances have an IAM Role with S3 write permissions — no access keys needed. This saves 10 hours/week vs. manual backups. They also use CLI in their CI/CD pipeline to deploy Lambda functions: aws lambda update-function-code --function-name myFunc --zip-file fileb://code.zip
