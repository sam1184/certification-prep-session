🧠 Exam Tips & Tricks — What the Top Scorers Know

📖 How to Read an SAA-C03 Question — 3-Step Attack

The SAA-C03 exam is NOT a knowledge recall test — it's a constraint-matching test. Every question hides 2–3 constraints that eliminate wrong answers. Learn to extract them before touching the options.

The 3-Step Attack:
  1. Step 1 — Find the HARD constraints. Keywords like "without downtime," "no code changes," "most cost-effective," "must remain private," "cannot leave the data center." These eliminate 2–3 options immediately.
  2. Step 2 — Find the SOFT requirements. Scale, latency, protocol (NFS vs SMB vs iSCSI), OS (Linux vs Windows), managed vs self-managed.
  3. Step 3 — Eliminate, don't guess. Cross out options that violate any single constraint. Usually one option survives cleanly.
🚨 Keywords That Signal the Correct Service:
"overlapping CIDRs" → PrivateLink (not Peering)
"static IP global" → Global Accelerator (not CloudFront)
"more than 15 minutes" → AWS Batch (not Lambda)
"block specific IP" → NACL (not Security Group)
"shared Linux NFS" → EFS (not EBS)
"shared Windows SMB" → FSx for Windows
"data must not leave premises" → Outposts or EKS Anywhere
"5G ultra-low latency" → Wavelength
"no DevOps / no infra code" → Elastic Beanstalk
"immediately visible / real-time shared" → EFS (not S3)

🎯 Exam Day — 10 Rules That Improve Your Score

RULE 1 — Read the last sentence of the question first. That's where the requirement lives ("Which is most cost-effective?").
RULE 2 — Eliminate options that violate a hard constraint before comparing remaining ones. Don't debate two valid options — find which one is actually invalid.
RULE 3 — "Managed" beats "self-managed" unless the question explicitly requires self-management. AWS fully managed = less operational overhead = correct answer.
RULE 4 — "Most cost-effective" and "least operational overhead" are often the SAME answer — fully managed serverless services tick both boxes simultaneously.
RULE 5 — If an option involves writing custom Lambda/code to replicate something a native AWS feature already does — it's wrong. AWS rewards built-in feature usage over custom code.
RULE 6 — Multi-select ("select TWO"): eliminate clearly wrong options first. No partial credit — both must be correct. If you only see one obvious answer, re-read the constraints.
RULE 7 — The correct answer is usually the most specific service for the job. "S3 Lifecycle Policy" beats "Lambda to move files." "AWS Batch" beats "ECS + custom scheduler."
RULE 8 — Never choose an option that creates a single point of failure unless the question explicitly trades reliability for cost. HA is the default expectation on this exam.
RULE 9 — When two options "both work" — pick the one with fewer moving parts, less custom code, and more AWS-native automation. Simplicity is a design value on the exam.
RULE 10 — Don't change your first answer unless you find a specific, concrete reason. Gut instinct built through practice is more reliable than second-guessing under exam pressure.

Question 01
Networking Architecture HIGH WEIGHT

A financial services company operates 15 VPCs across three AWS accounts — one for each business unit — and maintains a connection to its on-premises data center via AWS Direct Connect. The network team reports that as new VPCs are added, the number of peering connections is becoming unmanageable and on-premises routing tables are growing exponentially. A Solutions Architect must redesign the network so that all VPCs and the on-premises network can communicate with each other through a single, centrally-managed hub that scales without manual peering between every VPC pair.

Which solution meets these requirements with the least operational overhead?

  • A Create VPC peering connections between every VPC pair and advertise all CIDR ranges through the Direct Connect connection using Border Gateway Protocol (BGP).
  • B Deploy an AWS Transit Gateway, attach all 15 VPCs and the Direct Connect gateway to it, and configure route tables on the Transit Gateway to control inter-VPC and on-premises traffic.
  • C Enable VPC sharing using AWS Resource Access Manager (RAM) to share subnets across accounts and use a single shared VPC for all business units.
  • D Create a hub-and-spoke model using AWS PrivateLink endpoint services. Expose each VPC's resources through NLB-backed endpoint services and have other VPCs connect using Interface VPC Endpoints.
Correct Answer: B
AWS Transit Gateway is purpose-built for this architecture. A single TGW attachment point per VPC — regardless of how many VPCs exist — replaces O(N²) peering connections with O(N) attachments. TGW natively integrates with Direct Connect Gateway for on-premises connectivity, and route tables on TGW control exactly which VPCs can talk to each other. You can even share the TGW across accounts using AWS RAM, making this the canonical enterprise hub-and-spoke solution on AWS.
Option A (VPC Peering) is the classic trap — peering is non-transitive, so 15 VPCs need 105 individual peering connections, and on-premises routing explodes. Option D (PrivateLink) exposes services, not full network routing — it doesn't solve inter-VPC full-mesh communication.
Question 02
Networking Security

A SaaS company provides a payroll processing API to enterprise customers. Each customer operates their own AWS VPC. The SaaS company wants to expose the API privately — without sending traffic over the internet — to customer VPCs. The VPC CIDR ranges of customers often overlap with the SaaS company's VPC CIDR. The security team requires that customers can only access the API endpoint and cannot initiate any other communication into the SaaS VPC.

Which solution satisfies all requirements?

  • A Set up VPC peering between the SaaS company's VPC and each customer VPC. Add routes in both VPCs' route tables to direct API traffic through the peering connection.
  • B Create a public API Gateway endpoint fronting the payroll API. Provide customers with the API Gateway URL and require API key authentication.
  • C Create a VPC Endpoint Service using AWS PrivateLink backed by a Network Load Balancer in the SaaS VPC. Allow customers to create Interface VPC Endpoints in their own VPCs to connect to the service.
  • D Attach all customer VPCs to an AWS Transit Gateway along with the SaaS VPC and configure Transit Gateway route tables to allow only port 443 traffic to the API servers.
Correct Answer: C
AWS PrivateLink is the exact solution for exposing services privately across VPCs — even with overlapping CIDRs. The SaaS company creates a VPC Endpoint Service (backed by an NLB); customers create Interface VPC Endpoints in their own VPCs. Traffic flows entirely within the AWS network. Critically, PrivateLink is unidirectional — customers can only reach the endpoint service, not anything else in the SaaS VPC. The SaaS company controls which customer accounts are whitelisted.
Option A (VPC Peering) fails immediately — VPC peering does NOT work when CIDR ranges overlap. This is the most common PrivateLink exam trigger. Option D (Transit Gateway) gives full network access, violating the "only API endpoint" security requirement.
Question 03
Edge / 5G Latency

A startup is building a real-time multiplayer mobile game that is being launched in partnership with a major 5G carrier. The game requires that game state updates reach players' mobile devices in under 10 milliseconds. Testing shows that deploying the game servers in the nearest AWS Region adds 35–60ms of latency — which causes an unacceptable player experience. The engineering team wants to keep the game servers inside the AWS ecosystem (same APIs, IAM, monitoring) but must bring compute physically closer to the 5G radio towers.

Which AWS service should the Solutions Architect recommend?

  • A Deploy EC2 instances in the AWS Region with the shortest geographic distance to the target city and use Route 53 Latency-Based Routing to direct players to the nearest region.
  • B Use AWS Global Accelerator to onboard traffic at the nearest AWS edge location and route it over the AWS backbone to the Region for processing.
  • C Deploy the game servers in an AWS Wavelength Zone embedded inside the 5G carrier's network infrastructure so that game traffic never leaves the carrier's backbone before reaching compute.
  • D Install AWS Outposts racks at co-location facilities near the 5G towers and run game servers on the Outposts hardware, connected back to the parent AWS Region.
Correct Answer: C
AWS Wavelength places AWS compute directly inside carrier data centers at the edge of the 5G network. When a player sends data via 5G, it travels from the radio tower into the carrier's backbone and hits the Wavelength Zone compute without ever traversing the public internet or routing to an AWS Region. This achieves single-digit millisecond latency. Wavelength instances use the same EC2 APIs, IAM, and CloudWatch as regular regions.
Option B (Global Accelerator) reduces latency from internet jitter but still routes to an AWS Region — the fundamental 35–60ms problem remains. Option D (Outposts) brings AWS compute on-premises but doesn't integrate with the carrier's 5G backbone the way Wavelength does. The key discriminator: Wavelength = inside carrier 5G network; Outposts = inside YOUR facility.
Question 04
DNS High Availability HIGH WEIGHT

A global retail company has its primary application stack deployed in us-east-1. The company recently experienced a regional outage and lost 45 minutes of revenue. The disaster recovery team is now building a warm standby environment in eu-west-1. The DNS solution must automatically detect the primary region's health every 30 seconds and redirect all traffic to the EU region within 60 seconds of failure, with no manual intervention. The application load balancer endpoints in each region have different DNS names.

Which DNS and routing configuration achieves this?

  • A Create two Route 53 Alias records — one pointing to each ALB. Assign equal weight to both records so traffic is split 50/50 in normal operations and the healthy one continues when one fails.
  • B Create a Route 53 Failover routing policy with an active Primary record pointing to the us-east-1 ALB and a Secondary record pointing to the eu-west-1 ALB. Attach a Route 53 Health Check to the primary record that evaluates the us-east-1 ALB endpoint every 30 seconds.
  • C Use AWS Global Accelerator with two endpoint groups — one per region — and configure endpoint weights so that all traffic goes to us-east-1. Enable health checks on both endpoint groups to trigger automatic failover.
  • D Create Route 53 Latency-Based routing records pointing to both ALBs. Since us-east-1 is geographically closer to most customers, they will be routed there by default and automatically rerouted if latency increases due to an outage.
Correct Answer: B
Route 53 Failover routing policy with active/passive configuration and a Health Check is the exact pattern for this DR scenario. The Health Check polls the primary endpoint every 30 seconds. When the endpoint fails the configured health threshold, Route 53 automatically serves the secondary record. Failover is DNS-based, so the 60-second window is achievable (30s health check + DNS TTL propagation). Alias records to ALBs are required since CNAME at root domain isn't supported — use Alias records.
Option A (Weighted) does NOT perform health-check-driven failover — it still sends 50% to a failed region. Option D (Latency-Based) routes based on measured latency; an outage doesn't necessarily show higher latency immediately — health checks are missing. Note: Option C (Global Accelerator) ALSO works technically but the question specifically asks about DNS routing configuration, which points to Route 53.
Question 05
Networking Security Multi-Select
⚠️ SELECT TWO — Multiple answers required

A company has a three-tier web application running in a VPC. The web tier sits in public subnets, the application tier in private subnets, and the database tier in isolated subnets. The security team performs an audit and identifies two problems: (1) EC2 instances in the private app tier can communicate with each other on ALL ports, which violates least privilege. (2) A specific competitor's IP address range (203.0.113.0/26) has been scraping the web tier. The team needs to block the competitor's IPs and restrict inter-application communication at the subnet level.

Which two changes address both findings?

  • A Modify the Security Group on the application tier EC2 instances to allow only specific required ports from other application tier instances using the Security Group ID as source.
  • B Add an explicit DENY rule for 203.0.113.0/26 on the inbound rules of the Security Group attached to the web tier EC2 instances.
  • C Update the Network ACL (NACL) associated with the private app subnets to allow only the specific required ports between app tier instances, removing the implicit all-allow behavior.
  • D Add an inbound DENY rule for 203.0.113.0/26 to the Network ACL associated with the public subnets hosting the web tier, with a lower rule number than the existing ALLOW rules.
Correct Answers: A and D
A: Security Groups support referencing another Security Group as the source — this is the cleanest way to restrict inter-tier traffic to only specific ports. Only instances sharing that Security Group are allowed, on only the ports configured.

D: Network ACLs are the only VPC construct that supports explicit DENY rules. Security Groups are allow-only — you cannot add a deny rule to a Security Group. Adding a DENY for the competitor IP range at the NACL level (with a lower rule number so it's evaluated first) blocks it at the subnet boundary before it reaches any EC2 instance.
Option B is the classic exam trap — Security Groups DO NOT support DENY rules. You cannot block a specific IP using a Security Group. Always use NACLs for IP blocking. Option C confuses the tool — NACLs control traffic entering and exiting subnets; intra-subnet traffic between instances in the SAME subnet does NOT traverse the NACL. Use Security Groups for that.

Question 06
CDN Security HIGH WEIGHT

A media company stores all of its video assets in an Amazon S3 bucket. The bucket is currently private. The company wants to distribute these videos globally with low latency using a CDN. The security team requires that users can never access S3 directly — all requests must flow through the CDN, and the S3 bucket must remain private with no public access. The company also wants to ensure that the CDN can serve content even if a user guesses the S3 bucket URL.

Which configuration achieves this?

  • A Enable S3 static website hosting, make the bucket public, and set the CloudFront origin to the S3 website endpoint so CloudFront can cache the content.
  • B Create a CloudFront distribution with the S3 bucket as the origin. Configure an Origin Access Control (OAC) on the distribution and update the S3 bucket policy to allow s3:GetObject only from the CloudFront service principal.
  • C Create a CloudFront distribution with the S3 bucket as origin. Enable S3 Transfer Acceleration on the bucket so global users get faster uploads, and set a bucket policy to only allow requests from the CloudFront IP range list.
  • D Use AWS Global Accelerator with the S3 bucket as an endpoint group. This routes users to the nearest AWS edge node and then to S3, keeping the bucket private because Global Accelerator acts as a proxy.
Correct Answer: B
CloudFront + Origin Access Control (OAC) is the current best practice for restricting S3 bucket access to CloudFront only. OAC (which replaced the older OAI) allows CloudFront to sign requests to S3 using SigV4. The S3 bucket policy then explicitly denies all principals EXCEPT the CloudFront service principal — so even if someone discovers the S3 bucket URL, their direct request is denied by the bucket policy.
Option A defeats the entire purpose — making the bucket public means S3 is directly accessible. Option C is wrong because CloudFront IP ranges change dynamically and AWS publishes them but maintaining IP-based policies is NOT recommended — it breaks. Option D (Global Accelerator) is an Anycast TCP/UDP accelerator — it does not proxy S3 in the same way CloudFront does, and does not keep S3 private.
Question 07
Load Balancing Containers

A company is running a microservices application on Amazon ECS. The application has four distinct services: an authentication service (/auth/*), a product catalog (/products/*), a shopping cart (/cart/*), and a checkout service (/checkout/*). The team wants to use a single load balancer with a single DNS name to front all four services, routing requests to the correct ECS service based on the URL path. The checkout service handles credit card data and requires its own separate SSL certificate for PCI compliance on a dedicated subdomain.

Which load balancer configuration is correct?

  • A Use a Network Load Balancer with four target groups — one per service. Configure port-based routing rules to direct traffic from different ports to each target group.
  • B Use an Application Load Balancer with listener rules that match path patterns (/auth/*, /products/*, etc.) and route to separate target groups. Add a second listener or use SNI to attach the PCI SSL certificate for the checkout subdomain.
  • C Use a Classic Load Balancer and configure health checks per service. Create separate DNS CNAME records pointing to the same CLB for each service path.
  • D Use an Application Load Balancer for the three non-PCI services and a separate Network Load Balancer for the checkout service. Use Route 53 to route traffic between the two load balancers.
Correct Answer: B
Application Load Balancer (ALB) operates at Layer 7 and natively supports path-based routing rules — directing different URL paths to different target groups with a single load balancer. ALB supports Server Name Indication (SNI), allowing multiple SSL certificates on a single listener — so the checkout subdomain gets its own PCI certificate without requiring a separate load balancer. ALB integrates natively with ECS, allowing dynamic port mapping for containers.
Option A (NLB) is wrong — NLB operates at Layer 4 (TCP/UDP). It has no concept of HTTP paths or URL inspection; it cannot do path-based routing. Option C (CLB) is legacy and does not support path-based routing. Option D uses two load balancers unnecessarily — ALB's SNI support handles multiple certificates on one LB.
Question 08
Global Accelerator Performance

A global trading platform serves institutional clients from offices in Tokyo, Frankfurt, London, and New York. The application is deployed in two AWS Regions (us-east-1, ap-northeast-1). The compliance team requires that the platform's IP addresses are static and fixed — clients whitelist specific IPs in their corporate firewalls and cannot tolerate IP changes. The platform also needs sub-second automatic failover between regions if one becomes unhealthy, without relying on DNS TTL expiry. Currently the platform uses Route 53 with a 60-second TTL failover, which the team finds too slow.

Which service should replace or supplement the current DNS-based approach?

  • A Replace Route 53 failover with Route 53 health checks using a 10-second interval and reduce the TTL on records to 10 seconds to achieve faster failover.
  • B Deploy AWS Global Accelerator with two endpoint groups (one per region). Global Accelerator provides two static Anycast IP addresses. Health checks run continuously; if a region becomes unhealthy, traffic is rerouted at the network layer in under 30 seconds — no DNS propagation required.
  • C Use Amazon CloudFront with two origins — one per region — configured with origin failover. CloudFront's edge network provides static IPs and automatic failover.
  • D Configure an Elastic IP address in each region and advertise them through BGP using a third-party global load balancer appliance running on EC2.
Correct Answer: B
AWS Global Accelerator is specifically designed for this scenario — two requirements uniquely identify it: static IP addresses and non-DNS-based failover. Global Accelerator provides two static Anycast IP addresses that are globally routable. Traffic enters the AWS network at the nearest edge POP and routes over the AWS private backbone. Health checks run every 10 seconds; failover happens at the network level without requiring DNS changes or TTL expiry. Clients never need to update their firewall whitelists.
Option A is wrong — reducing TTL to 10 seconds helps but DNS still requires client-side cache expiry, and some corporate clients override TTL. Critically, Route 53 cannot provide static IPs. Option C is wrong — CloudFront IP addresses are NOT static; they are managed by AWS and change. CloudFront is a CDN, not a TCP-level accelerator for trading APIs.
Question 09
Auto Scaling Compute HIGH WEIGHT

A ride-sharing platform experiences highly predictable traffic patterns: usage spikes every weekday morning between 7–9 AM and evening between 5–7 PM, and drops significantly overnight. The application runs on an EC2 Auto Scaling group behind an Application Load Balancer. Currently the group is configured with a fixed minimum of 50 instances to handle peak load — resulting in 50 idle instances overnight. The engineering team wants to reduce overnight costs by 80% while still meeting peak capacity without impacting user experience.

Which Auto Scaling strategy achieves this?

  • A Configure Target Tracking Scaling based on ALB RequestCountPerTarget metric. Set the target to the expected request count at peak, so Auto Scaling will add instances when traffic increases.
  • B Configure Scheduled Scaling actions to increase the minimum and desired capacity before the 7 AM and 5 PM peaks, and reduce them overnight. Optionally layer a Target Tracking policy on top to handle unexpected traffic spikes.
  • C Enable Predictive Scaling on the Auto Scaling group, which will analyze EC2 usage history and automatically pre-scale capacity using machine learning.
  • D Create an AWS Lambda function triggered by CloudWatch Events at 6:45 AM and 4:45 PM to call the Auto Scaling API and update the desired capacity, and another at 9:01 AM and 7:01 PM to scale back down.
Correct Answer: B
Scheduled Scaling is the optimal primary strategy when traffic patterns are known and predictable (specific times). By scheduling capacity increases 15 minutes before peak (warm-up time for EC2 instances to launch and pass health checks), you ensure readiness. Setting minimum to 10 instances overnight vs. 50 during peak directly achieves the ~80% cost reduction. Layering a Target Tracking policy on top handles unexpected spikes that fall outside the scheduled windows — this is a defense-in-depth approach.
Option A (Target Tracking alone) is reactive — it responds after traffic increases, causing a lag where users experience degraded performance while instances spin up. For truly predictable patterns, Scheduled Scaling is superior. Option C (Predictive Scaling) also works well and is newer, but the question specifies "without impacting user experience" and "predictable patterns" — Scheduled Scaling is the most direct and controlled answer. Option D is an anti-pattern — using Lambda for this is reinventing what Scheduled Scaling already does natively.

Question 10
S3 Cost Optimization HIGH WEIGHT

A healthcare company stores medical imaging files (DICOM) in Amazon S3. Files are accessed frequently in the first 30 days after creation (doctors review recent scans). Between days 30 and 90, they are rarely accessed. After 90 days, they must be archived for regulatory compliance for 7 years and must remain retrievable, but retrieval within 12 hours is acceptable. After 7 years, they must be permanently deleted. The company wants to automate this entire lifecycle with minimum cost.

Which S3 configuration implements this correctly?

  • A Store objects in S3 Standard. Write a Lambda function that runs nightly, checks object creation dates, and moves files to S3 Standard-IA at 30 days, then deletes them at 90 days.
  • B Configure an S3 Lifecycle Policy: transition to S3 Standard-IA at 30 days, transition to S3 Glacier Flexible Retrieval at 90 days, and add an expiration action to permanently delete objects after 7 years (2,555 days).
  • C Store all files in S3 Intelligent-Tiering from day one. Intelligent-Tiering will automatically optimize costs as access patterns change, and no lifecycle policy is needed.
  • D Store in S3 Standard. Transition to S3 Glacier Deep Archive at 30 days for lowest cost. Configure a 7-year expiration rule to delete objects automatically.
Correct Answer: B
S3 Lifecycle Policies automate storage tier transitions without any code. The chain: S3 Standard (0–30 days, frequent access) → S3 Standard-IA (30–90 days, infrequent access, lower storage cost) → S3 Glacier Flexible Retrieval (90 days–7 years, archival, 1–12hr retrieval which meets the ≤12hr requirement) → Expiration action deletes at 2,555 days. All fully automated, no Lambda needed, and each tier is cheaper than the last.
Option D (Glacier Deep Archive at 30 days) is wrong — Deep Archive has a minimum 12-hour retrieval time AND a 12-hour delay. More importantly, moving to Glacier at 30 days misses the 30–90 day infrequent (not zero) access window, where Standard-IA is correct. Option C (Intelligent-Tiering) doesn't support automatic permanent deletion via lifecycle rules in the same integrated way, and the question specifies a known access pattern — Intelligent-Tiering is for unknown/changing patterns.
Question 11
Block Storage Database

A company runs a PostgreSQL database on an EC2 instance using a gp2 EBS volume. The database currently handles 3,000 IOPS. The DBAs report that during peak periods, I/O wait times spike, indicating that the volume is being throttled. The team's investigation shows the application needs sustained 16,000 IOPS at consistent low latency. They also want the ability to increase IOPS without stopping or restarting the EC2 instance.

Which EBS volume type and change approach should the Solutions Architect recommend?

  • A Create a new io1 EBS volume with 16,000 provisioned IOPS. Detach the existing gp2 volume, attach the new io1 volume to the EC2 instance, and restore the database from a snapshot.
  • B Use the Amazon EC2 console or CLI to modify the existing gp2 volume to gp3 and configure the provisioned IOPS to 16,000 and throughput to 1,000 MiB/s. The modification happens in-place with no downtime.
  • C Add a second gp2 EBS volume to the instance and configure software RAID-0 (striping) across both volumes to double the available IOPS.
  • D Replace the EBS-backed EC2 instance with an instance store (ephemeral storage) for maximum IOPS, accepting the data loss risk on instance termination.
Correct Answer: B
gp3 EBS volumes support up to 16,000 IOPS and 1,000 MiB/s throughput — independently configurable from volume size. Crucially, Amazon EBS supports elastic modification: you can change the volume type, IOPS, and throughput of an in-use, attached volume with no downtime required. gp3 is also approximately 20% cheaper than gp2 per GB. For databases needing 16,000 IOPS, gp3 is the recommended modern standard; io1/io2 is for workloads needing >16,000 IOPS.
Option A (detach/restore) causes unnecessary downtime — EBS Elastic Volumes allow in-place modification. Option D (instance store) is a critical trap for database workloads — instance store data is LOST when the instance stops, hibernates, or fails. NEVER use instance store for databases unless explicitly required for temp/cache workloads.
Question 12
Shared Storage Linux

A data science team runs a machine learning training cluster with 200 EC2 Linux instances across three Availability Zones. The training job requires all instances to read from a shared dataset simultaneously. The dataset is 4 TB and grows by approximately 50 GB per week. Instances frequently add new training data files that other instances must immediately see without any synchronization mechanism. The solution must scale automatically without pre-provisioning storage capacity, and it must be mountable using standard Linux file system tools (mount, POSIX-compliant).

Which storage solution is architecturally correct?

  • A Provision a 5 TB EBS io2 volume and attach it to a single "master" EC2 instance. Have the other 199 instances access the data via NFS exported from the master instance.
  • B Use Amazon EFS (Elastic File System) in the Bursting throughput mode, mount it on all 200 EC2 instances using the NFS v4.1 protocol. EFS automatically scales capacity as data grows.
  • C Store the dataset in Amazon S3 and have each EC2 instance download the required files at job start using the AWS CLI or boto3. Use an S3 Lifecycle Policy to manage growing data.
  • D Use Amazon FSx for Lustre configured with an S3 data repository. Mount it on all EC2 instances; FSx for Lustre delivers high-throughput shared access and links directly to S3 for data ingestion and export.
Correct Answer: B
Amazon EFS is the purpose-built AWS service for shared Linux file storage across multiple EC2 instances. It: auto-scales with no capacity provisioning, supports POSIX-compliant NFS v4.1 mount across thousands of instances simultaneously, spans multiple AZs, and provides consistent access to all instances. When one instance writes a file, it is immediately visible to all others — no synchronization needed.

Note: Option D (FSx for Lustre) is also technically valid and would deliver significantly higher throughput for HPC/ML — the question does NOT emphasize throughput requirements, making EFS the cleaner match. In a real exam, if the question mentions "high performance" or "millions of IOPS" → FSx for Lustre. If it just says "shared concurrent access, POSIX, Linux" → EFS.
Option A (EBS via NFS from master) introduces a single point of failure — if the master instance fails, all 199 workers lose storage. Option C (S3 download at start) is not a shared file system — it's object storage requiring API calls. Files added by one instance are NOT immediately visible as file system paths to others without re-downloading.
Question 13
Storage Gateway Hybrid

A manufacturing company has a legacy ERP system running on-premises that writes files to a local NFS share. The company wants to migrate this storage to AWS without modifying the ERP application (which cannot be changed — it only speaks NFS). They want frequently-accessed files to remain cached on-premises for low-latency reads, while the data of record is stored in Amazon S3. The IT team wants a managed solution that doesn't require them to build and maintain a custom sync mechanism.

Which AWS solution meets these requirements?

  • A Deploy an AWS DataSync agent on-premises. Configure a task to continuously sync the NFS share to an Amazon EFS file system in AWS. The ERP continues writing to the on-premises NFS share.
  • B Deploy AWS Storage Gateway as a File Gateway on-premises. Present a local NFS mount point to the ERP system. The File Gateway stores all data in Amazon S3 and maintains a local cache of frequently accessed files.
  • C Deploy AWS Storage Gateway as a Volume Gateway in stored mode. Present an iSCSI volume to the ERP system. The full dataset is stored on-premises and asynchronously backed up to S3 as EBS snapshots.
  • D Install the Amazon S3 File System (s3fs) FUSE driver on the on-premises server and mount the S3 bucket as a local NFS share. The ERP writes directly to S3 through the FUSE mount.
Correct Answer: B
AWS Storage Gateway — File Gateway is the exact fit: it presents an NFS (or SMB) interface to on-premises applications, stores the data in S3 as objects (the data of record is in S3), and maintains a local cache on-premises for low-latency access to frequently-used files. The ERP application continues writing to NFS — zero code changes required. The gateway handles all sync to S3 automatically, managed by AWS.
Option A (DataSync) is for one-time or scheduled data migrations — not for ongoing application storage with a local cache. It doesn't replace a file share. Option C (Volume Gateway stored mode) keeps the primary data on-premises (full volume) and backs up to S3 — the data of record remains on-prem, NOT in S3. This is for backup, not for cloud-native storage migration. Option D (s3fs) is community software, not an AWS service, not managed, and not recommended for production.
Question 14
S3 Glacier Compliance

A financial services firm is required by regulation to retain audit log files for a minimum of 5 years. These files must be stored in a way that makes them immutable — no one, including the AWS account root user, must be able to delete or overwrite them during the retention period. The CISO wants assurance that even if the AWS account is compromised, the audit logs are safe. Cost must be minimized since the logs are never accessed once archived.

Which combination of services and features satisfies the compliance requirement?

  • A Store audit logs in Amazon S3. Enable S3 Versioning and configure a bucket policy that denies s3:DeleteObject and s3:PutObject for all users including root.
  • B Store audit logs in S3 Glacier, then configure a Glacier Vault Lock policy with a WORM compliance rule for 5 years (1,825 days). Once the Vault Lock policy is locked, it cannot be changed or deleted even by root.
  • C Store audit logs in Amazon S3 with Object Lock in Compliance Mode with a 5-year retention period. Transition objects to S3 Glacier Instant Retrieval immediately using a Lifecycle Policy for cost savings.
  • D Use AWS CloudTrail with log file integrity validation enabled. Store CloudTrail logs in a dedicated S3 bucket with MFA delete enabled.
Correct Answer: C
S3 Object Lock in Compliance Mode with a 5-year retention period creates a legally binding WORM (Write Once, Read Many) lock at the object level. In Compliance Mode — unlike Governance Mode — the retention period cannot be shortened or overridden by ANY user including the AWS account root user. Transitioning to S3 Glacier Instant Retrieval (or Flexible Retrieval) via Lifecycle Policy immediately reduces storage cost while maintaining the Object Lock protection. S3 Object Lock works correctly with S3 Glacier storage classes.

Note: Option B (Vault Lock) is also a valid WORM mechanism for Glacier specifically — it is similarly unbreakable once locked. If the question said "S3 Glacier only with no S3 Standard," Vault Lock would be the answer. Here, the combined Object Lock + Glacier Lifecycle approach gives cost optimization flexibility.
Option A (Bucket Policy deny) is NOT immutable — the AWS account root user can always modify or delete bucket policies, even if the policy includes a root deny. Root user is not constrained by resource-based policies. Object Lock Compliance Mode is the only mechanism that constrains root. Option D (MFA delete) protects against accidental deletion but can still be bypassed by the root user.

Question 15
Containers Batch Processing

A pharmaceutical company runs drug simulation jobs using Docker containers. Each simulation takes between 4 and 48 hours to complete, depending on the molecule complexity. The company processes approximately 500 simulations per month. The infrastructure must automatically provision compute when jobs are submitted, use Spot Instances to minimize cost, and automatically retry any job interrupted by Spot reclamation — without requiring DevOps teams to build or manage custom job schedulers.

Which AWS service is the best fit?

  • A Use AWS Lambda with a 15-minute maximum duration. For jobs exceeding 15 minutes, chain multiple Lambda functions using AWS Step Functions and pass simulation state between invocations.
  • B Deploy an Amazon ECS cluster with a managed Spot capacity provider. Write a custom Python scheduler that submits ECS tasks, monitors for Spot interruption notices, and re-submits failed tasks.
  • C Use AWS Batch with a managed Spot compute environment. Define job definitions using the Docker container images. Submit jobs to a job queue; AWS Batch handles provisioning, scheduling, retries on Spot interruption, and termination when complete.
  • D Use AWS Elastic Beanstalk with a worker tier. Upload the simulation container as a Beanstalk application, configure auto-scaling, and use SQS to queue simulation jobs.
Correct Answer: C
AWS Batch is purpose-built for this exact scenario. It manages: job queuing, compute environment provisioning (EC2 Spot), scheduling, automatic retries when Spot instances are reclaimed, and compute termination when the queue is empty. Job Definitions wrap Docker containers. There is no time limit on AWS Batch jobs (unlike Lambda's 15-minute maximum). AWS Batch supports array jobs for parallelism and cost-effectively scales to zero between bursts. No custom scheduler required.
Option A (Lambda) is the top distractor — Lambda maximum duration is 15 minutes. Jobs taking 4–48 hours CANNOT run on Lambda under any circumstances. Chaining Step Functions doesn't overcome this for a single continuous computation. This is the #1 exam trap for long-running batch workloads. Option B (ECS + custom scheduler) technically works but defeats the "without custom scheduler" requirement — that's exactly what AWS Batch provides.
Question 16
PaaS Deployment

A three-person startup is building a Python Django web application. The founders have strong software development skills but no infrastructure or DevOps experience. They need to: deploy the application to AWS, configure auto-scaling, set up a load balancer, and establish a CI/CD process — but they want to write as little infrastructure code as possible. The application uses RDS PostgreSQL as the database. A zero-downtime deployment mechanism is also required for future updates.

Which AWS service is the most appropriate platform for this team?

  • A Write AWS CloudFormation templates to define the EC2 instances, ALB, Auto Scaling group, and RDS. Deploy using the AWS Management Console. Update the stack for each application deployment.
  • B Deploy the Django app on AWS Elastic Beanstalk. Upload the application code (or Git commit via EB CLI). Beanstalk automatically provisions and manages ALB, Auto Scaling, EC2, and CloudWatch. Choose the "Rolling with additional batch" deployment policy for zero downtime.
  • C Containerize the Django app, push the image to Amazon ECR, and deploy it on Amazon ECS with Fargate. Configure an ALB in front of ECS, set up ECS Service Auto Scaling, and use CodePipeline for CI/CD.
  • D Create an AMI with Django pre-installed, launch EC2 instances in an Auto Scaling group, configure the ALB manually, and write userdata scripts to pull and deploy code from CodeCommit on each instance boot.
Correct Answer: B
AWS Elastic Beanstalk is specifically designed for developers who want to focus on code, not infrastructure. It supports Python/Django natively. With a single eb deploy command (or Git push), Beanstalk handles: EC2 provisioning, ALB configuration, Auto Scaling group setup, RDS integration, CloudWatch monitoring, and deployment orchestration. Elastic Beanstalk is free — you only pay for the underlying resources it creates. Zero downtime is achieved via the "Rolling with additional batch" or "Immutable" deployment policy — configurable without infrastructure code.
Option C (ECS Fargate) is also a modern approach, but requires containerization knowledge (Dockerfile, ECR, ECS task definitions, CodePipeline) — significantly more infrastructure expertise than Beanstalk. For teams with "no DevOps experience," Beanstalk is the stated answer. Option A (CloudFormation) requires writing and maintaining IaC templates — the exact overhead the team wants to avoid.
Question 17
Hybrid Kubernetes

A government agency runs a classified workload that processes citizen data. By law, all data must remain within the agency's own data center and can never transit the public internet or any external cloud. The agency's development team has invested heavily in Kubernetes tooling — Helm charts, kubectl workflows, GitOps pipelines — and wants to maintain this standardized approach for both classified on-premises workloads and their future unclassified cloud workloads (which will run on Amazon EKS). The agency wants to use AWS for ongoing Kubernetes distribution support and tooling, even for the on-premises cluster.

Which service enables this hybrid Kubernetes strategy?

  • A Deploy Amazon EKS in the AWS GovCloud region. Use AWS Direct Connect to connect the on-premises data center to GovCloud, and run classified containers with strict IAM and network policies.
  • B Use AWS Outposts to deploy an AWS-managed rack in the agency's data center. Run Amazon EKS on Outposts, which provides Kubernetes within the facility. Outposts requires a connection back to the parent AWS Region.
  • C Deploy Amazon EKS Anywhere on the agency's own bare-metal servers. The cluster runs entirely within the data center with no mandatory connectivity to AWS. The same Kubernetes distribution, tooling, and Helm charts work identically on-premises and on Amazon EKS in the cloud.
  • D Install an open-source Kubernetes distribution (k3s or RKE2) on the agency's servers and manage it independently. Use infrastructure-as-code tools to maintain consistency between on-premises and cloud clusters.
Correct Answer: C
Amazon EKS Anywhere runs the same EKS Kubernetes distribution on the customer's own hardware with no mandatory connectivity to AWS (it can operate fully air-gapped). The agency's existing Helm charts, kubectl commands, and GitOps workflows work identically. When the agency eventually runs workloads on EKS in the cloud, the same tools and manifests apply — true hybrid consistency. AWS provides the K8s distribution, tooling, and optional support subscription.
Option B (EKS on Outposts) requires persistent connectivity back to the AWS parent Region — making it non-viable for a fully air-gapped classified environment where data can never transit externally. Option A (GovCloud) still runs workloads IN AWS infrastructure, violating the "data center only" requirement. Option D (open-source) doesn't use AWS tooling — the question requires AWS-supported distribution and tooling.
Question 18
Serverless Code Reuse

A large enterprise has 200 development teams, each building serverless applications on AWS. The platform engineering team has built a set of reusable Lambda-based components — including a common audit logging function, a GDPR data redaction processor, and an automated cost-tagging pipeline. The team wants to make these components available to all 200 teams so they can be deployed into any account with a single click, without sharing source code. The components should be deployable via the AWS Management Console or CLI.

Which AWS capability enables this?

  • A Package each Lambda function as a .zip file and upload to a shared S3 bucket. Grant all teams read access to the bucket so they can download and deploy the function in their own accounts.
  • B Publish each component to the AWS Serverless Application Repository (SAR) as a private application. Teams deploy by searching SAR in the console or CLI and clicking Deploy — it creates a CloudFormation stack with all resources in their account. Source code is not exposed.
  • C Create AWS CloudFormation StackSets for each component and deploy them to all 200 accounts simultaneously from a centralized management account, ensuring all teams have the components installed.
  • D Share Lambda function ARNs across accounts using resource-based policies. Each team invokes the shared function cross-account instead of deploying their own copy.
Correct Answer: B
AWS Serverless Application Repository (SAR) is exactly the "App Store for Lambda applications." The platform team publishes applications using AWS SAM templates (SAM = Serverless Application Model, a simplified CloudFormation). Applications can be private (org-only) or public. Each application in SAR deploys as a CloudFormation stack into the target account — so each team gets their own isolated copy. Critically, the source code is packaged and abstracted — the consuming team doesn't see the Lambda source code if the publisher chooses not to share it. Deployable via console search or aws serverlessrepo CLI.
Option D (cross-account Lambda invocation) introduces a shared resource that becomes a dependency — if the shared function goes down or has a breaking change, all 200 teams are impacted simultaneously. SAR gives each team their own isolated deployment. Option C (StackSets push-to-all) removes team autonomy — teams can't choose which components to adopt or when to update them.

Question 19
Hybrid Cloud Multi-Service Architecture

A semiconductor manufacturer operates an automated chip inspection system on the factory floor. The system uses machine learning to analyze wafer images in real time and must respond in under 5 milliseconds — far too fast to make a round-trip to any cloud region. The factory loses internet connectivity for short periods due to electromagnetic interference from production equipment. The manufacturer requires that the inspection system continues operating during connectivity outages. The engineering team wants to use the same AWS APIs they use for all other workloads, and AWS must manage the hardware lifecycle.

Which solution meets all requirements?

  • A Deploy the inspection system on EC2 instances in the nearest AWS Region and use AWS Direct Connect with a redundant backup VPN for near-zero-latency connectivity to the factory floor cameras.
  • B Deploy an AWS Outposts rack on the factory floor. Run the ML inference workload on EC2 instances on the Outpost. The Outpost continues to operate locally during connectivity interruptions to the parent region. AWS manages and maintains the hardware.
  • C Use AWS Wavelength Zones inside a carrier's 5G network near the factory. Deploy EC2 instances in the Wavelength Zone to process wafer images with ultra-low latency via 5G connectivity.
  • D Deploy EKS Anywhere on the factory's own servers. Run the ML inference containers using Kubernetes and use GitOps to manage deployments. Connect to AWS for monitoring only when connectivity is available.
Correct Answer: B
AWS Outposts delivers a fully managed AWS rack to the factory floor. Key discriminators here: (1) Sub-5ms latency — Outposts are on-premises, eliminating all network round-trips. (2) Operates during connectivity outage — Outposts instances continue running and processing locally when the parent Region connection is interrupted. (3) Same AWS APIs — EC2, SageMaker, S3 on Outposts use identical APIs. (4) AWS manages hardware — AWS handles patching, monitoring, and hardware replacement.
Option A (Region + Direct Connect) fundamentally cannot achieve sub-5ms for camera-to-compute — even Direct Connect has 1–10ms latency for regional round-trips, plus camera interface overhead. Option C (Wavelength) requires a 5G carrier network and carrier gateway — manufacturing floors don't have 5G-based architectures built into the electromagnetic-interference environment described. Option D (EKS Anywhere) uses YOUR hardware — "AWS manages hardware lifecycle" eliminates this choice.
Question 20
Multi-Service Architect Thinking Multi-Select
⚠️ SELECT TWO — Multiple answers required

A media streaming company serves pre-recorded video content to millions of global users. The architecture team reviews their current setup and identifies two problems that are causing cost overruns and performance issues:

Problem 1: Video files stored in Amazon S3 (us-east-1) are being served directly from S3 to users globally. International users experience high latency and download speeds are slow. Data transfer costs from S3 are significant.

Problem 2: The company stores video files for 6 months post-publication with active streaming, then they become "library content" rarely watched. These library files use S3 Standard storage indefinitely, costing full Standard pricing even though they're accessed once a month or less.

Which two changes address each problem independently?

  • A Enable S3 Transfer Acceleration on the bucket. This uses CloudFront edge locations to accelerate uploads and downloads globally and reduces data transfer costs.
  • B Create an Amazon CloudFront distribution with the S3 bucket as origin. Configure appropriate cache TTLs. Global users fetch from the nearest CloudFront edge POP; cache hits don't consume S3 data transfer bandwidth.
  • C Configure an S3 Lifecycle Policy to transition video objects to S3 Standard-IA after 180 days (6 months). This reduces storage cost for library content while maintaining availability for streaming with millisecond access.
  • D Move all video files to Amazon S3 Glacier Flexible Retrieval immediately after upload and use Glacier's expedited retrieval (1–5 minutes) whenever a user requests a stream.
Correct Answers: B and C
B — CloudFront: Purpose-built CDN for global video delivery. Video files are cached at 450+ edge locations worldwide. Users stream from the nearest PoP instead of origin S3 in us-east-1. Cache hits dramatically reduce S3 data transfer costs (CloudFront-to-internet pricing is lower than S3-to-internet, and cache hits reduce origin fetches). This directly solves Problem 1.

C — S3 Standard-IA Lifecycle: After 6 months of active streaming, transitioning to Standard-IA reduces storage cost by ~58% vs Standard while keeping objects immediately available (millisecond access) for streaming. Standard-IA has a per-GB retrieval fee, but for occasional streaming this is cheaper than paying full Standard pricing indefinitely. This directly solves Problem 2.
Option A (S3 Transfer Acceleration) optimizes uploads — not content delivery for streaming. It also doesn't reduce per-GB transfer costs; it adds cost. Choosing this is the classic mix-up between Transfer Acceleration (upload speed) and CloudFront (distribution/delivery). Option D (Glacier) is disqualifying — Glacier Flexible Retrieval takes 1–5 minutes minimum for expedited retrieval. You CANNOT stream a video in real-time with 1–5 minute retrieval delay. Glacier is for archival, never for active serving.
