Question 01
Security KMS · Secrets Manager Hard
A financial services company runs a multi-tier application. The application tier (EC2 Auto Scaling group) must connect to an Amazon RDS Aurora MySQL cluster. The security team mandates: (1) database credentials must never appear in application code or environment variables, (2) credentials must be automatically rotated every 30 days, (3) all secrets must be encrypted with a customer-managed KMS key (CMK), and (4) if the CMK is deleted, access to the secret must be immediately revoked. The EC2 instances operate in private subnets with no internet access. Which combination of actions satisfies ALL four requirements with the least operational overhead?
EC2 ASG (private subnet)→ ??? credentials store→ Aurora MySQL
Requirement: auto-rotation every 30d+ CMK encryption+ No internet access
  • AStore credentials in SSM Parameter Store SecureString encrypted with the CMK. Schedule a Lambda function via EventBridge to rotate the password every 30 days. Assign EC2 instances an IAM role with ssm:GetParameter and provision a VPC endpoint for SSM.
  • BStore credentials in an S3 bucket with SSE-KMS using the CMK. Use a Lambda function on a schedule to rotate and re-upload the credentials file. Mount the S3 object on EC2 at startup via a user-data script. Create a VPC endpoint for S3.
  • CStore credentials in AWS Secrets Manager with a CMK-encrypted secret and enable native RDS rotation (30-day schedule). Grant EC2 instances an IAM role with secretsmanager:GetSecretValue and kms:Decrypt. Deploy VPC interface endpoints for Secrets Manager and KMS. Deleting the CMK makes the secret permanently inaccessible.
  • DStore credentials in AWS Secrets Manager with a CMK-encrypted secret and enable native RDS rotation. Grant EC2 instances a broad IAM role with secretsmanager:*. Access secrets over the internet via NAT Gateway. Enable automatic CMK deletion with a 7-day waiting period.
Correct Answer: C — Secrets Manager + CMK + VPC Endpoints + Native RDS Rotation
AWS Secrets Manager has built-in, native Lambda rotation for RDS — no custom Lambda code needed, zero operational overhead ✅. CMK encryption: Secrets Manager natively supports CMKs. When the CMK is deleted (after the mandatory 7-30 day waiting period), Decrypt calls fail and the secret becomes permanently inaccessible, satisfying requirement 4 ✅. No internet access in private subnets: VPC interface endpoints for both secretsmanager and kms allow the EC2 instances to call these APIs without traversing the internet or requiring a NAT Gateway ✅. IAM least privilege: secretsmanager:GetSecretValue + kms:Decrypt — no wildcards ✅.
Option A (SSM Parameter Store): SSM Parameter Store SecureString does NOT have native RDS credential rotation. You'd need to write and maintain a custom Lambda for rotation logic, violating "least operational overhead." Also, rotation must update the password in RDS AND in the store atomically — Parameter Store can't do this natively. Option B (S3): No rotation mechanism, no versioning of secrets, credentials stored as plaintext in a file — multiple violations. This is not a secrets management solution. Option D: Two critical flaws — (1) broad secretsmanager:* violates least privilege, and (2) accessing Secrets Manager over the internet via NAT Gateway violates the "no internet access" constraint and adds unnecessary cost and attack surface.
SSM Parameter Store vs Secrets Manager — the most frequently confused pair:
🔹 SSM Parameter Store SecureString: Stores config values encrypted with KMS. No built-in rotation. Free tier (Standard parameters). Use for: app config, feature flags, non-DB secrets.
🔹 AWS Secrets Manager: Built-in rotation for RDS, Redshift, DocumentDB, and custom. Costs ~$0.40/secret/month. Use when: automatic rotation is required, especially for DB credentials.
Keyword that forces Secrets Manager: "automatic rotation," "database password rotation," "least operational overhead for credential management."
Keyword that allows SSM: "store API keys," "configuration values," "no rotation needed," "free."
Correct Architecture Flow:
Secrets Manager · Private Subnet · No Internet Required
EC2 → VPC Interface Endpoint → Secrets Manager (CMK-encrypted) · Zero internet path · Built-in 30-day Lambda rotation
EC2 IAM Role→ VPC Endpoint (Secrets Manager)→ Secrets Manager (CMK-encrypted)→ Lambda Rotation Function→ Aurora MySQL (new password every 30d)
EC2 SDK call→ VPC Endpoint (KMS)→ CMK Decrypt→ Plaintext credential returned to app
KMS CMK deletion → Decrypt fails → Secrets Manager secret is locked → Aurora becomes inaccessible. This chain satisfies requirement 4 without any additional configuration.
Question 02
Security Macie · Security Hub · Inspector · Network Firewall Hard
A healthcare company stores patient records in S3. They are preparing for a HIPAA audit and need to: (1) automatically detect S3 buckets that contain PHI (Protected Health Information), (2) identify OS-level CVE vulnerabilities on EC2 instances running the application, (3) aggregate all security findings from multiple AWS security services into a single dashboard with compliance scores, and (4) block outbound HTTP calls to known malware domains from within the VPC. Which combination of AWS services, used together, satisfies ALL four requirements?
  • AAmazon Macie (detect PHI in S3) + Amazon Inspector (EC2 CVE scanning) + AWS Security Hub (aggregate findings, HIPAA compliance score) + AWS Network Firewall with domain-based deny rules (block malware domains)
  • BAmazon GuardDuty (detect PHI in S3) + Amazon Inspector (EC2 CVE scanning) + AWS Security Hub (aggregate) + AWS WAF (block malware domains)
  • CAmazon Macie (detect PHI in S3) + AWS Trusted Advisor (EC2 CVE scanning) + AWS Config (aggregate findings) + AWS Shield Advanced (block malware domains)
  • DAmazon Macie (detect PHI in S3) + Amazon Inspector (EC2 CVE scanning) + Amazon GuardDuty (aggregate findings dashboard) + AWS WAF (block outbound malware domains from VPC)
Correct Answer: A — Macie + Inspector + Security Hub + Network Firewall
Macie: Uses ML to discover and classify sensitive data (PHI, PII, financial data) inside S3 buckets automatically. Purpose-built for requirement 1 ✅. Inspector: Continuously scans EC2 instances and container images for software vulnerabilities (CVEs) and network exposure. Purpose-built for requirement 2 ✅. Security Hub: Aggregates findings from Macie, Inspector, GuardDuty, Firewall Manager, and partner tools. Provides HIPAA, PCI-DSS, NIST compliance scores in a single dashboard. Purpose-built for requirement 3 ✅. Network Firewall: Stateful, managed network firewall with domain-based filtering. Can block outbound FQDN/IP calls from VPCs to known malware C2 domains. Purpose-built for requirement 4 ✅.
Option B: GuardDuty does NOT detect PHI in S3 buckets. GuardDuty detects threat behaviours (unusual API calls, port scans, cryptocurrency mining) — not data classification. AWS WAF is a Layer-7 web application firewall for HTTP/HTTPS inbound traffic, NOT for blocking outbound VPC traffic to malware domains. Option C: Trusted Advisor does NOT scan for CVEs. It checks broad best practices (underutilised resources, security groups with open ports, MFA on root). AWS Config tracks resource configuration compliance, not security findings aggregation. Shield Advanced protects against DDoS — it cannot block outbound domain calls. Option D: GuardDuty generates its own findings but is NOT an aggregation dashboard for cross-service findings — that's Security Hub's job. WAF is inbound only; it cannot block outbound VPC traffic.
The "Which service detects WHAT" matrix — memorise this:
🔹 Macie: Finds sensitive DATA inside S3 (PHI, PII, credit cards). Content classification.
🔹 Inspector: Finds VULNERABILITIES in compute (EC2, Lambda, ECR). CVE scanning.
🔹 GuardDuty: Finds THREAT BEHAVIOUR in API calls, VPC flow logs, DNS. Anomaly detection.
🔹 Security Hub: AGGREGATES findings from all three above + partners. Compliance scoring.
🔹 Network Firewall: Layer-3/4/7 stateful filtering — blocks outbound domain/IP traffic inside VPC.
🔹 WAF: Protects HTTP/HTTPS endpoints (ALB, CloudFront, API GW) from inbound web attacks.
These four services are a security quartet. They rarely substitute for each other.
HIPAA Security Architecture on AWS:
Security Quartet — Four Services, Four Distinct Jobs
Macie (data) · Inspector (compute) · GuardDuty (behaviour) → all feed Security Hub (aggregate + compliance score) · Network Firewall (outbound blocking)
S3 (PHI data)→ Amazon Macie (scan)→ Security Hub (findings)
EC2 (App servers)→ Amazon Inspector (CVE scan)→ Security Hub (findings)
VPC (outbound traffic)→ AWS Network Firewall→ Block malware domains / Allow legitimate traffic
Security Hub→ HIPAA compliance score dashboard→ Audit-ready report
Question 03
Security IAM · RAM · KMS Hard
A large enterprise runs a multi-account AWS Organisation with a networking account, a security account, and 20 application accounts. The architecture team needs to: (1) allow all 20 application accounts to share a central Transit Gateway owned by the networking account without deploying separate TGWs per account, (2) enforce that all EBS volumes in any account are encrypted using a KMS key owned by the security account, and (3) ensure an IAM role in an application account can assume a role in the security account to read secrets, with least privilege. Which approach correctly satisfies all three requirements?
Security Account (owns CMK)+ Networking Account (owns TGW)+ 20 App Accounts
  • A(1) Create a VPC Peering connection from each app account to the networking account VPC. (2) Use AWS Config rules to check encryption and auto-remediate with SSM Automation. (3) Use identity federation with SAML to assume a cross-account role.
  • B(1) Use AWS RAM to share the Transit Gateway from the networking account to the Organisation. (2) Apply an SCP (Service Control Policy) that denies ec2:CreateVolume unless the KMS key ARN matches the security account CMK, but manage the CMK key policy manually per account. (3) Use IAM role chaining — app account role assumes security account role via sts:AssumeRole with ExternalId.
  • C(1) Share the Transit Gateway using AWS Resource Access Manager (RAM) with the entire AWS Organisation. (2) Use an SCP to deny ec2:CreateVolume without encryption, and separately enforce the CMK via AWS Config. (3) Attach a blanket sts:AssumeRole * permission to the app account role.
  • D(1) Share the Transit Gateway from the networking account using AWS RAM scoped to the AWS Organisation. (2) Apply an Organisation-wide SCP denying ec2:CreateVolume unless ec2:KmsKeyId matches the security account CMK ARN; grant app account roles cross-account KMS permissions via key policy. (3) Use a trust policy on the security account role that restricts assumption to the specific app account role ARN with an aws:PrincipalOrgID condition.
Correct Answer: D — RAM (TGW share) + SCP with KMS condition + Cross-account IAM with trust policy
RAM + Transit Gateway sharing: AWS RAM is the only mechanism to share a TGW across accounts without deploying duplicate TGWs. Scoping to the Organisation automatically includes all current and future member accounts ✅. SCP with KMS key condition: An SCP with a Deny on ec2:CreateVolume unless ec2:KmsKeyId matches the specific CMK ARN prevents any account from creating unencrypted volumes. Cross-account access to the CMK is granted via the KMS key policy (resource-based policy) in the security account — this is the correct dual-control: IAM permission + key policy ✅. Least-privilege cross-account role: App account role → sts:AssumeRole → security account role. The trust policy uses aws:PrincipalOrgID condition to limit to only accounts in the Org. This is least privilege ✅.
Option A: VPC Peering is non-transitive and cannot replace Transit Gateway for hub-and-spoke. With 20 accounts you'd need up to 20 peering connections, and they don't allow transitive routing. This doesn't scale. Option B: Manually managing CMK key policies per account defeats the purpose of centralised security. As new accounts are onboarded, they require manual CMK policy updates. Also, ExternalId alone is not sufficient for Organisation-scope least privilege — aws:PrincipalOrgID is the correct condition. Option C: sts:AssumeRole * wildcard in the permission policy is NOT least privilege — it allows assuming ANY role in any account, which is a massive privilege escalation risk. This single flaw disqualifies option C.
RAM vs other cross-account sharing mechanisms:
🔹 AWS RAM: Share AWS RESOURCES (TGW, subnets, Route 53 Resolver rules, License Manager configs, prefix lists). No data copy — the actual resource is shared. Scope: individual accounts or entire Organisation.
🔹 Resource-based policies (e.g., S3 bucket policy, KMS key policy): Share access to a resource from one account to another. Works for S3, KMS, SQS, Lambda, etc. The resource owner controls access via a policy document.
🔹 Cross-account IAM roles: Identity-based sharing — an IAM principal in account A assumes a role in account B. Used for compute access (EC2, Lambda calling services in another account).
Key exam pattern: "Share a VPC subnet/TGW/Route 53 rule across accounts" → RAM. "Grant cross-account S3/KMS access" → Resource-based policy. "Cross-account API calls from EC2/Lambda" → IAM cross-account role.
Multi-Account Network + Security Architecture:
RAM · TGW · SCP · Cross-Account IAM · KMS Key Policy
Management SCP enforces CMK encryption org-wide · TGW shared via RAM · CMK cross-account via key policy · Cross-account role with PrincipalOrgID condition
Networking Account (TGW)→ RAM share to Org → App Account 1 VPC attaches to TGW
Security Account (CMK)→ Key Policy cross-account → App Account EC2: CreateVolume encrypted with CMK
Management Account SCP→ Deny CreateVolume without CMK → All 20 App Accounts
App Account Role→ sts:AssumeRole → Security Account Role (reads Secrets Manager)
Question 04
Security Network Firewall · Security Groups · NACLs Hard
A company runs a three-tier application (public ALB → private EC2 app tier → private RDS). The security team requires: (1) block inbound traffic from known malicious IP prefixes (a managed threat-intelligence feed, updated weekly), (2) inspect and block outbound HTTP calls from the app tier to domains matching a blocklist (e.g., known C2 callback domains), and (3) ensure the solution does not require changes to existing Security Group rules and scales automatically with traffic. A solution architect proposes using AWS Network Firewall deployed in a dedicated inspection VPC with a centralised deployment model via Transit Gateway. Which statement about this architecture is most accurate?
  • ANetwork Firewall cannot inspect outbound HTTP traffic from EC2 instances because it operates at Layer 3/4 only. AWS WAF should be used on the ALB for domain-based filtering of app-tier outbound calls.
  • BNetwork Firewall supports stateful rules with Suricata-compatible rule groups that can match on HTTP host headers (Layer 7), enabling outbound domain blocklisting. Managed rule groups integrate AWS threat intelligence for IP reputation, eliminating manual IP list management. A centralised deployment in an inspection VPC with TGW routes all spoke VPC egress through the firewall without modifying Security Groups.
  • CNetwork Firewall integrates with AWS Shield Advanced to automatically update IP blocklists. The stateful engine inspects HTTPS (TLS) traffic by decrypting it inline before passing to rule evaluation, so encrypted C2 callbacks will also be blocked.
  • DThe centralised inspection VPC model requires VPC Peering (not Transit Gateway) because Network Firewall endpoints cannot attach to Transit Gateway route tables. Security Groups on each EC2 instance must be updated to send traffic to the firewall endpoint IP.
Correct Answer: B — Network Firewall is stateful Layer-7 with managed threat-intel IP feeds and TGW-based centralised deployment
Suricata-compatible stateful rules: AWS Network Firewall's stateful engine uses the open-source Suricata rule format, which can inspect HTTP host headers, SNI in TLS (without decryption), URLs, and payload patterns — enabling domain-based filtering at Layer 7 ✅. Managed rule groups: AWS provides managed rule groups including threat intelligence IP reputation lists that are updated automatically — no manual IP list maintenance ✅. Centralised TGW model: Network Firewall endpoints are created in subnets. These subnets sit in an inspection VPC. TGW route tables direct all egress from spoke VPCs through the inspection VPC's Network Firewall endpoint. No Security Group changes required ✅.
Option A: Network Firewall is NOT Layer 3/4 only. It has a stateful engine that understands HTTP (Layer 7) via Suricata rules. WAF is designed for inbound web app protection on ALB/CloudFront/API GW — it cannot inspect outbound traffic FROM EC2 instances. Option C: Network Firewall does NOT perform TLS termination/decryption. It can inspect SNI in the TLS ClientHello (which reveals the destination domain in most cases) but does not decrypt HTTPS body content. For full SSL inspection you'd need a proxy solution. Shield Advanced is a DDoS protection service — it does not provide threat-intel IP blocklists for Network Firewall. Option D: TGW is EXACTLY how centralised Network Firewall inspection works. TGW route tables are configured to route egress to the inspection VPC — this is the AWS-recommended architecture. VPC Peering is non-transitive and cannot centralise inspection across 20 VPCs.
Network Firewall vs WAF vs Security Groups vs NACLs — pick the right tool:
🔹 Security Groups: Instance-level, stateful, allow-only rules. Cannot block IP ranges with deny. No Layer-7.
🔹 NACLs: Subnet-level, stateless, allow + deny rules. IP ranges only. No Layer-7.
🔹 AWS WAF: Layer-7 for INBOUND HTTP/HTTPS to ALB/CloudFront/API GW. Managed rules for OWASP Top 10.
🔹 AWS Network Firewall: Layer 3-7 stateful firewall for INBOUND AND OUTBOUND VPC traffic. Domain filtering, IP reputation, Suricata rules. Best for: blocking outbound C2 callbacks, centralised VPC inspection.
Memory hook: "Outbound from VPC" → Network Firewall. "Inbound to ALB/CloudFront" → WAF.
Centralised Network Firewall Architecture (TGW model):
App VPC (spoke)→ TGW → Inspection VPC→ Network Firewall endpoint→ Stateful rules engine (Suricata)
Managed IP reputation rule group→ Block malicious IPs (inbound)
HTTP host header rule→ Block outbound C2 domains (app tier egress)
No Security Group changes. No NACL changes. Centralised inspection scales automatically.

Question 05
Analytics Kinesis · MSK · Lake Formation Hard
A retail company processes 1 million clickstream events per second from their website. Events must be: (1) ingested in real-time and stored in S3 in Parquet format for analytics, (2) consumed simultaneously by a fraud detection microservice (written in Python using the Kafka client library), and (3) catalogued so that Athena can query the S3 data using a schema-on-read approach with row-level access control restricting analysts to data only from their assigned region. What is the most architecturally sound design?
1M events/sec clickstream→ ??? Ingestion→ S3 (Parquet)+ Fraud Detection (Kafka client)
  • AAmazon MSK (Kafka-compatible) for ingestion — fraud detection microservice connects using native Kafka consumer API. Kinesis Data Firehose with MSK as source delivers to S3 in Parquet via Glue schema conversion. AWS Glue Data Catalog registers the S3 schema. AWS Lake Formation enforces row-level access control for regional analysts querying via Athena.
  • BAmazon Kinesis Data Streams for ingestion — fraud detection microservice connects using Kinesis SDK. Kinesis Data Firehose delivers to S3 in Parquet. AWS Glue Data Catalog registers schema. AWS IAM policies on S3 bucket enforce row-level access control per analyst.
  • CAmazon MSK for ingestion. A custom Kafka consumer on EC2 transforms and writes to S3 in Parquet. AWS Glue crawlers catalogue the data. AWS IAM row-level conditions on the Glue Data Catalog restrict analyst access.
  • DAmazon Kinesis Data Streams for ingestion with Kafka-compatible client library — fraud detection connects natively. Kinesis Data Firehose delivers to S3. Athena applies row-level filters via S3 bucket prefixes. Lake Formation manages column masking only.
Correct Answer: A — MSK + Firehose (Parquet) + Glue Catalog + Lake Formation (row-level)
MSK (Managed Streaming for Kafka): The fraud detection microservice uses a native Kafka client library — Kinesis Data Streams uses a proprietary SDK and is NOT Kafka-compatible. MSK is the only managed AWS service that provides a true Kafka API endpoint ✅. Kinesis Firehose with MSK source: Firehose can read from MSK topics and deliver to S3 with Parquet conversion (via a Glue schema) — no custom consumer code needed ✅. Glue Data Catalog: Registers the Parquet schema so Athena can query with schema-on-read ✅. Lake Formation row-level security: Lake Formation can apply row-level filters on Glue tables that restrict which rows an analyst sees based on their identity — S3 bucket policies cannot do row-level filtering, and IAM alone cannot enforce data-level row filtering on Athena queries ✅.
Option B: Kinesis Data Streams does NOT support Kafka client libraries natively. The Kafka Consumer API uses the Kafka protocol — Kinesis uses its own GetRecords/GetShardIterator API. A Python Kafka client pointed at Kinesis will fail. This disqualifies option B immediately. Option C: Custom EC2 Kafka consumer for S3 write is operationally heavyweight and doesn't use managed services — violates "architecturally sound." IAM conditions on Glue Data Catalog are NOT the same as Lake Formation row-level security. IAM cannot filter data rows inside Athena query results. Option D: Same Kafka-incompatibility flaw as B. Additionally, using S3 bucket prefixes for row-level access control is brittle and doesn't scale across many analysts — analysts with broad S3 permissions could bypass prefix restrictions. Lake Formation manages table-level, column-level, and row-level security, not just column masking.
Kinesis Data Streams vs MSK — the one keyword that separates them every time:
🔹 Amazon MSK: Fully managed Apache Kafka. True Kafka protocol compatibility. Kafka consumer groups, topic partitions, Kafka Connect, Kafka Streams. Use when: existing Kafka apps, Kafka client library required.
🔹 Kinesis Data Streams: AWS-native real-time streaming. Proprietary API (PutRecord, GetRecords). KCL (Kinesis Client Library). Use when: AWS-native integration (Lambda, Firehose), no Kafka dependency.
The exam keyword: "Kafka client library," "Kafka-compatible," "Kafka consumer groups" → MSK. "AWS-native streaming, Lambda consumer" → Kinesis.
Lake Formation row-level vs IAM: IAM policies control API-level access. Lake Formation controls DATA-level access (which rows/columns a query returns). For Athena query-time row filtering → Lake Formation.
Correct Real-time Analytics + Governance Architecture:
Website (1M events/s)→ Amazon MSK (Kafka topic)→ Fraud Detection (Kafka consumer, Python)
MSK topic→ Kinesis Firehose (Parquet transform via Glue schema)→ S3 data lake (partitioned by region/date)
Glue Data Catalog (schema registry)→ Athena (SQL queries)
Lake Formation (row-level filter: analyst_region = data_region)→ Analyst sees only their region's data in Athena results
Question 06
Analytics Glue · EMR · Redshift · Athena Hard
A data engineering team needs to process 500 TB of raw JSON logs stored in S3. Their requirement has three phases: (1) schema discovery and cataloguing of the JSON structure without writing any code, (2) a complex multi-step Spark transformation involving a custom machine learning feature engineering library that is not available in managed Glue — the data science team insists on controlling the exact Spark version and adding custom Python libraries, and (3) the final transformed Parquet data (approx. 50 TB after compression) should be loaded into a columnar data warehouse for BI analysts who need sub-second query performance on filtered aggregations. Which combination of services best maps to the three phases?
  • APhase 1: Glue Crawler (schema discovery, no code). Phase 2: AWS Glue ETL (managed Spark). Phase 3: Amazon Athena (S3 direct query, sub-second with partition pruning).
  • BPhase 1: Amazon Macie (scan S3 JSON for schema). Phase 2: AWS Lambda (Spark transformation). Phase 3: Amazon OpenSearch (columnar BI queries).
  • CPhase 1: AWS Glue Crawler (auto-discovers JSON schema, populates Glue Data Catalog, no code). Phase 2: Amazon EMR with custom Spark configuration and bootstrapped Python libraries (full control over Spark version and dependencies). Phase 3: Amazon Redshift with COPY command loading Parquet from S3 (columnar MPP warehouse, sub-second aggregations on 50 TB with RA3 nodes).
  • DPhase 1: AWS Glue Crawler (schema discovery). Phase 2: AWS Glue Studio (custom Spark with managed runtime, supports any Python library). Phase 3: Amazon Redshift with Redshift Spectrum (query S3 Parquet directly without loading).
Correct Answer: C — Glue Crawler + EMR (custom Spark) + Redshift (columnar warehouse)
Phase 1 — Glue Crawler: Crawls S3, infers schema from JSON/CSV/Parquet, populates the Glue Data Catalog automatically. Zero code required ✅. Phase 2 — Amazon EMR: The requirement explicitly states "custom ML library not in managed Glue" and "control exact Spark version." Glue uses a managed Spark environment with limited library customisation. EMR allows you to specify exact Spark version, bootstrap custom Python packages (via pip or conda), and install any library on the cluster nodes. EMR is purpose-built for this use case ✅. Phase 3 — Redshift: Redshift is a massively parallel processing (MPP) columnar data warehouse. Loading Parquet data with the COPY command and using RA3 nodes delivers sub-second query times on filtered aggregations for BI analysts — this is Redshift's core strength ✅.
Option A: AWS Glue ETL does run managed Spark, but it does NOT support custom Spark versions or arbitrary Python library bootstrapping. Glue has a managed runtime; you cannot install custom C-extension ML libraries. Also, Athena is NOT a data warehouse — it queries S3 data in-place and has query latency of seconds to minutes depending on data volume and partitioning. "Sub-second performance" on BI aggregations over 50 TB requires Redshift, not Athena. Option B: Macie scans for sensitive data classification, not schema discovery. When a team member saves a document, a GraphQL mutation triggers AppSync to push the update to all subscribed mobile clients within milliseconds. No polling ✅. ⚡ Question 16 ALB · Auto Scaling · RDS Multi-AZ · Route 53 · ElastiCache · S3 · CloudFront · Domain: Design Resilient Architectures (26% of exam)
Question 16
Resilience ALB · Auto Scaling · RDS · ElastiCache · Route 53 Hard
A global e-commerce company runs a three-tier web application in a single AWS Region (us-east-1) with the following current state: an Application Load Balancer fronting an Auto Scaling group of EC2 instances, a single-AZ RDS MySQL instance, and user session data stored in memory on each EC2 instance. During a recent AZ failure, the application experienced 45 minutes of downtime. The following requirements must ALL be met after the redesign:

(1) The application must survive the complete failure of any single Availability Zone with no more than 30 seconds of disruption.
(2) User sessions must not be lost if an EC2 instance is terminated by Auto Scaling.
(3) The database must automatically failover within 60 seconds with no application code changes required.
(4) Static assets (images, CSS, JS) must be served with sub-10ms latency globally to reduce origin load.

Which combination of changes correctly satisfies ALL four requirements with the least operational overhead?
Current: Single-AZ RDS + In-memory sessions + Single-AZ ASG→ Problem: AZ failure = 45 min outage
  • ADeploy EC2 instances across 3 AZs with an ALB. Store sessions in DynamoDB global tables. Promote the RDS instance to a Read Replica in a second AZ and manually promote it during failures. Serve static assets from S3 with Transfer Acceleration enabled.
  • BConfigure the ALB and Auto Scaling group across a minimum of 3 AZs. Migrate session storage to Amazon ElastiCache for Redis (Multi-AZ with automatic failover). Convert the RDS instance to RDS Multi-AZ (synchronous standby in a second AZ, automatic DNS failover within 60 seconds, no app changes required). Distribute static assets via Amazon CloudFront with an S3 origin (global edge cache, sub-10ms for cache hits).
  • CConfigure the ALB and Auto Scaling group across 2 AZs. Store session data in an SQS queue. Upgrade the RDS instance to Aurora Global Database for cross-region replication. Use CloudFront with an EC2 origin for static assets.
  • DDeploy EC2 instances across 3 AZs with the ALB. Store sessions using AWS Systems Manager Session Manager. Enable RDS automated backups with a 1-hour RPO and restore from snapshot during failures. Deliver static assets through S3 with a public bucket and S3 Intelligent Tiering.
Correct Answer: B — Multi-AZ ALB/ASG + ElastiCache Redis Multi-AZ + RDS Multi-AZ + CloudFront/S3
Multi-AZ ALB + ASG (Req 1): The ALB is inherently multi-AZ — it distributes traffic across healthy instances in all configured AZs. When one AZ fails, the ALB health checks detect unhealthy targets within seconds and route traffic only to the surviving AZs. With a minimum of 3 AZs in the ASG, losing one AZ leaves 2 AZs serving traffic with no more than 30 seconds of disruption ✅. ElastiCache for Redis Multi-AZ (Req 2): Redis is a purpose-built in-memory store for session data. By externalising sessions from EC2 into ElastiCache, any new or replacement EC2 instance can immediately read the session — no session loss on instance termination. Multi-AZ with automatic failover means the cache survives an AZ failure too ✅. RDS Multi-AZ (Req 3): RDS Multi-AZ maintains a synchronous standby replica in a second AZ. On primary failure, AWS automatically updates the DNS endpoint (CNAME) to the standby — typically within 60–120 seconds. No application connection string changes needed — the app reconnects to the same endpoint ✅. CloudFront + S3 (Req 4): CloudFront caches static content at 450+ edge locations globally. Cache hits are served from the nearest edge — well under 10ms for most users. S3 as the origin stores assets durably. The ALB and EC2 fleet serve zero static asset requests once cached ✅.
Option A: A Read Replica is asynchronous (there can be replication lag) and does NOT automatically failover — you must manually promote it. This means downtime during an RDS failure could easily exceed 60 seconds and requires operator intervention. This fails Requirement 3. S3 Transfer Acceleration speeds up uploads TO S3 (from users to S3), not downloads. For global caching of static assets with sub-10ms latency, CloudFront is required. Option C: Storing sessions in SQS is architecturally incorrect — SQS is a message queue for decoupled processing, not a session store. There is no mechanism for an EC2 instance to "look up" a user's session from an SQS queue. Aurora Global Database adds cross-region replication (for DR) — the question asks for single-region AZ resilience, where RDS Multi-AZ is sufficient and far simpler. CloudFront with an EC2 origin is valid but less optimal than S3 (S3 scales to unlimited concurrent requests; EC2 is the bottleneck you're trying to bypass). Option D: AWS Systems Manager Session Manager is for managing SSH/RDP sessions to EC2 instances — it has nothing to do with application user session data. RDS automated backups with a 1-hour RPO means up to 1 hour of data loss and requires a manual restore — there is no automated failover. This clearly fails Requirements 2 and 3. S3 Intelligent Tiering optimises storage cost for infrequently accessed objects — it provides no latency improvement and is irrelevant to CDN delivery.
The SAA-C03 Resilience Checklist — 4 layers every HA architecture needs:
🔹 Compute layer (AZ resilience): ALB across ≥2 AZs + ASG with multi-AZ. ALB health checks = auto-recovery from AZ failure.
🔹 Session layer (stateless EC2): Externalise sessions to ElastiCache (Redis) or DynamoDB. Never store state in-memory on EC2 — any instance termination loses that state.
🔹 Database layer (AZ failover): RDS Multi-AZ = synchronous replication + automatic DNS failover (<2 min). Read Replicas = asynchronous, manual promotion = NOT HA.
🔹 Content layer (global latency): CloudFront + S3 = static asset delivery at edge. Reduces origin load and drops latency globally to single-digit ms for cached content.
Critical distinctions: RDS Multi-AZ ≠ Read Replica (HA vs scale). ElastiCache ≠ SQS (session store vs queue). CloudFront ≠ S3 Transfer Acceleration (download CDN vs upload accelerator).
Redesigned Three-Tier HA Architecture:
Three-Tier HA — Multi-AZ · Stateless EC2 · Redis Sessions · RDS Standby · CloudFront Edge
ALB health checks reroute within seconds · EC2 instances are stateless · Sessions live in Redis · RDS standby auto-promotes · CloudFront serves static globally
Users→ CloudFront (static assets, <10ms)→ S3 origin
Users (dynamic)→ ALB (3 AZs)→ EC2 ASG (stateless, 3 AZs)→ ElastiCache Redis (sessions)
EC2 ASG→ RDS Multi-AZ Primary→ sync replication → Standby (auto-failover <60s, same endpoint)
