Question 01
Networking Transit Gateway PrivateLink HIGH WEIGHT

A large enterprise runs 120 AWS VPCs spread across 6 AWS accounts and 3 Regions. The network team has these non-negotiable requirements:
R1. All VPCs must communicate with each other through a centrally managed, scalable hub-and-spoke topology.
R2. A centralised inspection VPC must inspect all East-West (VPC-to-VPC) traffic using a third-party virtual firewall appliance.
R3. An internal shared service (a proprietary data platform) must be exposed to consumer VPCs that have overlapping RFC 1918 CIDRs — traditional VPC peering is not possible.
R4. Inter-Region traffic must be encrypted in transit using AWS-native controls.

Which combination of services satisfies ALL four requirements with the least operational overhead?

120 VPCs (6 accounts) → hub-and-spoke → ? Central Hub → East-West inspection → Firewall appliance VPC
Consumer VPC (overlapping CIDR) → private access → Shared data platform → cross-Region encrypted → Remote Region VPCs
🏗️ Architecture — Multi-VPC Enterprise Network with Inspection & PrivateLink
TGW hub-and-spoke with appliance-mode inspection routing. PrivateLink bypasses CIDR conflicts for the shared service. Inter-Region peering is natively encrypted.
  • AVPC Peering for all 120 VPCs + AWS Network Firewall in a hub VPC + VPC Peering again for the shared service + VPN for inter-Region encryption.
  • BAWS Transit Gateway (single Region) + AWS Network Firewall for East-West inspection + VPC Peering for the shared service consumer VPCs + Transit Gateway inter-Region peering for cross-Region.
  • CAWS Transit Gateway for hub-and-spoke + AWS PrivateLink for the shared service + Site-to-Site VPN for inter-Region + AWS WAF for East-West inspection.
  • DAWS Transit Gateway (with inter-Region peering, encrypted in transit) for hub-and-spoke + a centralized inspection VPC attached to the TGW using an appliance-mode route table for East-West traffic + AWS PrivateLink (endpoint service backed by NLB) for the overlapping-CIDR shared service.
Correct Answer: D — Transit Gateway + Inspection VPC + PrivateLink
TGW with inter-Region peering natively encrypts cross-Region traffic using AES-256 without needing VPN overlays ✅. Appliance mode on the TGW route table ensures symmetric East-West routing so the firewall sees both directions of every flow ✅. PrivateLink (NLB-backed endpoint service) solves overlapping CIDRs — consumer VPCs create Interface Endpoints and reach the service over AWS backbone without any route table concerns ✅.
Option A (VPC Peering mesh) fails at scale: 120 VPCs require up to 7,140 peering connections — unmanageable. VPC Peering is non-transitive; a firewall hub VPC in a peering mesh CANNOT intercept traffic between two other peered VPCs. Option B is close but uses VPC Peering for the shared service — this breaks on overlapping CIDRs (R3). Option C uses AWS WAF for East-West inspection — WAF is an L7 web application firewall, not a network inspection tool for VPC-to-VPC traffic. WAF cannot inspect arbitrary TCP flows between VPCs.
The "scale + overlap + inspect" triad — three signals, three services:
🔹 "100+ VPCs, hub-and-spoke, central routing" → Transit Gateway (TGW). VPC Peering cannot route transitively and doesn't scale.
🔹 "Overlapping CIDRs, private service exposure" → PrivateLink. TGW and Peering both fail on overlapping CIDRs because routing becomes ambiguous.
🔹 "Inspect all VPC-to-VPC traffic, firewall appliance" → TGW appliance-mode + inspection VPC. Appliance mode forces symmetric routing so the firewall sees both request and response flows. Without appliance mode, asymmetric routing breaks stateful firewalls.
On the exam: if you see "overlapping CIDR" → immediately eliminate VPC Peering and TGW as the exposure mechanism; PrivateLink is the only correct answer.
TGW in the real world: A global bank's AWS Landing Zone uses a single TGW per Region (peered inter-Region) connecting 200+ account VPCs. All outbound internet traffic is routed through a centralised Egress VPC; all VPC-to-VPC traffic flows through a Security VPC running Palo Alto VM-Series firewalls on EC2 via appliance-mode TGW attachments. The bank's internal risk rating platform is exposed as a PrivateLink endpoint service — 60 consumer account VPCs reach it regardless of CIDR collisions.
Spoke VPCs (120)→ TGW attachment → Transit Gateway Hub→ appliance route table → Inspection VPC (firewall) → approved traffic → Destination VPC
Consumer VPCs (overlapping CIDRs)→ Interface Endpoint → PrivateLink→ NLB → Shared Data Platform
Question 02
Networking Global Accelerator CloudFront HIGH WEIGHT

A global gaming company serves players in 40 countries. Their application has two distinct traffic types that must be handled differently:
Traffic Type 1: Real-time multiplayer game state updates using UDP and custom TCP protocols. Players experience high latency and jitter because traffic traverses the public internet to reach EC2 game servers in us-east-1. Players need static IPs to whitelist in national firewall rules.
Traffic Type 2: Game asset downloads (large binary files, images, video trailers) served from S3 — this is standard HTTPS content that benefits from edge caching.

Which combination of services optimally handles BOTH traffic types with the lowest latency and cost?

Global players (40 countries) → Type 1: UDP/TCP game state → ? Service A → EC2 game servers (us-east-1)
Global players (40 countries) → Type 2: HTTPS assets → ? Service B → S3 origin
🏗️ Architecture — Global Accelerator (UDP/TCP) vs CloudFront (HTTPS Assets)
Global Accelerator routes UDP/TCP game traffic over the AWS backbone to EC2. CloudFront caches HTTPS assets at the edge from S3. Two services, two traffic types, zero overlap.
  • AAWS Global Accelerator for Traffic Type 1 (UDP/TCP anycast, 2 static IPs, routes over AWS backbone) + Amazon CloudFront for Traffic Type 2 (HTTPS caching at 600+ edge locations, S3 origin).
  • BAmazon CloudFront for both traffic types — CloudFront supports UDP and TCP protocols in addition to HTTP, and can assign static Anycast IP addresses that players whitelist.
  • CAWS Global Accelerator for both traffic types — it supports HTTPS content caching at the edge, eliminating the need for a separate CloudFront distribution for asset downloads.
  • DAmazon Route 53 latency-based routing for Traffic Type 1 + CloudFront for Traffic Type 2 — Route 53 provides static IPs for whitelisting and routes UDP traffic over the AWS backbone.
Correct Answer: A — Global Accelerator + CloudFront
AWS Global Accelerator is purpose-built for non-HTTP traffic: it supports TCP and UDP, assigns two static Anycast IPv4 addresses globally (players whitelist these), and routes all traffic over the AWS private backbone from the nearest AWS edge location — eliminating public internet jitter ✅. CloudFront caches HTTPS content at 600+ edge locations with S3 as an origin, dramatically reducing per-byte transfer costs for large asset downloads ✅. These two services are complementary — not overlapping.
Option B is the classic distractor: CloudFront is an HTTP/HTTPS CDN only. It does NOT support raw UDP or custom TCP protocols. It also does NOT assign static IP addresses — CloudFront uses thousands of dynamic anycast IPs via DNS, which cannot be individually whitelisted. Option C is the mirror trap: Global Accelerator does NOT cache content at the edge. It is a network accelerator that routes traffic to your origin — it has no CDN layer. Choosing it for asset downloads means every byte still hits your origin (and incurs origin data transfer costs). Option D: Route 53 is a DNS service. It routes DNS queries, not traffic. It has no static IPs to whitelist, and it cannot route UDP game traffic over the AWS backbone.
Global Accelerator vs CloudFront — the two sentences that separate them on the exam:
🔹 Global Accelerator: "I need my TCP/UDP traffic to travel over AWS's private network instead of the public internet. I need static IPs. I do NOT need caching."
🔹 CloudFront: "I need HTTP/HTTPS content cached closer to my users at the edge. I don't need static IPs."
The killer keyword for Global Accelerator: "UDP", "TCP passthrough", "static IP whitelist", "gaming", "VoIP".
The killer keyword for CloudFront: "cache", "edge", "CDN", "origin", "TTL", "Lambda@Edge".
Real-world gaming stack: A major MMO game uses Global Accelerator for all real-time game socket connections (TCP 443 for the game protocol). Players connect to one of two static Anycast IPs that telecom carriers worldwide have permanently whitelisted. The connection enters the AWS network at the nearest PoP and rides the AWS backbone to us-east-1 game servers. Game trailer downloads, map tiles, and skin assets are served via CloudFront from S3, cached for 24 hours at edge locations. The architecture costs 60% less than running a global fleet of EC2 proxy servers.
Question 03
Networking Route 53 ALB HIGH WEIGHT

A healthcare SaaS provider serves hospitals in the EU and US. Their application is deployed in eu-west-1 and us-east-1. They have these routing requirements:
R1. EU patients must be served from eu-west-1 by default due to GDPR data residency laws — EU traffic must NEVER route to us-east-1 under normal conditions.
R2. If the eu-west-1 ALB health check fails, EU traffic must automatically fail over to us-east-1 (regulators accept cross-border failover during outages).
R3. Within each Region, the ALB must route /api/* requests to an ECS Fargate cluster and /static/* requests to an S3 bucket (via S3 website endpoint).
R4. A canary deployment is in progress — 10% of US traffic must go to a new ECS task definition revision for testing.

Which combination satisfies ALL four requirements?

🏗️ Architecture — Route 53 Geolocation + Failover + ALB Path Routing + Canary
Geolocation enforces data residency. Failover health check catches outages. ALB handles path-based routing and weighted canary splits natively.
  • ARoute 53 Latency-based routing to both Regions + ALB path-based routing rules + ALB weighted target groups (10%/90%) for the canary.
  • BRoute 53 Geolocation routing (EU → eu-west-1, default → us-east-1) + Route 53 health checks without failover + ALB path-based routing + ALB weighted target groups.
  • CRoute 53 Geolocation routing with Failover records (primary: eu-west-1 ALB, secondary: us-east-1 ALB, health check on eu-west-1) + ALB path-based routing rules (/api/* → Fargate, /static/* → S3) + ALB weighted target groups (10% new revision, 90% stable) for canary in us-east-1.
  • DAWS Global Accelerator with endpoint weights (90%/10%) + Route 53 Geolocation routing for EU residency enforcement + ALB for path routing — Global Accelerator health checks handle the eu-west-1 failover automatically.
Correct Answer: C — Route 53 Geolocation + Failover + ALB path-based + Weighted Target Groups
Geolocation routing enforces data residency — EU traffic goes to eu-west-1, non-EU to us-east-1, by geography, not latency ✅. Route 53 Failover with health checks on the eu-west-1 ALB automatically switches EU traffic to us-east-1 if the primary is unhealthy ✅. ALB path-based routing handles /api/* → Fargate and /static/* → S3 at L7 ✅. ALB weighted target groups (10%/90%) implement canary without needing a separate service ✅. All four requirements met with standard managed services.
Option A uses Latency-based routing — this routes by network latency, NOT by geography. A low-latency path from France to us-east-1 could route EU patients across the Atlantic, violating GDPR. Option B is close but omits failover — without a failover record, if eu-west-1 goes down, EU traffic has no fallback and results in an outage instead of cross-border failover. Option D is over-engineered: Global Accelerator is designed for TCP/UDP acceleration, not geolocation compliance enforcement. Mixing Global Accelerator AND Route 53 geolocation creates a double-routing layer — Global Accelerator routes by network proximity which can override geolocation intent.
Route 53 routing policy cheat-sheet — 4 policies, 4 distinct use cases:
🔹 Latency-based: "Route to the fastest Region for this user's network." No geographic guarantee.
🔹 Geolocation: "Route strictly by user's country/continent." Enforces data residency. Has a Default record for unmatched locations.
🔹 Geoproximity: "Route by geography but bias toward a Region." Useful for gradual Region migration.
🔹 Failover: "Route to primary; switch to secondary if health check fails." Always combine with health checks.
These can be combined (chained): Geolocation → ALB record → Failover. The exam often tests combinations like this question. Keywords: "data residency" → Geolocation. "lowest latency" → Latency. "disaster recovery" → Failover.
ALB Weighted Target Groups for canary: ALB can send X% of traffic to Target Group A (stable) and (100-X)% to Target Group B (canary) — no Route 53 record changes needed, no deployment of a separate load balancer. This is natively supported on a single ALB listener rule. Adjust the weight from 10% to 100% incrementally as confidence grows.
EU users→ Route 53 Geolocation → eu-west-1 ALB (primary)→ failover if unhealthy → us-east-1 ALB (secondary)
ALB us-east-1→ /api/* → Fargate TG (90% stable + 10% canary) → /static/* → S3 bucket TG
Question 04
Networking Wavelength VPC Security

A telecom company is building a real-time vehicle collision prediction application for autonomous vehicles. The application processes sensor data from vehicles traveling on highways and must respond in under 10 milliseconds. The company has a partnership with a major 5G carrier. Processing must happen as close to the vehicle as possible — within the carrier's 5G network itself, not in a central AWS Region. The backend model inference is done on GPU-equipped EC2 instances.

Additionally, the security team requires that the Wavelength Zone EC2 instances can reach the parent VPC's backend services (in us-east-1) without traversing the public internet. VPC Security Groups must enforce that only the Wavelength instances can initiate connections to the backend — never the reverse.

Which architecture meets these requirements?

🏗️ Architecture — AWS Wavelength Zone inside 5G Carrier Network
Wavelength Zone sits physically inside the 5G carrier's infrastructure. Carrier Gateway is the only ingress point from 5G devices. Private route table connects Wavelength to parent VPC without internet.
  • ADeploy inference EC2 instances in us-east-1 behind a Global Accelerator endpoint — the Anycast network brings 5G traffic to the nearest AWS PoP before routing to us-east-1, achieving sub-10ms latency from the vehicle.
  • BDeploy GPU EC2 instances in an AWS Wavelength Zone (inside the carrier's 5G network) within a subnet of the parent VPC. Use a Carrier Gateway for 5G device ingress. Route backend traffic from Wavelength to the parent VPC through the VPC's private route table (no internet gateway needed). Apply Security Groups allowing only Wavelength→Backend, not Backend→Wavelength.
  • CDeploy EC2 instances in an AWS Local Zone nearest to the 5G cell towers. Local Zones extend AWS infrastructure to metro areas and provide sub-10ms latency from 5G devices on the carrier network.
  • DDeploy the application on AWS Outposts installed in the telecom carrier's 5G core data centre. Outposts bring AWS infrastructure on-premises and would achieve sub-10ms latency for 5G traffic originating from the carrier's network.
Correct Answer: B — AWS Wavelength Zone + Carrier Gateway + Security Groups
AWS Wavelength places compute directly inside the carrier's 5G network infrastructure — traffic from 5G devices never leaves the carrier's network to reach the compute. This achieves single-digit millisecond latency ✅. A Carrier Gateway allows ingress from 5G devices and egress to the carrier network — it is the Wavelength-specific equivalent of an Internet Gateway ✅. Wavelength subnets are part of the parent VPC — traffic back to the parent VPC uses the private route table without an internet gateway ✅. Security Groups are stateful — allowing Wavelength→Backend implicitly allows return traffic, while blocking unsolicited Backend→Wavelength connections ✅.
Option A (Global Accelerator): Global Accelerator routes traffic over the AWS backbone from the nearest AWS edge PoP to your EC2. The traffic still has to travel from the vehicle over the 5G radio network to an AWS PoP — this is NOT within the carrier's 5G core. Sub-10ms from a moving vehicle to a PoP is physically impossible for vehicles more than a few km from the PoP. Option C (Local Zone): Local Zones extend AWS to metro areas for users with single-digit ms requirements to on-prem workloads. They are NOT embedded in 5G carrier networks. The latency is better than a Region, but not comparable to Wavelength's within-carrier placement. Option D (Outposts): Outposts is customer-managed hardware in your own data centre. A telecom carrier would need to contractually host Outposts hardware in their 5G core — operationally complex and NOT the designed use case. Wavelength is the AWS product specifically co-designed with telecom operators for exactly this.
The edge hierarchy — four levels, four services:
🔹 AWS Region: Full service portfolio. Latency: 10–100ms globally.
🔹 CloudFront Edge / Global Accelerator PoP: 600+ locations. HTTP caching or TCP/UDP acceleration. Latency: 5–50ms from population centres.
🔹 AWS Local Zone: Single-digit ms to on-prem users in a metro area. Subset of Region services (EC2, EBS, RDS, ECS). NOT inside carrier networks.
🔹 AWS Wavelength Zone: Physically INSIDE 5G carrier infrastructure. Ultra-low latency (<10ms) for 5G devices. Only reachable via Carrier Gateway.
The trigger keyword for Wavelength: "5G", "carrier", "mobile edge computing", "MEC", <10ms from mobile device.
Wavelength in the real world: Verizon has partnered with AWS to offer Wavelength Zones in Boston, Dallas, Denver, Las Vegas, Miami, New York, San Francisco, and Seattle. A robotics company runs computer vision inference on g4dn.xlarge GPU instances in Wavelength Boston — factory floor robots send camera frames over 5G and get back bounding box coordinates in 8ms. The inference instances connect to the main VPC (for model updates from S3) through the VPC private route table — no internet egress.
Question 05
Networking EC2 Auto Scaling ALB Multi-AZ HIGH WEIGHT

A company runs a multi-tier web application with an Application Load Balancer in front of an EC2 Auto Scaling Group across 3 Availability Zones. A security audit reveals two findings:
Finding 1: EC2 instances in the Auto Scaling Group are in a public subnet with direct internet access — this violates security baselines.
Finding 2: During a recent traffic spike, the Auto Scaling policy scaled out correctly but the new instances took 12 minutes to pass ALB health checks due to a slow application startup. During these 12 minutes, the ALB sent traffic to the new instances before they were ready, causing HTTP 502 errors.

Which two changes fix BOTH findings with minimal architectural disruption? (Select TWO)

⚡ SELECT TWO answers for this question
🏗️ Architecture — Before (Broken) vs After (Fixed) 3-Tier ALB + ASG
Left: broken public-subnet ASG with instant traffic on launch → 502s. Right: private subnets + NAT for security, Slow Start for gradual traffic ramp.
  • AMove EC2 instances to private subnets. Keep the ALB in public subnets. Add a NAT Gateway in each public subnet for outbound internet access from private instances. Update the Auto Scaling Group launch template to use the private subnet IDs.
  • BEnable ALB Slow Start mode on the target group and set a warm-up duration of 720 seconds (12 minutes) so the ALB gradually increases the traffic weight to new instances during their startup period.
  • CIncrease the ALB idle timeout to 720 seconds to prevent connections from timing out while new instances are warming up.
  • DConfigure the Auto Scaling Group health check grace period to 720 seconds. This delays the ALB from sending any traffic to new instances until the grace period expires.
Correct Answers: A and B
A — Private subnets + NAT Gateway: Moving instances to private subnets removes their direct internet route ✅. The ALB stays in public subnets and forwards to private instances — this is the standard AWS 3-tier architecture. NAT Gateway provides outbound-only internet access for updates, agent traffic, etc. ✅

B — ALB Slow Start mode: When Slow Start is enabled on a target group, the ALB linearly ramps traffic weight to a newly registered target from 0 to 100% over the configured duration. New instances receive progressively increasing traffic as they warm up — preventing the 502 spike ✅.
Option C (Idle Timeout): The ALB idle timeout controls how long an established connection can be idle before the ALB closes it. It has NOTHING to do with when traffic is sent to new targets. Setting it to 720 seconds would not prevent 502s from half-started instances — it would just keep idle connections open longer, which is unrelated to the problem. Option D (Health Check Grace Period): The ASG health check grace period tells the ASG how long to wait before checking EC2 instance health (so it doesn't replace an instance that's still booting). It does NOT delay ALB from sending traffic — once the instance registers with the ALB target group and passes the ALB health check, the ALB routes traffic to it. The grace period is about ASG termination protection, not ALB traffic distribution.
Four "delays" in the ALB/ASG lifecycle — know which does what:
🔹 ALB Health Check (Healthy Threshold): Instance must pass N consecutive checks before ALB routes traffic. Increase threshold for slow-starting apps.
🔹 ALB Slow Start: Once registered as healthy, ALB ramps traffic weight gradually. Best for slow-warming apps (caches, JVM warm-up).
🔹 ASG Health Check Grace Period: How long ASG waits before checking and potentially terminating an instance. Protects from ASG terminating a booting instance.
🔹 Connection Draining / Deregistration Delay: How long ALB keeps routing in-flight requests to a de-registering instance before forcibly removing it.
The 502 error from a new instance = traffic sent before app ready → fix with Slow Start or Health Check Threshold.
Standard 3-tier AWS architecture: ALB in public subnets receives internet traffic. EC2 Auto Scaling Group in private subnets processes requests. RDS in isolated subnets stores data. NAT Gateways (one per AZ for HA) in public subnets give private instances outbound internet. This architecture has been the AWS Well-Architected benchmark for web applications since 2015.
Internet→ ALB (public subnets, 3 AZs)→ EC2 ASG (private subnets)→ RDS (isolated subnets)
EC2 (private)→ outbound only → NAT Gateway (public)→ Internet Gateway→ Internet

Question 06
Storage S3 Object Lock S3 Lifecycle HIGH WEIGHT

A financial institution must store trade confirmation records with the following regulatory requirements from SEC Rule 17a-4:
REQ1. Records must be immutable for exactly 7 years from the date of trade — no one, including the AWS root account and the CEO of the company, can delete or overwrite them during this period.
REQ2. Records older than 90 days but less than 7 years must move to a lower-cost storage tier automatically while remaining immutable.
REQ3. After 7 years, records must be automatically deleted.
REQ4. The storage solution must be able to provide a WORM (Write Once Read Many) attestation letter to auditors.

Which configuration satisfies ALL four requirements?

🏗️ Architecture — S3 Object Lock Compliance + Lifecycle Tiering
Compliance mode = true immutability. Object Lock follows the object through every Lifecycle transition. Expiration fires automatically after 2,555 days.
  • AS3 Versioning + S3 MFA Delete + S3 Lifecycle policy to transition to Glacier after 90 days and expire after 7 years. MFA Delete prevents any deletion without a second factor, satisfying REQ1.
  • BS3 Object Lock in Governance mode with a 7-year retention period + S3 Lifecycle to transition to Glacier Flexible Retrieval after 90 days + S3 Lifecycle expiration after 7 years.
  • CS3 Object Lock in Compliance mode with a 7-year retention period + S3 Lifecycle policy transitioning objects to S3 Glacier Instant Retrieval after 90 days + S3 Lifecycle expiration rule at 2,555 days (7 years). Enable S3 Object Lock on the bucket (requires Versioning). AWS provides a WORM attestation letter for Compliance mode buckets.
  • DS3 Object Lock in Compliance mode + AWS Backup with a 7-year retention vault lock + S3 Lifecycle transition to Glacier after 90 days. AWS Backup Vault Lock provides the root-account-proof immutability required by SEC 17a-4.
Correct Answer: C — S3 Object Lock Compliance Mode + Lifecycle
S3 Object Lock Compliance mode: Once set, the retention period CANNOT be shortened or removed by any user — including root. Unlike Governance mode, there is no override permission ✅. S3 Lifecycle transitions to Glacier Instant Retrieval (or Flexible Retrieval) work WITH Object Lock — the lock persists across storage class transitions ✅. S3 Lifecycle expiration fires automatically after 2,555 days — deletes the object after the lock period expires ✅. AWS provides official WORM attestation documentation for Compliance mode that satisfies SEC 17a-4 auditors ✅.
Option A (MFA Delete): MFA Delete prevents accidental deletion and requires MFA for permanent deletion of versioned objects. HOWEVER, it can be disabled by the root account — it does NOT prevent a determined root user from disabling MFA Delete and then deleting records. SEC 17a-4 requires that even root cannot delete — MFA Delete does not meet this bar. Option B (Governance mode): Object Lock Governance mode allows users with the s3:BypassGovernanceRetention IAM permission to override the retention period. A privileged admin or the root account can bypass it. Compliance mode has NO bypass path. Option D: Using both S3 Object Lock Compliance AND AWS Backup Vault Lock is redundant and adds unnecessary complexity. AWS Backup Vault Lock is for backup vaults, not for S3 object storage. The question is about the primary record store, not a backup.
S3 Object Lock — Governance vs Compliance in one sentence each:
🔹 Governance mode: "Root and privileged IAM users can override the lock." Good for internal governance. Not SEC 17a-4 compliant.
🔹 Compliance mode: "Nobody can override the lock during the retention period — not root, not AWS Support." Required for regulated financial/healthcare WORM.
The keyword that forces Compliance mode: "SEC 17a-4", "FINRA", "HIPAA WORM", "even root account cannot delete", "regulatory immutability".
Bonus: S3 Object Lock requires S3 Versioning to be enabled on the bucket — you cannot enable Object Lock on a bucket without Versioning. This is a frequent trick in distractor options.
SEC 17a-4 in practice: Every broker-dealer registered with FINRA is required to store records in non-rewriteable, non-erasable format under SEC Rule 17a-4(f).   • DAction 1 → SQS FIFO Queue | Action 2 → Amazon EventBridge with 3 Lambda targets (EventBridge fan-out) | Action 3 → Amazon AppSync subscription triggered by Shopify GraphQL events.
Correct Answer: C — SQS FIFO + SNS fan-out + EventBridge partner source
SQS FIFO: Exactly-once delivery with deduplication ID. Message Group ID scopes ordering per product (orders for Product A don't block orders for Product B) ✅. SNS fan-out to SQS subscriptions: SNS publishes to multiple SQS queues simultaneously — email, push, and analytics services each consume from their own SQS queue at their own pace, completely decoupled ✅. EventBridge partner event source: Shopify is an official EventBridge partner — Shopify can send events directly to an EventBridge event bus. An EventBridge rule matches the "shipped" event pattern and invokes Lambda ✅. This is the canonical SNS-SQS fan-out architecture.
Option A: SQS Standard is wrong for Action 1 — Standard queues provide at-least-once delivery (duplicates possible) and best-effort ordering (order not guaranteed). Both properties violate REQ1. SNS does NOT have Shopify as a native partner event source — EventBridge does. Option B: Amazon MQ is correct for existing AMQP/STOMP applications migrating from ActiveMQ/RabbitMQ. For a new design, MQ adds operational complexity (manage brokers) without benefit. EventBridge for fan-out in Action 2 is technically possible but EventBridge is better suited for event routing with rules, not simple parallel fan-out to consumers. AppFlow is for no-code SaaS data ingestion flows (bulk data), not event-driven real-time triggers. Option D: AppSync is a GraphQL API service. It has no native integration to receive Shopify webhook events and trigger Lambda. Shopify webhooks → EventBridge (not AppSync).
The messaging service selector — 5 services, 5 distinct use cases:
🔹 SQS Standard: High-throughput, at-least-once, best-effort ordering. Use when duplicates are tolerable and order doesn't matter.
🔹 SQS FIFO: Exactly-once, strict ordering within message group. Throughput limit: 3,000 msg/s with batching. Use when order and exactly-once matter.
🔹 SNS: Push-based pub/sub. One publish → many subscribers simultaneously. Fan-out. No replay or buffering at SNS level (use SQS subscriptions for that).
🔹 EventBridge: Event bus with rules and filters. Partner event sources (Shopify, Zendesk, Stripe, etc.). Schema registry. Event replay. Best for event routing and SaaS integration.
🔹 Amazon MQ: Managed ActiveMQ/RabbitMQ. Use only when existing apps use AMQP/STOMP/OpenWire and you can't refactor to SQS/SNS.
Fan-out pattern: SNS topic → multiple SQS queues → multiple consumer services. This is a fundamental AWS architecture pattern that appears in multiple exam questions.
SNS-SQS fan-out in e-commerce: When an order is placed on a major retail platform, an SNS message is published to an "orders" topic. 7 SQS queues are subscribed: inventory reservation, payment processing, fraud detection, warehouse picking, email confirmation, loyalty points, and analytics. Each service consumes from its own queue at its own pace — a slow analytics service never delays inventory reservation. Each SQS queue acts as a buffer with configurable retry and dead-letter queue logic.

🏗️ Architecture — AppFlow (SaaS Sync), AppSync (GraphQL Real-Time), Cost Explorer + Budgets
AppFlow: no-code Salesforce→S3 nightly. AppSync: GraphQL subscriptions push DynamoDB changes to mobile clients instantly. Cost Explorer for 12-month trends; Budgets for $50K threshold alerts.
  • AChallenge 1 → Amazon AppFlow (no-code Salesforce → S3 scheduled flow) | Challenge 2 → AWS AppSync (GraphQL subscriptions, real-time DynamoDB resolvers) | Challenge 3 → AWS Cost Explorer (cost trend analysis, filtering by service/tag/account) + AWS Budgets (alert when EC2 spend > $50K/month).

AppFlow: No-code SaaS data integration service. Native connectors for Salesforce, Zendesk, Slack, ServiceNow, etc. Scheduled or event-triggered flows to/from S3, Redshift, Snowflake. No Lambda, no Glue, no code ✅. AppSync: Managed GraphQL service with native real-time subscriptions. When a DynamoDB item changes (via DynamoDB Stream resolver), AppSync pushes the update to all subscribed WebSocket clients instantly — no polling ✅. Cost Explorer + Budgets: Cost Explorer provides interactive charts for cost trends by service, tag, linked account, going back 12 months. AWS Budgets creates proactive alerts (email/SNS) when actual or forecasted spend exceeds a threshold ✅.
Option B: AWS Glue is a code-based ETL service (PySpark/Python). It requires writing code — the operations team has no coding skills, so this is wrong. API Gateway WebSocket is valid for real-time but requires building custom connection management, routing, and push logic — AppSync does this natively with GraphQL subscriptions and DynamoDB resolvers. Cost and Usage Report (CUR) is the raw, granular billing data export (CSV/Parquet to S3) — it requires Athena or QuickSight to query. It is NOT an interactive trend analysis tool. Option C: EventBridge for Salesforce → S3 would require: EventBridge partner source, Lambda to transform, Lambda to write to S3 — this is code-heavy and definitely not "no code." AWS Budgets is for alerts, NOT for trend analysis — it doesn't have a historical trend view. Option D: Amazon SNS sends push notifications (mobile push via APNs/FCM) — this is one-time push, not a GraphQL subscription model with real-time streaming data. SNS cannot subscribe mobile clients to DynamoDB change events directly with the rich filtering AppSync provides. CUR has no built-in budget alerting — that's what Budgets is for.
Cost management service roles — four tools, four distinct functions:
🔹 AWS Budgets: "Alert me when I'm about to spend too much." Proactive threshold alerts. Can alert on cost, usage, RI utilisation, Savings Plans coverage. Action-based (can automatically apply IAM policies or SCPs when budget is breached).
🔹 AWS Cost Explorer: "Show me where my money went." Interactive console, 12-month history, filter by service/tag/account/region. Forecasting built-in.
🔹 AWS Cost and Usage Report (CUR): "Give me raw billing data to load into my data warehouse." Hourly/daily granularity. S3 export. Athena/Redshift/QuickSight for analysis. Most detailed of all tools.
🔹 Savings Plans: "I commit to $X/hour of compute for 1 or 3 years and get a discount." Compute Savings Plans (any EC2/Fargate/Lambda) vs EC2 Instance Savings Plans (specific family).

Question 15
Cost Optimisation Savings Plans EC2 Auto Scaling Lambda SCENARIO · HIGH WEIGHT

A company runs the following workloads on AWS and wants to maximise cost savings. A Solutions Architect is asked to recommend the optimal combination of purchasing options and tools:

Workload A: A 3-tier web application with a stable baseline of 10 EC2 c5.2xlarge instances (24/7, all year), plus spiky additional capacity (up to 50 instances) during business hours. The team may want to change instance families in 6 months as workload evolves.
Workload B: A batch ETL pipeline running on Fargate tasks for 8 hours per night. Jobs are fault-tolerant — if a task is interrupted and retried, the pipeline still completes correctly.
Workload C: A Lambda-based microservice that processes 200 million invocations per month with 512MB memory and 300ms average duration. This is a committed, steady workload.
Workload D: The finance team needs a tool that automatically identifies which EC2 and Fargate resources are over-provisioned and provides right-sizing recommendations without manual analysis.

Which combination of actions is MOST cost-optimised?

🏗️ Architecture — Compute Savings Plans + Spot + Fargate Spot + Compute Optimizer
Compute SP covers EC2+Fargate+Lambda with full instance-family flexibility. Fargate Spot saves 70% for fault-tolerant nightly ETL. Compute Optimizer uses ML to right-size — not Trusted Advisor.
  • AA → Standard Reserved Instances (1yr, c5.2xlarge, all upfront) for all 60 instances + On-Demand for spiky additional capacity | B → On-Demand Fargate | C → no discount (Lambda has no Reserved capacity) | D → AWS Cost Explorer right-sizing recommendations.
  • BA → Compute Savings Plans (1yr) for the 10 baseline + Spot Instances for the spiky 50 + On-Demand for overflow | B → Spot Fargate tasks | C → Compute Savings Plans also cover Lambda (compute Savings Plans apply to Lambda) | D → AWS Trusted Advisor cost optimization checks.
  • CA → EC2 Instance Savings Plans (1yr, c5 family) for baseline + On-Demand for spiky | B → On-Demand Fargate (Spot Fargate doesn't exist) | C → Compute Savings Plans for Lambda | D → AWS Compute Optimizer.
  • DA → Compute Savings Plans (1yr, covers baseline EC2 + allows instance family change) for the 10 stable instances + EC2 Spot Instances in the Auto Scaling Group for the spiky 50 (fault-tolerant web tier can handle interruptions with proper ASG mixed instances policy) | B → Fargate Spot (up to 70% cheaper; nightly ETL tasks are fault-tolerant = perfect Spot use case) | C → Compute Savings Plans (explicitly cover Lambda invocations, offering up to 17% discount) | D → AWS Compute Optimizer (uses ML to analyse CloudWatch utilisation and recommends right-sizing for EC2, Lambda, Fargate, EBS).
Correct Answer: D — Compute Savings Plans + Spot (EC2 + Fargate) + Compute Optimizer
Compute Savings Plans for Workload A baseline: Unlike EC2 Instance Savings Plans (locked to c5 family in one Region), Compute Savings Plans apply to any EC2 instance family, size, Region, and OS — perfect for a team expecting to change instance families ✅. EC2 Spot Instances for spiky capacity: Spot can save 70-90% vs On-Demand. Web tier with Auto Scaling using mixed instances policy (Spot + On-Demand) is a well-established AWS pattern — interruptions result in a brief capacity reduction, not a failure ✅. Fargate Spot: Fargate has a Spot pricing tier for interruptible tasks. Nightly ETL pipelines that are fault-tolerant (retry-safe) = textbook Fargate Spot use case, saving up to 70% ✅. Compute Savings Plans for Lambda: This is the key exam fact — Compute Savings Plans DO cover Lambda. A 1-year commitment at $X/hour covers Lambda invocations, Fargate, and EC2 compute ✅. Compute Optimizer: ML-driven right-sizing recommendations for EC2, Lambda, EBS, ECS on Fargate, and Auto Scaling Groups — no manual analysis ✅.
Option A: Reserving all 60 instances (including the spiky 50) with Standard RIs wastes money — you'd pay Reserved prices for capacity you only use during business hours. Standard RIs are locked to a specific instance type and Region; if the team switches from c5 to c6i in 6 months, the RIs become stranded. Also, Cost Explorer has a right-sizing tab, but it provides static analysis reports — not ML-based continuous recommendations like Compute Optimizer. Option B: Trusted Advisor's cost optimisation checks are limited to simple heuristics (underutilised EC2, idle load balancers) — they are not ML-driven right-sizing like Compute Optimizer. The Trusted Advisor checks are a starting point, not a replacement for Compute Optimizer's granular analysis. Option C: EC2 Instance Savings Plans are locked to a specific EC2 instance family (c5) in a specific Region. If the team wants to change to c6i or m6i in 6 months, the Savings Plan no longer applies. Compute Savings Plans are more flexible. Also, "On-Demand Fargate (Spot Fargate doesn't exist)" in option C is WRONG — Fargate Spot is a real product and saves up to 70%. This is a classic exam trap.
Savings Plans vs Reserved Instances — the two most confused cost tools:
🔹 Standard Reserved Instances: Specific instance type + Region + OS + tenancy. Highest discount (~72%). No flexibility. Good for DB instances (RDS) where type rarely changes.
🔹 Convertible Reserved Instances: Same as Standard but can be exchanged for different type/family. Lower discount (~54%). Good when type might change.
🔹 EC2 Instance Savings Plans: Commit $/hr to a specific EC2 instance family in one Region. Any size/OS. ~72% discount. Flexible on size, locked on family+Region.
🔹 Compute Savings Plans: Commit $/hr to ANY compute (EC2 any type/Region + Fargate + Lambda). ~66% EC2 discount. Maximum flexibility.
The keyword that forces Compute Savings Plans: "may change instance family", "multi-region", "Lambda + EC2 + Fargate combined commitment".
Fargate Spot exists and is heavily tested. Any fault-tolerant container workload running on Fargate should use Fargate Spot.
AWS Compute Optimizer vs Trusted Advisor — one sentence each: Compute Optimizer uses CloudWatch metrics from the last 14 days (or 93 days with enhanced infrastructure metrics) and machine learning to recommend the right EC2 instance type, Lambda memory size, EBS volume type, and ECS task CPU/memory. Trusted Advisor checks 5 categories (cost, security, fault tolerance, performance, service limits) using simple rule-based thresholds — no ML, no resource-specific granularity. For the exam: "ML right-sizing recommendation" = Compute Optimizer. "Broad cost/security/fault tolerance check" = Trusted Advisor.

🚨 Top 15 Exam Traps — Session 3

❌ VPC Peering for 100+ VPCs → Use Transit Gateway (TGW). Peering is non-transitive and doesn't scale.
❌ TGW for overlapping CIDRs → Only PrivateLink solves overlapping CIDRs. TGW requires unique CIDRs.
❌ CloudFront for UDP/TCP acceleration → CloudFront is HTTP/HTTPS only. Use Global Accelerator for UDP/TCP.
❌ Global Accelerator for content caching → GA accelerates traffic, does NOT cache. CloudFront caches.
❌ Route 53 Latency routing for data residency → Latency routes by speed, not geography. Use Geolocation for GDPR.
❌ Local Zones for 5G MEC workloads → Local Zones are metro AWS extensions. Wavelength is inside 5G carrier networks.
❌ S3 Object Lock Governance for SEC 17a-4 → Governance has an override bypass. Only Compliance mode is truly immutable.
❌ EFS for Windows SMB workloads → EFS is NFS (Linux). Use FSx for Windows File Server for SMB/ACLs/DFS.
❌ EFS for HPC/rendering scratch space → EFS can't approach hundreds of GB/s. Use FSx for Lustre for HPC.
❌ DataSync as an NFS gateway (ongoing) → DataSync is for migration/sync tasks. Storage Gateway File Gateway is the live NFS proxy.
❌ Lambda for 2-hour batch jobs → Lambda max is 15 minutes. Use AWS Batch for multi-hour workloads.
❌ AWS Outposts in air-gapped environments → Outposts requires connectivity to parent AWS Region. EKS Anywhere supports fully disconnected.
❌ SQS Standard for exactly-once, ordered processing → Standard = at-least-once, best-effort order. FIFO = exactly-once, strict order.
❌ Fargate Spot doesn't exist → It does. Up to 70% cheaper. Use for fault-tolerant container workloads.
❌ Compute Savings Plans don't cover Lambda → They do. Compute SP covers EC2 + Fargate + Lambda. This is a frequently tested fact.
