🏛️ Block 1 — Governance & Multi-Account

⏱️ Time: 00:00 – 00:25 4 Services · ~6 min each
AWS Organizations
Multi-Account Management · Central Governance · SCPs
Governance HIGH EXAM WEIGHT

AWS Organizations is the root service for multi-account strategy. Think of it as the parent corporation that owns all the child AWS accounts. It lets you group accounts into Organizational Units (OUs), apply central billing, and — critically for the exam — attach Service Control Policies (SCPs) to restrict what anyone in that OU can do, no matter what their IAM policy says.

SCPs
Service Control Policies
Maximum permission guardrails — IAM can only grant UP TO what SCP allows
OUs
Org Units
Logical groupings: Dev, Test, Prod, Security, Sandbox
1 Bill
Consolidated Billing
Combined usage = volume discounts on EC2, S3, etc.
🚨 #1 Exam Trap — SCPs do NOT grant permissions. SCPs are guardrails, not grants. An SCP that says "Allow S3" doesn't give any account access to S3 — it just says "don't restrict S3". The IAM policy in the account still has to explicitly grant it. If an SCP denies an action, even a root user in the member account cannot perform it.
🏢 Real-World Use Case
Enterprise Bank — 200 AWS Accounts, One Control Plane

A large bank has separate AWS accounts per product team (Credit Card, Loans, Payments), per environment (Dev/Test/Prod), and for shared services (Security, Logging). AWS Organizations groups them into OUs. The Security OU has an SCP that says "No account in this OU can ever disable CloudTrail or GuardDuty" — this blocks even an account admin from accidentally (or maliciously) turning off audit logging. Consolidated billing saves them ~18% via aggregated EC2 Reserved Instance sharing.

🎯 SAA-C03 Exam Tips:
• "Prevent all accounts from leaving the organisation" → SCP on root OU
• "Restrict EC2 instance types to t3 and m5 only across all dev accounts" → SCP on Dev OU
• "Centralise billing and get volume discounts" → AWS Organizations consolidated billing
• SCPs affect all principals in the account including the root user — this is unique and exam-tested
AWS Control Tower
Landing Zone Automation · Guardrails · Account Factory
Governance Multi-Account

If Organizations is the framework, Control Tower is the automated setup wizard on steroids. It builds a Landing Zone — a pre-configured multi-account environment with security baselines, centralized logging, and audit accounts already wired up. It uses Guardrails (preventive via SCPs, or detective via AWS Config rules) to enforce policies across all accounts.

Mental Model — Organizations vs Control Tower:
Think of Organizations as the raw structural framework (like building a skeleton). Control Tower is the fully furnished house built on that skeleton — it automatically sets up Organizations, SSO, CloudTrail, Config, and the Log Archive account for you. On the exam: "set up a new multi-account environment with security best practices automatically" = Control Tower.

Guardrails (Preventive)SCPs that block non-compliant actions before they happen"prevent" actions across accounts
Guardrails (Detective)AWS Config rules that detect and flag policy violations"detect" non-compliance, "audit"

🚀 Real-World Use Case
SaaS Startup Scaling from 1 to 50 Accounts

A fast-growing SaaS company goes from 1 AWS account to needing 50 (one per enterprise customer for data isolation). Without Control Tower, this would take weeks per account. With Account Factory, each new customer account is provisioned in minutes with VPC, CloudTrail, Config, and IAM baselines pre-applied. Guardrails ensure no one accidentally makes an S3 bucket public in any customer account.

🎯 SAA-C03 Exam Tips:
• "Automatically provision new AWS accounts with security baselines" → Control Tower Account Factory
• "Detect if MFA is disabled across all accounts" → Control Tower Detective Guardrail
• Control Tower sits on top of Organizations — it needs Organizations to function
AWS Service Catalog
Self-Service Portal · Approved Products · IT Governance
Governance IaC

Service Catalog solves a real enterprise problem: "How do I let developers self-serve AWS resources without them spinning up anything they want?" IT admins create a portfolio of approved CloudFormation templates (products), and end users can only deploy from that approved catalog — like an app store for your infrastructure.

👨‍💼 Admin side: Creates "Products" (CloudFormation templates), groups them into "Portfolios", sets constraints (e.g. only launch in us-east-1, only use t3.micro), and shares portfolios with specific OUs or accounts.
👩‍💻 End-user side: Opens the self-service portal, picks "Web App Stack" or "Data Science Notebook", clicks launch, and gets exactly what the template defines — no access to the underlying CloudFormation console needed.
🏭 Real-World Use Case
Insurance Company — Compliant Infrastructure in One Click

An insurance company's compliance team mandates that every database must be encrypted at rest, every EC2 must have a specific security agent installed, and all resources must be tagged with a cost centre. They build 5 approved CloudFormation templates and publish them to Service Catalog. Developer teams pick from the catalog — they cannot deviate from the approved patterns, so every resource is automatically compliant. Audit time goes from weeks to minutes.

🎯 SAA-C03 Exam Tips:
• "Allow developers to self-service resources WITHOUT giving them CloudFormation or IAM access" → Service Catalog
• "Enforce tagging and region constraints on all provisioned resources" → Service Catalog constraints
• Service Catalog ≠ Marketplace. Catalog is YOUR approved templates. Marketplace is third-party software.

E-commerce Company Saves $40K/Month

An e-commerce company runs Trusted Advisor and discovers: 12 idle EC2 instances from a cancelled project ($8K/mo), 40 unattached EBS volumes ($3K/mo), 3 RDS databases on Multi-AZ in dev environments ($12K/mo — overkill for dev), and Reserved Instance coverage at 34% for predictable workloads ($17K/mo savings potential). Total: ~$40K/month in waste identified in 5 minutes of reading the report.

🎯 SAA-C03 Exam Tips:
• "Identify underutilised resources and cost-saving opportunities automatically" → Trusted Advisor
• "Receive alerts when approaching service limits" → Trusted Advisor + CloudWatch Events
• "Check if root account has MFA enabled" → Trusted Advisor (free security check)
• Don't confuse with Cost Explorer (shows spending history) — TA gives actionable recommendations

Amazon Redshift
Cloud Data Warehouse · Petabyte Scale · OLAP · BI Analytics
Data Warehouse Analytics

Redshift is AWS's massively parallel processing (MPP) data warehouse. It's built for OLAP — Online Analytical Processing — meaning analytical queries across billions of rows. Not for transactional (OLTP) workloads. The killer feature: it stores data in columnar format which compresses beautifully and makes aggregations blazing fast.

OLTP vs OLAP — The Core Distinction (exam-critical):
OLTP (Online Transaction Processing) = fast inserts/updates for live transactions → use RDS, Aurora, DynamoDB
OLAP (Online Analytical Processing) = complex queries on huge historical datasets → use Redshift
Question trigger words: "business intelligence", "data warehouse", "aggregate millions of rows", "analytics dashboard" → Redshift.

📊 Real-World Use Case
Netflix-Style Streaming Analytics Platform

A video streaming company needs to answer "What content drove the most completions in the last 90 days, broken down by geography, device, and subscription tier?" — against 4 billion viewing events. This is classic OLAP. They load daily event batches from S3 into Redshift using COPY, keep 3 years of data, and run Tableau / QuickSight dashboards on top. Redshift Spectrum lets them also query raw JSON logs sitting in the data lake without ETL-loading them first.

🎯 SAA-C03 Exam Tips:
• "Data warehouse", "business intelligence", "analytics on billions of rows" → Redshift
• "Query S3 data lake using SQL" → Redshift Spectrum or Athena (Athena is serverless one-off, Spectrum integrates with existing warehouse)
• "Real-time transactions" → NOT Redshift, use RDS/Aurora
• Redshift uses leader node + compute nodes — leader distributes queries, nodes execute in parallel
Amazon Neptune
Fully Managed Graph Database · Relationships · Connections
Graph DB Relationships

Neptune is AWS's purpose-built graph database. It excels at storing and traversing relationships between entities. If your data looks like a network of connected nodes (people, accounts, products, pages), and your queries are about how things connect rather than what they are, Neptune is the right choice.

Gremlin (Property Graph): Great for social networks, fraud detection, recommendation engines. "Find all friends-of-friends who bought X."
SPARQL (RDF): Great for knowledge graphs, semantic data, linked data. Used by enterprises for master data management.
🚨 Exam Keyword Trigger — When to pick Neptune:
• "Social network" — relationships between users
• "Fraud detection" — detect patterns in transaction relationships
• "Knowledge graph" — Wikipedia-style entity connections
• "Recommendation engine" — "users like you also liked…"
• "Highly connected data" — any time the question emphasises relationships over data points
🕵️ Real-World Use Case
Financial Services Fraud Detection

A payment network needs to detect fraud rings — groups of fake accounts that transfer money between each other to launder funds. In a relational DB, this requires expensive multi-table JOINs across millions of rows. In Neptune, each account is a node, each transaction is an edge. Querying "find all accounts within 3 hops of this suspicious account that were created in the same 24-hour window" is a single graph traversal — completing in milliseconds instead of minutes.

🎯 SAA-C03 Exam Tips:
• "Graph database", "highly connected data", "relationships between entities" → Neptune
• Neptune is Multi-AZ by default with up to 15 read replicas
• Don't confuse with DynamoDB (key-value/document) or ElastiCache (in-memory cache)
Amazon QLDB
Quantum Ledger Database · Immutable · Cryptographically Verifiable
Ledger Immutable

QLDB is a fully managed ledger database that provides a transparent, immutable, and cryptographically verifiable transaction log. Every change to data is recorded in a journal that cannot be altered or deleted. You can mathematically prove the complete history of any record — who changed what, when, and in what order.

Mental Model — QLDB vs Blockchain:
Both track history immutably. The key difference: QLDB is centralised (owned by one organisation, no consensus mechanism needed) and therefore much faster and simpler. Blockchain is decentralised (multiple parties must agree). Exam keyword: "single trusted authority" → QLDB. "Multiple parties, no central trust" → Managed Blockchain.
⚖️ Real-World Use Case
Motor Vehicle Registry — Tamper-Proof Car History

A government vehicle licensing agency records every ownership transfer, title change, and lien on every vehicle. With a traditional database, a corrupt insider could alter records. With QLDB, every change is permanently written to the journal with a cryptographic hash. An auditor can verify that the complete history of vehicle #ABC-1234 has never been tampered with — the math proves it. Used by CarFax-style services to guarantee vehicle history integrity.

🎯 SAA-C03 Exam Tips:
• "Immutable", "audit trail", "cryptographically verifiable history", "tamper-proof" → QLDB
• "Supply chain tracking by single company" → QLDB
• "Multiple untrusted parties need to agree on records" → Amazon Managed Blockchain
• QLDB uses PartiQL (SQL-compatible) — no need to learn a new query language

🔧 Block 3 — Management & Observability

⏱️ Time: 00:45 – 01:10 5 Services · ~5 min each
AWS Management Console
Web UI · Point-and-Click · All Services · Dashboard
Foundation Beginner

The Management Console is your web-based control plane for all of AWS. If you're new to AWS, this is where you start — it's the visual interface to every service. From an exam perspective, it's the entry point for everything, but the real insight is knowing what it cannot do that CLI, SDK, or CloudFormation can.

Console Strengths: Visual exploration, one-time manual tasks, monitoring dashboards, reading logs, setting up billing alerts. Great for learning and ad-hoc work.
Console Limitations: Cannot automate. Cannot reproduce steps exactly. For automation → CLI/SDK. For repeatable infrastructure → CloudFormation/CDK. Console is a human interface, not a machine interface.
🎯 SAA-C03 Exam Tips:
• The console is rarely the answer in exam scenarios — it's the starting point
• "Automate account provisioning across 200 accounts" → Not the console, use CLI + Organizations
• Know that IAM policies apply to the Console, CLI, and SDK identically — no difference in permissions
• AWS Console Mobile App exists — allows monitoring on the go (occasionally appears in questions)
AWS CloudFormation
Infrastructure as Code · Stacks · Templates · Drift Detection
IaC HIGH EXAM WEIGHT

CloudFormation is AWS's native Infrastructure as Code (IaC) service. You write a template (JSON or YAML) describing ALL the resources you want, and CloudFormation creates, updates, and deletes them as a single atomic unit called a Stack. One template can spin up an entire VPC with subnets, EC2 fleet, RDS, ALB, security groups — in the right order, every time.

🔄 Real-World Use Case
SaaS Company — Environment Parity in 20 Minutes

A SaaS company uses a single CloudFormation template to create Dev, Test, and Prod environments with identical topology. Parameters in the template (like instance type and Multi-AZ flag) differ per environment. StackSets deploy the same infrastructure config to 5 regions simultaneously for a new feature rollout. When a developer accidentally deletes a security group in Test, Drift Detection immediately flags it. The entire environment can be torn down and recreated in 20 minutes.

🎯 SAA-C03 Exam Tips:
• "Deploy same infrastructure across multiple accounts/regions" → CloudFormation StackSets
• "Preview changes before applying" → Change Sets
• "Detect manual changes to CloudFormation-managed resources" → Drift Detection
• CloudFormation is FREE — you pay only for the resources it creates, not the service itself
Amazon CloudWatch
Metrics · Logs · Alarms · Events · Dashboards
Observability HIGH EXAM WEIGHT

CloudWatch is the central observability platform for AWS. It collects metrics from every AWS service, ingests logs from your applications, triggers alarms when thresholds are breached, and fires events to automate responses. If something goes wrong in your AWS environment, CloudWatch is usually the first place you look — and the first responder.

CloudWatch Core Capabilities

AlarmsTrigger action (SNS, Auto Scaling, EC2 action) when metric crosses threshold"notify", "scale when CPU > 80%"

🚨 Exam Trap — Default vs Detailed Monitoring:
EC2 metrics are collected every 5 minutes by default (free). Enable Detailed Monitoring for 1-minute intervals (extra cost). If an exam question asks "how do I get EC2 metrics more frequently" → enable Detailed Monitoring. Also: Memory and disk utilisation are NOT collected by default — you need to install the CloudWatch Agent on the instance.
🚨 Real-World Use Case
Automated Self-Healing Production System

An e-commerce platform sets up a CloudWatch Alarm on the UnhealthyHostCount metric for their ALB. When it crosses 2, the alarm triggers an SNS notification to the on-call team AND triggers an EventBridge rule that calls a Lambda function to automatically replace the unhealthy EC2 instances. CloudWatch Logs Insights lets the SRE team query across 10 million log lines in seconds to find the root cause. MTTR drops from 45 minutes to 8 minutes.

🎯 SAA-C03 Exam Tips:
• "Monitor CPU and trigger auto scaling" → CloudWatch Alarm → Auto Scaling Policy
• "Memory utilisation metrics for EC2" → CloudWatch Agent (not available by default)
• "Centralize logs from multiple EC2 instances" → CloudWatch Logs Agent / Unified Agent
• "React to AWS service events automatically" → Amazon EventBridge (formerly CloudWatch Events)
AWS CloudTrail
API Audit Log · Who Did What · When · From Where
Security Audit HIGH EXAM WEIGHT

CloudTrail records every API call made to your AWS account — whether from the Console, CLI, SDK, or another AWS service. It answers the forensic questions: "Who deleted that S3 bucket? When did someone disable GuardDuty? Who modified this security group at 2am?" It's the foundational audit and compliance service.

Management Events (default ON): Control plane operations — creating/deleting resources, changing IAM policies, starting/stopping EC2. These are always logged at no cost.
Data Events (off by default): Data plane operations — S3 object-level reads/writes, Lambda invocations, DynamoDB item-level activity. High volume; extra cost to log.
🚨 CloudTrail vs CloudWatch — Critical Distinction:
CloudTrail = WHO did WHAT action (API-level audit) → "who deleted the EC2?"
CloudWatch = HOW is the system performing (metrics/logs) → "CPU is at 90%"
These appear together in nearly every security scenario. CloudTrail logs feed INTO CloudWatch Logs for alerting. Know both directions.
🔍 Real-World Use Case
Security Incident Investigation — "Who Exfiltrated Our Data?"

A healthcare company suspects a rogue employee exfiltrated patient data from S3. With CloudTrail Data Events enabled on that bucket, the security team queries CloudTrail Lake (the managed analytics layer) and within 10 minutes identifies: User arn:aws:iam::123:user/john.doe ran GetObject on 40,000 objects between 11pm and 2am from an IP in a foreign country. The trail is also archived to a separate security account's S3 bucket with MFA Delete enabled — even a compromised admin can't destroy the evidence.

🎯 SAA-C03 Exam Tips:
• "Who made this API call / who deleted this resource" → CloudTrail
• "Tamper-proof audit logs" → Store CloudTrail logs in S3 with MFA Delete + log file validation
• "Detect root account login" → CloudTrail + CloudWatch Alarm on root login events
• CloudTrail delivers logs to S3 within 15 minutes; near-real-time via CloudWatch Logs integration
AWS Systems Manager
Operational Hub · Patch Manager · Session Manager · Parameter Store
Operations Hybrid

Systems Manager (SSM) is a unified operations platform for managing your fleet of EC2 instances and on-premises servers. It replaces the need to SSH into individual machines, centralises configuration management, automates patching, and — critically — provides Parameter Store for securely storing secrets and config values.

🚨 Exam Trap — Parameter Store vs Secrets Manager:
Both store secrets, but: Parameter Store (SSM) = free for standard params, no auto-rotation, great for config + secrets. 🛠️ Real-World Use Case
Zero-Bastion Secure Operations for 2,000 EC2 Instances

A financial services firm runs 2,000 EC2 instances with no public IPs, no open port 22, and no bastion hosts. All operational access goes through Session Manager — audit logs of every session are sent to CloudTrail and CloudWatch. Monthly patching happens via a Patch Manager maintenance window at 2am Sunday. Database passwords are stored in Parameter Store (SecureString with KMS) and pulled by applications at runtime — never hardcoded in code or AMIs. Compliance audit passes in one day instead of two weeks.

📦 Block 4 — Containers & Modern Development

⏱️ Time: 01:10 – 01:35 6 Services · ~4 min each
Amazon EKS
Elastic Kubernetes Service · Managed Control Plane · K8s on AWS
Containers Kubernetes

EKS is AWS's managed Kubernetes service. AWS runs and maintains the Kubernetes control plane (API server, etcd) — you just manage worker nodes. It gives you the power of Kubernetes with dramatically less operational overhead. Worker nodes run on EC2 (you manage) or Fargate (AWS manages — fully serverless K8s).

EKS on EC2: You control the worker nodes — choose instance type, AMI, scaling group. More control, more responsibility. Good for GPU workloads, custom networking.
EKS on Fargate: No worker nodes to manage. You define pods, AWS handles the compute. Fully serverless Kubernetes — ideal for variable workloads.
🐳 Real-World Use Case
Microservices Platform at Scale

A ride-sharing company runs 200+ microservices (pricing, matching, mapping, notifications) as containerised workloads. EKS lets their platform team standardise deployments with Helm charts, use Kubernetes-native autoscaling (HPA/KEDA), and integrate with AWS-native services (ALB Ingress Controller, IAM Roles for Service Accounts). The control plane is 100% managed by AWS — no more 3am etcd upgrades.

🎯 SAA-C03 Exam Tips:
• "Run Kubernetes on AWS without managing the control plane" → Amazon EKS
• "Serverless Kubernetes pods" → EKS on Fargate
• "ECS vs EKS" → ECS is simpler (AWS-native orchestration), EKS is K8s-compatible (portability). Exam prefers ECS for simplicity questions.
Amazon ECS Anywhere
Run ECS on Your Own Hardware · On-Prem Containers · Hybrid
Hybrid Containers

ECS Anywhere extends Amazon ECS to run containers on your own on-premises servers, edge hardware, or any VM — not just AWS. You install the ECS Agent and SSM Agent on your machines, register them as ECS external instances, and manage them from the same ECS console alongside your cloud workloads. One control plane, everywhere.

Why does this exist? Some workloads legally or practically cannot move to the cloud (data sovereignty, ultra-low latency, existing hardware investments). ECS Anywhere lets you use AWS tooling and pipelines for those workloads without moving the compute.
🏭 Real-World Use Case
Factory Floor Automation — AWS Orchestration On-Site

A manufacturing company runs ML-based quality inspection on the factory floor — cameras capture product images and a containerised model checks for defects in real time. The compute must be on-premises (sub-5ms latency, no internet dependency). With ECS Anywhere, the quality inspection containers are deployed from the same ECS console and CI/CD pipeline as their cloud services. Updates roll out via ECR image push, and CloudWatch collects metrics from on-prem instances.

🎯 SAA-C03 Exam Tips:
• "Run ECS workloads on-premises with AWS management" → ECS Anywhere
• "Kubernetes equivalent" → EKS Anywhere (same concept, K8s flavour)
• Requires: ECS Agent + SSM Agent on each on-prem node. Nodes must reach SSM endpoints.
Amazon EKS Distro
K8s Distribution · Self-Managed · Same EKS Binaries
Kubernetes Self-Managed

EKS Distro (EKS-D) is the open-source Kubernetes distribution that powers Amazon EKS — packaged for you to run anywhere: on-premises, on other clouds, or in your own data centre. It includes the same Kubernetes version, dependencies, patches, and test results that AWS uses in production EKS. You get AWS's battle-tested Kubernetes, with full control over your own infrastructure.

EKS vs EKS Anywhere vs EKS Distro — The Trilogy Explained:
EKS = AWS runs everything (cloud only) — least effort
EKS Anywhere = AWS tooling on your hardware — medium effort, hardware stays on-prem
EKS Distro = just the K8s distribution — you install, manage, and operate it yourself anywhere
Think: cloud → hybrid managed → self-managed. Each step gives more control, more responsibility.
🎯 SAA-C03 Exam Tips:
• "Use the same K8s version AWS uses, but manage the cluster yourself" → EKS Distro
• "Kubernetes on-premises with AWS-managed tooling" → EKS Anywhere
• EKS-D is open source; available via GitHub, compatible with standard K8s tools
Amazon ECR
Elastic Container Registry · Private Docker Registry · Image Scanning
Containers Registry

ECR is AWS's fully managed private Docker/OCI container image registry. It's the place you store container images before deploying them with ECS, EKS, or Lambda container images. Integrates natively with IAM for access control, scans images for vulnerabilities, and eliminates the need to run your own registry infrastructure.

Key Features: Private repositories (IAM-controlled), Public Gallery (public.ecr.aws), image vulnerability scanning (Enhanced via Inspector), lifecycle policies to auto-delete old images, cross-region and cross-account replication.
Why ECR over Docker Hub? ECR is inside the AWS network — ECS/EKS pulls happen at network speed with no internet hop. IAM authentication — no separate credentials. Lifecycle rules — no manual cleanup. Free data transfer to ECS/EKS.
🐋 Real-World Use Case
CI/CD Pipeline — Secure, Scanned Images Every Commit

A fintech company's pipeline: developer pushes code → CodeBuild builds and tags the Docker image → image pushed to ECR → ECR Enhanced Scanning (powered by Inspector) checks for CVEs → if no critical vulnerabilities, CodeDeploy triggers ECS blue/green deployment with the new image. The entire pipeline takes 8 minutes. Lifecycle policies automatically delete images older than 90 days. Cross-region replication ensures DR region always has the latest image.

🎯 SAA-C03 Exam Tips:
• "Private container image registry on AWS" → ECR
• "Scan container images for vulnerabilities" → ECR Enhanced Scanning (Amazon Inspector)
• "Clean up old container images automatically" → ECR Lifecycle Policies
• ECR is the source; ECS/EKS is the destination — they work as a pair
AWS AppSync
Managed GraphQL API · Real-Time Data · Offline Sync
GraphQL Real-Time

AppSync is a fully managed GraphQL API service. It lets frontend clients request exactly the data they need (no over-fetching), subscribe to real-time data changes via WebSockets, and sync data for offline-capable mobile apps. Think of it as the API gateway specifically designed for modern frontend applications.

GraphQL in 30 Seconds: Instead of calling 5 REST endpoints to build a screen, a GraphQL client sends one query describing exactly what it needs. The GraphQL server stitches the response together. Less round trips, smaller payloads, faster apps. AppSync connects that GraphQL API to DynamoDB, Lambda, RDS, Elasticsearch, HTTP endpoints — all in one schema.
💬 Real-World Use Case
Collaborative Document Editor — Real-Time Sync

A team productivity app (like Notion) uses AppSync: when User A types a paragraph, AppSync publishes the change via a GraphQL subscription and all connected clients receive the update via WebSocket within milliseconds. When User B goes offline, AppSync's offline sync engine queues their changes locally. When they reconnect, AppSync applies conflict resolution logic and syncs everyone back to a consistent state. The backend is DynamoDB — all wired through AppSync resolvers.

🎯 SAA-C03 Exam Tips:
• "GraphQL API", "real-time data sync for mobile/web", "offline sync" → AppSync
• "REST API" → API Gateway. "GraphQL" → AppSync. Know the distinction.
• AppSync can connect to DynamoDB, Lambda, RDS Data API, OpenSearch, HTTP without any custom code in resolvers (using VTL or JS)
AWS Proton
Self-Service Infrastructure Templates · Platform Engineering · IaC Vending
Platform Eng DevOps

AWS Proton is designed for platform engineering teams who need to give developers standardised, self-service infrastructure templates for containerised and serverless workloads. Think of it as Service Catalog but specifically designed for modern microservice architectures — with built-in CI/CD pipeline provisioning and environment management.

Service Catalog vs AWS Proton:
Service Catalog = general self-service for any AWS resource (VPCs, databases, EC2 fleets)
AWS Proton = specialised for containerised/serverless app deployment templates with pipeline integration. Proton understands the concept of services (code + infrastructure + pipeline) as a unit.
🏗️ Real-World Use Case
Internal Developer Platform at Scale

A bank's platform team creates two Proton templates: "Fargate Microservice" and "Lambda API." Each template bundles the ECS/Lambda infrastructure, VPC config, CodePipeline, and CloudWatch dashboards. When a product team says "I need a new payments microservice," they pick "Fargate Microservice" from Proton, fill in name/repository details, and in 10 minutes have a production-grade service with deployment pipeline. Platform team updates the template once — all services using it can be updated centrally.

🎯 SAA-C03 Exam Tips:
• "Self-service infrastructure templates for containerised/serverless apps with CI/CD" → AWS Proton
• "Standardise microservice deployments for developer teams" → AWS Proton
• Proton is a newer service; on the exam it typically appears in "platform engineering" or "DevOps automation" scenarios

🚨 Direct Connect vs VPN — The Critical Exam Decision:
Direct Connect: Physical connection, weeks to provision, expensive, consistent bandwidth, NOT encrypted by default
Site-to-Site VPN: Over internet, minutes to provision, cheap, variable latency, encrypted by default (IPSec)

🏦 Real-World Use Case
Investment Bank — Migrating 200TB with Consistent Performance

An investment bank needs to migrate 200TB of trading data to S3 and run daily batch jobs that transfer 5TB of market data between their data centre and AWS Redshift. Over the internet this is wildly unpredictable — latency spikes kill their SLA. With a 10 Gbps Direct Connect connection (via a Public VIF to S3), the daily 5TB transfer takes 70 minutes at consistent speed. Trading applications connect via a Private VIF to a dedicated VPC. A backup Site-to-Site VPN kicks in if the Direct Connect circuit fails.

🎯 SAA-C03 Exam Tips:
• "Consistent, dedicated bandwidth between on-prem and AWS" → Direct Connect
• "Encrypt Direct Connect traffic" → Add an IPSec VPN on top of Direct Connect
• "Redundant Direct Connect" → Use multiple connections or Direct Connect Gateway for multi-region
• Direct Connect does NOT provide encryption — exam trap!

Remote Workforce — Secure Access to Private Cloud Resources

VMware Cloud on AWS
VMware SDDC on AWS Bare Metal · Seamless Hybrid · vSphere
Hybrid Migration

VMware Cloud on AWS runs VMware's Software-Defined Data Center (SDDC) stack — vSphere, vSAN, NSX — directly on dedicated AWS bare metal hardware. You manage it with the same VMware tools you already know. This is the premier solution for enterprises doing large-scale VMware migrations to AWS without re-platforming every workload.

The "Lift and Shift VMware" Play: Most enterprises run thousands of VMs on VMware on-premises. Migrating to AWS normally means either re-platforming to EC2/ECS (expensive, risky) or retiring apps. VMware Cloud on AWS lets you move VMware VMs to AWS as-is — same tools, same operational model — then modernise at your own pace. Direct network connection to native AWS services (RDS, S3, etc.) enables gradual modernisation.
🏢 Real-World Use Case
Data Centre Exit — 5,000 VMs in 18 Months

A retail chain needs to exit two physical data centres in 18 months (lease expiry). They have 5,000 VMware VMs — modernising all of them is a 5-year project. Solution: migrate all VMs to VMware Cloud on AWS using vSphere vMotion (live migration, zero downtime). They exit the data centres on time. Over the next 3 years, they gradually modernise: stateless apps move to ECS, databases move to RDS Aurora, and the VMware cluster shrinks. Native AWS connectivity means they access S3, RDS, and Lambda from VMs without internet exposure.

🎯 SAA-C03 Exam Tips:
• "Migrate VMware workloads to AWS without changing the hypervisor" → VMware Cloud on AWS
• "Extend VMware environment into AWS" → VMware Cloud on AWS
• Uses AWS Dedicated Hardware — not shared EC2. Run on i3.metal instances.
• Managed by VMware (not AWS support) — but natively connects to AWS services

AWS Backup
Centralised Backup · Policy-Driven · Cross-Account · Vault Lock
Backup/DR Compliance

AWS Backup is a fully managed, centralised service for backing up data across AWS services. Instead of configuring snapshots individually for EC2, RDS, EFS, DynamoDB, S3, FSx, and on-premises systems — you set backup plans once in AWS Backup and it handles everything. One console, one policy, all services.

Supported Services: EC2, EBS, RDS (all engines), Aurora, DynamoDB, EFS, FSx, DocumentDB, Neptune, S3, Storage Gateway, VMware Cloud on AWS, and on-premises via Storage Gateway.
Backup Plans: Define frequency (hourly, daily, weekly), retention period, and transition to cold storage (cheaper Glacier-backed vaults). Apply plans to resources by tag — any new resource with tag Backup=true is automatically covered.
Backup Vault Lock (WORM): Write-Once-Read-Many protection. Once a backup is locked, even the root account cannot delete it before the retention period expires. The exam tests this for "immutable backup" scenarios.
🔄 Real-World Use Case
Healthcare Company — HIPAA-Compliant Centralised Backup

A healthcare company has RDS, DynamoDB, EFS, and on-premises VMware workloads. Compliance requires 7-year retention of patient records with immutability. AWS Backup with Vault Lock creates an immutable vault — backups cannot be deleted by anyone, including admins, before the 7-year window expires. A cross-account backup copies critical backups to an isolated security account — even if the primary account is compromised by ransomware, the backups in the locked vault in another account are untouched.

🎯 SAA-C03 Exam Tips:
• "Centralise backups across multiple AWS services and accounts" → AWS Backup
• "Immutable backups that cannot be deleted" → AWS Backup Vault Lock (WORM)
• "Protect backups from ransomware / accidental deletion" → Cross-account backup vault
• AWS Backup ≠ individual service snapshots — it's the unified management layer on top

📝 20 SAA-C03 Style Practice Questions

Question 1
A company wants to prevent any AWS account in their organisation from disabling Amazon GuardDuty or deleting CloudTrail trails. The restriction must apply even to account administrators. What should they implement?

A) IAM permissions boundary on all users    B) SCP on the root OU in AWS Organizations    C) AWS Config rule with auto-remediation    D) Control Tower detective guardrail

✅ Answer: B — SCPs applied to the root OU override all IAM policies including admin roles. Only SCPs can restrict actions for ALL principals in ALL accounts including the management account (with caveats). Config rules (D) detect but don't prevent. IAM permission boundaries (A) apply per-user.
Question 2
A data analytics team needs to run complex SQL queries joining 10 billion rows of historical sales data to generate weekly business reports. Query latency must be under 30 seconds. Which database should they use?

A) Amazon RDS MySQL Multi-AZ    B) Amazon DynamoDB with DAX    C) Amazon Redshift    D) Amazon Aurora PostgreSQL

✅ Answer: C — Redshift is the OLAP data warehouse built for exactly this: complex analytical queries on billions of rows using MPP and columnar storage. RDS and Aurora are OLTP (transactional). DynamoDB is key-value, not suited for complex SQL aggregations at this scale.
Question 3
A security team discovers that an S3 bucket containing sensitive data was made public two days ago. They need to identify which IAM user changed the bucket policy and from which IP address. Where should they look?

A) Amazon CloudWatch Logs    B) AWS Config change history    C) AWS CloudTrail event history    D) S3 server access logging

✅ Answer: C — CloudTrail records every API call including PutBucketPolicy with the identity (IAM user ARN) and source IP. CloudWatch Logs (A) captures application logs not API calls. Config (B) shows what changed but not WHO changed it. S3 access logs track object access, not bucket policy changes.
Question 4
A financial services company wants to track every monetary transaction in their system with a guarantee that the history can never be altered and can be mathematically proven to be complete and unmodified. Which service fits best?

A) Amazon RDS with automated backups    B) Amazon QLDB    C) Amazon Managed Blockchain    D) DynamoDB with DynamoDB Streams

✅ Answer: B — QLDB provides a cryptographically verifiable, immutable journal — you can mathematically prove the complete history of any record. Blockchain (C) is for when multiple untrusted parties need consensus — a single financial institution is a single trusted authority, so QLDB is preferred. RDS and DynamoDB have no cryptographic verification of history.
Question 5
A company has 1,500 EC2 instances running in private subnets with no public IPs. Operations engineers need shell access to these instances for troubleshooting. The solution must require no open inbound ports and log all session activity. What should they use?

A) Deploy a bastion host in a public subnet with port 22 open    B) Open port 22 on security groups for the operations team IP range    C) AWS Systems Manager Session Manager    D) AWS Direct Connect with a private VIF

✅ Answer: C — Session Manager provides browser-based shell access with zero open ports, zero SSH keys, and full CloudTrail/CloudWatch audit logging. Bastion hosts (A) require open ports and are operationally expensive to manage at scale. Session Manager is the modern, secure answer.
Question 6
A company is building a social networking app. They need to query relationships like "find all mutual friends between User A and User B" efficiently across millions of users. Which database is most appropriate?

A) Amazon RDS PostgreSQL    B) Amazon DynamoDB    C) Amazon Neptune    D) Amazon Redshift

✅ Answer: C — Neptune is the graph database purpose-built for relationship traversal queries. "Find mutual friends" = graph traversal across relationship edges. This is extremely expensive in relational DBs (multi-level self-joins) but a single graph query in Neptune. DynamoDB can't do relational queries. Redshift is for analytics.
Question 7
A platform team wants developers to be able to provision a pre-approved "web application stack" (VPC + EC2 + RDS) without granting them CloudFormation or IAM permissions. What should they use?

A) AWS CloudFormation StackSets    B) AWS Control Tower Account Factory    C) AWS Service Catalog    D) AWS Proton

✅ Answer: C — Service Catalog lets admins create approved CloudFormation-based products in portfolios that users can deploy without direct CloudFormation/IAM access. Proton (D) is specifically for containerised/serverless app templates. Control Tower (B) provisions AWS accounts, not application stacks.
Question 8
An EC2 instance is showing high CPU and the operations team wants to receive a notification when CPU exceeds 85% for 5 consecutive minutes. What is the simplest solution?

A) Enable AWS Trusted Advisor performance checks    B) Create a CloudWatch Alarm on CPUUtilization metric → SNS topic → email    C) Enable CloudTrail data events for EC2    D) Configure VPC Flow Logs

✅ Answer: B — CloudWatch Alarm monitors CPUUtilization (a default EC2 metric) and triggers SNS notification. Trusted Advisor (A) gives recommendations but doesn't alert in real-time on metric thresholds. CloudTrail (C) logs API calls, not performance metrics. VPC Flow Logs (D) capture network traffic.
Question 9
A company needs to replicate the same CloudFormation stack configuration — including security policies, VPC setup, and IAM roles — across 30 AWS accounts in 3 regions automatically. What should they use?

A) AWS CloudFormation nested stacks    B) AWS CloudFormation StackSets    C) AWS Service Catalog    D) AWS Config conformance packs

✅ Answer: B — CloudFormation StackSets deploys a single CloudFormation template across multiple accounts and regions in one operation. It integrates with Organizations for automatic deployment to new accounts. Nested stacks (A) are within a single account. Service Catalog (C) is for end-user provisioning.
Question 10
A company has a dedicated 10 Gbps AWS Direct Connect connection for their primary workloads. They need to ensure connectivity to AWS is maintained if the Direct Connect circuit fails. What is the most cost-effective failover solution?

A) Provision a second 10 Gbps Direct Connect connection    B) Configure a Site-to-Site VPN as a backup    C) Use Amazon CloudFront for failover    D) Configure AWS Global Accelerator

✅ Answer: B — Site-to-Site VPN is the cost-effective failover for Direct Connect. It uses the internet (lower performance than DX but acceptable for failover) and is pre-configured to activate if the DX BGP session goes down. A second DX connection (A) is for resilience at full capacity but costs significantly more. CloudFront and Global Accelerator don't replace private connectivity.
Question 11
A company wants to run their containerised microservices on Kubernetes but does not want to manage Kubernetes worker nodes. All they want to define are pods and deployments. What should they choose?

A) Amazon ECS with EC2 launch type    B) Amazon EKS with EC2 worker nodes    C) Amazon EKS with AWS Fargate    D) Amazon ECS Anywhere

✅ Answer: C — EKS with Fargate is serverless Kubernetes — no worker node management. You define pods and Fargate handles the underlying compute. EKS with EC2 (B) still requires managing node groups. ECS (A) is not Kubernetes. ECS Anywhere (D) is for on-premises deployments.
Question 12
A development team needs to store Docker images privately for use in their ECS deployment pipeline. Images must be scanned for vulnerabilities before deployment. Which service handles both requirements natively?

A) Docker Hub Private Registry    B) Amazon ECR with Enhanced Scanning    C) Amazon S3 with versioning    D) AWS CodeArtifact

✅ Answer: B — ECR is the native AWS private container registry integrated with ECS/EKS. ECR Enhanced Scanning (powered by Amazon Inspector) scans for OS and language package CVEs. Docker Hub (A) is external, not IAM-integrated. S3 (C) stores files but not OCI/Docker images. CodeArtifact (D) is for package dependencies (npm, Maven), not container images.
Question 13
A company is building a mobile app that requires real-time collaboration features — users should see each other's changes to documents instantly. The backend should push updates to all connected clients without polling. Which AWS service is purpose-built for this?

A) Amazon API Gateway WebSocket API    B) AWS AppSync with subscriptions    C) Amazon SQS FIFO queue    D) Amazon Kinesis Data Streams

✅ Answer: B — AppSync is purpose-built for real-time, collaborative apps with GraphQL subscriptions via WebSockets. Clients subscribe to changes and receive push updates automatically. API Gateway WebSocket (A) requires custom logic to manage subscriptions and fanout. AppSync handles all of this natively with DynamoDB integration.
Question 14
A company needs to run an ML-based workload on factory floor servers due to latency requirements. They want to use the same container orchestration tooling and CI/CD pipeline they use for their AWS-hosted applications. What should they use?

A) AWS Outposts    B) Amazon EKS with EC2    C) Amazon ECS Anywhere    D) AWS Wavelength

✅ Answer: C — ECS Anywhere lets you run ECS container workloads on your own on-premises hardware using the same ECS control plane and tooling. Outposts (A) brings AWS infrastructure to your premises (full rack). ECS Anywhere runs on your existing hardware with just agents installed. Wavelength (D) is for 5G edge compute at telecom facilities.
Question 15
A database administrator stores an application's RDS password in a config file on the EC2 instance. The security team wants this practice eliminated. The application should retrieve the credential at runtime without the secret ever being in code or config files. What is the best solution?

A) Store the password in an EC2 User Data script    B) Use an IAM instance profile with hardcoded credentials    C) Store the password in AWS Systems Manager Parameter Store as a SecureString, retrieved at runtime    D) Encrypt the config file with KMS

✅ Answer: C — Parameter Store SecureString encrypts the value with KMS and the application retrieves it at runtime via the SSM API using the EC2 instance's IAM role (no credentials needed). If auto-rotation is required, use Secrets Manager instead. User Data (A) and encrypted config files (D) still store the secret on the instance.
Question 16
An enterprise has 5,000 VMware VMs on-premises and needs to exit their data centre in 12 months due to a lease expiry. Re-architecting all workloads is not feasible in this timeframe. Which solution allows them to migrate without changing the hypervisor or operational tools?

A) AWS Server Migration Service to convert VMs to EC2 AMIs    B) VMware Cloud on AWS    C) AWS Application Migration Service    D) Deploy AWS Outposts in the data centre

✅ Answer: B — VMware Cloud on AWS runs VMware SDDC on dedicated AWS hardware. VMs can be migrated using vSphere vMotion with zero downtime. Operations teams continue using the same vSphere/vCenter tools. This is the only option that preserves the VMware environment entirely. SMS (A) and MGN (C) convert VMs but require at least re-platforming to EC2.
Question 17
A company has a weekend batch workload that processes 10TB of data. The job runs every Saturday for 6 hours and is designed to restart automatically if interrupted. Which EC2 pricing model minimises cost?

A) On-Demand instances    B) Reserved Instances (1-year)    C) Spot Instances    D) Dedicated Hosts

✅ Answer: C — Spot instances offer up to 90% savings and are perfect for fault-tolerant batch workloads that can handle interruption and restart. The job is specifically designed to restart, and it doesn't run continuously — making Reserved (B) wasteful. On-Demand (A) is safest but most expensive. Dedicated Hosts (D) are for licensing compliance, not cost optimisation.
Question 18
A healthcare company must retain patient data backups for 7 years and ensure that no one — including AWS account administrators — can delete them before the retention period expires. How can they achieve this?

A) Enable S3 Object Lock on the backup bucket    B) Apply an SCP denying delete actions on backup resources    C) Use AWS Backup with Vault Lock enabled on the backup vault    D) Store backups in Glacier with a vault access policy

✅ Answer: C — AWS Backup Vault Lock (WORM) prevents deletion of backups even by the root account. It covers all supported services (RDS, EFS, DynamoDB, EC2 EBS, etc.) — not just S3. S3 Object Lock (A) only applies to S3 objects. SCPs (B) don't apply to the management account. Vault Lock is the specific feature for immutable multi-service backups.
Question 19
A company uses AWS Direct Connect for their primary connection to AWS. They need to ensure that traffic between their data centre and AWS is encrypted, as the current Direct Connect connection does not provide encryption. What is the recommended approach?

A) Request encrypted Direct Connect from AWS    B) Configure MACsec on the Direct Connect connection    C) Deploy a Site-to-Site VPN over the Direct Connect connection    D) Use AWS PrivateLink instead

✅ Answer: C — Running an IPSec Site-to-Site VPN tunnel over the Direct Connect connection provides encryption while maintaining the low-latency, consistent path of Direct Connect. This is the standard AWS recommendation for encrypting DX traffic. MACsec (B) is available on some DX connections but is less commonly tested at SAA level. PrivateLink (D) is for accessing AWS services, not for on-prem connectivity.
Question 20
A solutions architect needs to set up a new multi-account AWS environment for a company with separate accounts for Security, Logging, Dev, Test, and Production — all with security baselines, centralised CloudTrail, and SSO pre-configured. What is the fastest way to achieve this?

A) Manually create each account and configure services    B) Write a CloudFormation StackSet    C) Use AWS Control Tower to create a Landing Zone    D) Use AWS Organizations with SCPs

✅ Answer: C — Control Tower's Landing Zone automation sets up the entire multi-account structure with Log Archive, Audit, and management accounts, enables CloudTrail, Config, SSO (IAM Identity Center), and applies baseline guardrails — automatically and in one workflow. Organizations (D) is the underlying primitive but doesn't auto-configure security baselines. StackSets (B) require you to write and manage all the templates manually.
