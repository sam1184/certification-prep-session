🔗 Block 1 — Data & API Integration

Amazon AppFlow
No-Code SaaS Data Integration · Bidirectional · Fully Managed
Integration NO-CODE MEDIUM EXAM WEIGHT

Amazon AppFlow is a fully managed, no-code integration service that securely transfers data between AWS services and SaaS applications — think Salesforce, ServiceNow, Slack, SAP, Zendesk, and dozens more. You set up a "flow" (source → optional transform → destination) in the Console, no Lambda or custom code needed. It handles authentication, scheduling, filtering, and field mapping for you.

🧠 Mental Analogy
AppFlow = Plumbing between SaaS apps and AWS. Imagine your CRM (Salesforce) is a water tank and your data warehouse (Redshift) is your house. AppFlow is the pipes + valves — it automatically moves water on a schedule, filters out dirty water, and transforms pressure along the way. No digging required.

🚨 Exam Trap — AppFlow vs AWS Glue vs Data Pipeline:
• AppFlow → Move data FROM SaaS apps (Salesforce, Slack, ServiceNow). No-code, managed connectors.

• AWS Data Pipeline → Legacy. Schedule data movement between AWS services. Avoid in new designs.
If the question mentions a SaaS app name (Salesforce, SAP, Zendesk) → think AppFlow first.
🌍 Real-World Use Case
Retail Company — Sync Salesforce CRM Data to Redshift for Analytics

A UK retailer's sales team uses Salesforce for managing 50,000 customer accounts. The analytics team needs this data in Amazon Redshift every night for BI dashboards. Without AppFlow, they'd need a developer to write Python scripts, handle Salesforce API auth tokens, manage failures, and re-run partial loads.

With AppFlow: A business analyst sets up a flow in 20 minutes — select Salesforce as source, pick Accounts/Opportunities objects, map fields to Redshift columns, schedule nightly at 1am, enable PII masking on email fields. Done. No code. No maintenance. The data lands in Redshift every morning. Cost: fractions of a penny per record transferred.

🎯 SAA-C03 Exam Tips:
• Trigger words: "Salesforce," "SAP," "ServiceNow," "SaaS to AWS," "no-code integration," "transfer SaaS data to S3/Redshift"
• AppFlow is bidirectional — can push data FROM AWS back TO Salesforce too
• Supports data transformation (field mapping, filtering, PII masking) without writing code
• Data stays private via AWS PrivateLink — never crosses public internet
• NOT for real-time streaming (use Kinesis). AppFlow is batch/scheduled or event-triggered.
AWS AppSync
Managed GraphQL API · Real-Time Subscriptions · Offline Sync
GraphQL REAL-TIME MEDIUM EXAM WEIGHT

AWS AppSync is a fully managed service that creates GraphQL APIs. Unlike REST APIs (multiple endpoints for each resource), GraphQL gives clients a single endpoint where they ask for exactly the data they need — no more, no less. AppSync adds real-time subscriptions (push updates to clients when data changes) and offline support with automatic sync when reconnected.

🧠 Mental Analogy
REST API = À la carte restaurant (order each dish separately — multiple trips to the kitchen). GraphQL/AppSync = Buffet (one trip, pick exactly what you want, nothing you don't). AppSync manages the buffet table — it automatically fetches data from DynamoDB, Lambda, RDS, or any HTTP endpoint, combines it, and serves only what the client asked for.

🌍 Real-World Use Case
Ride-Sharing App — Real-Time Driver Location Tracking

A ride-sharing startup builds their mobile app on AWS AppSync. When a passenger books a ride, the app subscribes to the driver's location updates via a GraphQL subscription. Every 3 seconds, the driver's phone sends a GPS coordinate via a GraphQL mutation → AppSync writes it to DynamoDB → AppSync automatically pushes the updated location to the passenger's app via WebSocket subscription. The passenger sees the driver moving on the map in real-time — no polling, no custom WebSocket server.

Bonus: If the passenger loses mobile signal in a tunnel, the app works in offline mode — queuing mutations locally and syncing when back online. All of this runs without the startup managing any servers.

🏗️ Architecture — AWS AppSync GraphQL API with Multiple Data Sources
AppSync: single GraphQL endpoint handles queries, mutations, and real-time subscriptions — resolves data from multiple AWS backends simultaneously.
🎯 SAA-C03 Exam Tips:
• Trigger words: "GraphQL API," "real-time updates," "mobile app offline sync," "WebSocket," "push data to clients," "single endpoint for multiple data sources"
• AppSync = GraphQL. API Gateway = REST. Know which the question is asking for.
• Subscriptions = real-time push to clients (chat apps, live dashboards, tracking apps)
• AppSync can merge data from multiple data sources in a single query (DynamoDB + Lambda + RDS)
• Caching is built in — reduce calls to data sources, lower costs

⚡ Block 2 — Messaging & Event-Driven Architecture

⏱️ Time: 00:30 – 01:05 EventBridge · Amazon MQ · SNS · SQS — Most Exam-Heavy Block

This block is the most heavily tested in SAA-C03. Messaging and event-driven architecture questions appear in almost every exam. The core skill: given a scenario, pick the right service. The confusion zone is SQS vs SNS vs EventBridge — we'll kill that confusion for good.

Amazon EventBridge
Serverless Event Bus · Rule-Based Routing · SaaS + AWS Events
HIGH EXAM WEIGHT Event-Driven SERVERLESS

Amazon EventBridge is a serverless event bus that connects AWS services, your own applications, and SaaS providers using events. An event is just a JSON message saying "something happened." EventBridge receives these events, evaluates rules (filters), and routes matching events to targets — Lambda, SQS, SNS, Step Functions, API Gateway, and more. Formerly called CloudWatch Events (same service, renamed and extended).

🧠 Mental Analogy
EventBridge = Air Traffic Control for your events. Events are planes landing at the airport. EventBridge is the tower — it listens to all flights, reads their flight plan (event pattern), and directs them to the right gate (target). Gate 1 is Lambda, Gate 2 is SQS, Gate 3 is another account. Planes (events) can even come from other airports (SaaS apps, partner accounts).

Event Rule — How Routing Works

Event Pattern Example: "Route to Lambda when an EC2 instance changes state to 'stopped' in us-east-1"

{
  "source": ["aws.ec2"],
  "detail-type": ["EC2 Instance State-change Notification"],
  "detail": { "state": ["stopped"] }
}
EventBridge evaluates every incoming event against all rules. If an event matches, it goes to the configured target(s). One event can match multiple rules → go to multiple targets simultaneously.

EventBridge Scheduler

EventBridge also has a Scheduler — replaces CloudWatch Events scheduled rules. You can invoke targets on a fixed rate (every 5 minutes), a cron expression (every Monday 9am), or a one-time schedule. This is the managed replacement for cron jobs on EC2 — no server needed.

🌍 Real-World Use Case
E-commerce Platform — Order Processing Automation

An e-commerce platform processes 100,000 orders/day. When an order is placed, the Order Service publishes an event to a custom EventBridge bus: {"source": "orders.service", "detail-type": "OrderPlaced", "detail": {"orderId": "X123", "amount": 49.99}} EventBridge rules fan out the event to 4 targets simultaneously:
→ Lambda: sends order confirmation email
→ SQS: queues item for warehouse picking system
→ SNS: notifies fraud detection team if amount > £500
→ Step Functions: starts the fulfilment workflow

The Order Service doesn't know about any of these consumers — it just publishes the event. Adding a new consumer? Add a new EventBridge rule. No code change to Order Service. This is true decoupling.

🏗️ Architecture — EventBridge Event-Driven Microservices
EventBridge: events from any source → evaluated against rules → routed to multiple targets simultaneously. One event can match multiple rules.
🎯 SAA-C03 Exam Tips:
• Trigger words: "schedule Lambda," "react to AWS service events," "route events to multiple targets," "cross-account events," "SaaS events to AWS," "cron job serverless"
• EventBridge replaced CloudWatch Events — they're the same service. Don't be confused if a question says "CloudWatch Events."
• Near-real-time event delivery (milliseconds) — NOT a message queue (no retention, no replay by default)
• EventBridge Archive + Replay — you CAN replay past events for debugging
• Schema Registry — auto-discovers event schemas, generates code bindings
Amazon MQ
Managed Message Broker · Apache ActiveMQ & RabbitMQ · Legacy Migration
Messaging MIGRATION MEDIUM EXAM WEIGHT

Amazon MQ is a managed message broker service for Apache ActiveMQ and RabbitMQ. It's designed for companies migrating existing on-premises message brokers to AWS without rewriting application code. Your applications use standard protocols (AMQP, MQTT, OpenWire, STOMP) — they don't need to know they're now talking to AWS. Amazon MQ manages provisioning, patching, high availability, and monitoring.

🧠 Mental Analogy
Amazon MQ = AWS running your existing post office. If your company already has a postal system (ActiveMQ/RabbitMQ), moving to SQS means redesigning your entire mail system. Amazon MQ instead picks up your existing post office, puts it in the cloud, and manages it — all the old postmen (applications) still work the same way.
ActiveMQ
Engine Option 1
Supports AMQP, MQTT, OpenWire, STOMP, WebSocket. High compatibility with enterprise Java apps.
RabbitMQ
Engine Option 2
AMQP 0-9-1. Excellent for complex routing, exchange patterns, microservices messaging.
🚨 The Most Important Exam Distinction — Amazon MQ vs SQS/SNS:

Use Amazon MQ when: The application is being migrated from on-premises, it already uses ActiveMQ/RabbitMQ, and you need protocol compatibility (AMQP, MQTT, STOMP) without rewriting code.

Use SQS/SNS when: Building a new cloud-native application. SQS/SNS are infinitely scalable, cheaper, and fully serverless. No capacity planning, no broker management.

⚡ Exam shortcut: "migrating" + "ActiveMQ/RabbitMQ" + "on-premises" + "existing application" = Amazon MQ. Everything else = SQS/SNS.

Amazon MQ High Availability

Active/Standby for HA: Amazon MQ can deploy as an active/standby broker pair across two AZs. Both share the same EFS storage. If the active broker fails, the standby broker takes over automatically in under 60 seconds. Clients reconnect transparently using the failover URL. For ActiveMQ, a single instance (no HA) is also available for dev/test at lower cost.
🌍 Real-World Use Case
Insurance Company — Legacy System Lift & Shift to AWS

A large insurance company has 200 Java microservices running on-premises, all communicating via Apache ActiveMQ using AMQP. They want to migrate to AWS but cannot rewrite all 200 services — it would take 3 years and cost millions.

Solution: They deploy Amazon MQ (ActiveMQ) in an active/standby configuration across two AZs. They update only the connection URL in each service's config file (pointing to Amazon MQ endpoint instead of the on-premises broker). No code changes. No protocol changes. No retraining developers.

Result: 200 services migrated in 6 weeks. Amazon MQ handles patching, monitoring, and failover. They save on hardware costs and gain AWS security/compliance benefits. Eventually, they'll gradually migrate individual services to SQS/SNS — but on their own timeline.

🏗️ Architecture — Amazon MQ Migration: On-Premises to AWS
Amazon MQ: Lift existing ActiveMQ/RabbitMQ to AWS. Update only the connection URL. Multi-AZ active/standby with shared EFS ensures high availability.
🎯 SAA-C03 Exam Tips:
• Trigger words: "existing ActiveMQ/RabbitMQ," "lift and shift," "AMQP/MQTT/STOMP protocol," "migrate message broker," "on-premises to cloud without code changes"
• Amazon MQ runs on dedicated broker instances — NOT serverless like SQS/SNS
• Supports multi-AZ active/standby — know this for HA questions
• Both queues (point-to-point like SQS) and topics (pub/sub like SNS) are supported in Amazon MQ
• Storage is on Amazon EFS (shared between active/standby nodes)
Amazon SNS
Simple Notification Service · Pub/Sub · Push to Multiple Subscribers
HIGH EXAM WEIGHT Pub/Sub SERVERLESS

Amazon SNS (Simple Notification Service) is a fully managed pub/sub messaging service. A producer publishes a message to an SNS Topic, and SNS immediately pushes that message to all subscribers. Subscribers can be SQS queues, Lambda functions, HTTP endpoints, email addresses, SMS numbers, or mobile push notifications. One message → many destinations simultaneously. This is a push model — SNS delivers without subscribers polling.

🧠 Mental Analogy
SNS = A PA system announcement in a shopping mall. One person speaks into the microphone (publisher). Everyone in the mall hears it simultaneously (all subscribers). The PA system (SNS Topic) doesn't know or care who's listening — it just broadcasts. If you're not listening at that moment, you miss it (no message persistence by default).

The SNS + SQS Fan-Out Pattern — Memorise This

📌 Pattern: One SNS Topic → Multiple SQS Queues → Different processing per queue

Why it's powerful: Publish once, process differently in parallel. Example: an image uploaded to S3 triggers SNS → SQS-1 (generate thumbnail) + SQS-2 (run virus scan) + SQS-3 (update metadata DB). Each queue has its own Lambda consumer. Adding a new processing step = just subscribe a new SQS queue. Zero changes to existing code.

Why SQS, not direct Lambda? SQS provides durability + retry. If Lambda fails, message stays in SQS and retries. Direct SNS → Lambda has limited retry control.
🏗️ Architecture — SNS Fan-Out Pattern with SQS (Most Tested SAA-C03 Pattern)
SNS Fan-Out: 1 publish → 4 SQS queues processed independently in parallel. Filter policies on Queue 4 = only high-value orders. DLQ catches failures.

SNS FIFO Topics

SNS FIFO guarantees strict message ordering and exactly-once delivery to SQS FIFO queues. Only SQS FIFO queues can subscribe to an SNS FIFO topic. Use when order matters — financial transactions, inventory updates. Throughput is limited (300 msg/s) compared to standard SNS (unlimited).

🌍 Real-World Use Case
News Platform — Breaking News to 10 Million Mobile Users

A news platform needs to instantly push breaking news alerts to 10 million mobile users across iOS and Android, while also emailing 2 million subscribers and posting to 5 Slack channels.

Architecture: Editor publishes article → triggers SNS Topic. SNS fans out to:
→ Mobile Push endpoint (Amazon SNS sends to APNs for iOS, FCM for Android) — 10M push notifications delivered in seconds
→ SQS Queue → Lambda processes email list in batches (2M emails via SES)
→ Lambda → posts to Slack channels via Slack API

One SNS.Publish() API call handles all of this. Cost: fractions of a penny per notification.

🎯 SAA-C03 Exam Tips:
• Trigger words: "notify multiple subscribers," "fan-out," "pub/sub," "push notification," "broadcast message," "one-to-many"
• SNS = PUSH model. SQS = PULL model. Know the difference.
• Message Filtering: Subscribers can set filter policies — only receive messages matching certain attributes (e.g., only OrderType=Premium)
• SNS does NOT persist messages — if no subscriber is listening, message is lost (unlike SQS which stores messages)
• SNS + SQS fan-out = the single most tested pattern in Application Integration domain
Amazon SQS
Simple Queue Service · Decoupling · Pull-Based · Durable Message Storage
HIGH EXAM WEIGHT Queue SERVERLESS

Amazon SQS (Simple Queue Service) is the oldest AWS service (launched 2004) and one of the most exam-critical. It's a fully managed message queue that decouples application components. A producer sends messages to the queue; one or more consumers poll (pull) messages and process them. Messages are stored durably until processed (or expire). SQS handles the pace mismatch between fast producers and slow consumers.

🧠 Mental Analogy
SQS = A queue at a coffee shop counter. Orders (messages) go into the queue. Baristas (consumers) take one order at a time, make the coffee, and mark it done. If all baristas are busy, orders wait safely in the queue — they don't get lost. When rush hour hits, you can hire more baristas (scale out consumers). The customer placing the order doesn't wait for the coffee — they got a receipt (acknowledgement) and can go sit down (async processing).

Dead Letter Queue (DLQ) — Critical for Exam

What it is: A separate SQS queue where messages go after failing to process N times (maxReceiveCount). Like a "returns desk" for failed orders.

Why it matters: Without DLQ, a poison pill message (one that always fails) loops forever, blocking your queue. With DLQ, after 3 failures (for example), it moves to the DLQ for manual inspection or separate error processing.

DLQ for Lambda: Lambda also supports DLQs — failed async invocations go to SQS or SNS DLQ.
Exam trap: DLQ must be the same type as the source queue (Standard DLQ for Standard queue, FIFO DLQ for FIFO queue).

Visibility Timeout — Understand This Deeply

🚨 Visibility Timeout Explained:
When Consumer A receives a message, SQS makes it invisible to other consumers for the visibility timeout period (default: 30 sec). Consumer A processes it and deletes it. If A crashes before deleting, the timeout expires → message becomes visible again → Consumer B picks it up.

Common exam trap: "Messages are being processed twice" → Visibility timeout is too short (processing takes longer than the timeout → message becomes visible before deletion → duplicate processing). Solution: Increase visibility timeout OR use ChangeMessageVisibility to extend it during processing.
🌍 Real-World Use Case
Video Streaming Platform — Async Video Transcoding

A video platform receives 10,000 video uploads per hour. Transcoding each video takes 5-20 minutes and is CPU-intensive. You cannot transcode synchronously — the uploader would time out.

Architecture:
1. User uploads video → stored in S3
2. S3 event notification → message sent to SQS Standard Queue (videoTranscodeQueue)
3. Auto Scaling Group of EC2 spot instances poll the queue
4. Each instance picks one video, transcodes to 1080p/720p/480p, uploads outputs to S3, deletes message from queue
5. If instance fails mid-transcode: visibility timeout expires → another instance picks up the job
6. Videos that fail 3 times → sent to DLQ → ops team investigates

Result: Upload confirmation is instant for users. Transcoding happens in the background at scale. Queue acts as a shock absorber — during viral upload events, the queue simply grows and more spot instances spin up to drain it.

🏗️ Architecture — SQS Message Lifecycle: Visibility Timeout & Dead Letter Queue
SQS lifecycle: Producer sends → Consumer receives (message hidden) → Success: delete | Failure: timeout expires, retry → after maxReceiveCount, goes to DLQ.
🎯 SAA-C03 Exam Tips:
• Trigger words: "decouple," "buffer," "asynchronous," "queue," "handle traffic spikes," "retry failed messages," "at-least-once delivery"
• "Process messages in order, no duplicates" → SQS FIFO
• "Reduce cost of polling empty queue" → Long Polling (WaitTimeSeconds = 20)
• "Message processed multiple times" → increase Visibility Timeout
• "Failed messages for analysis" → Dead Letter Queue (DLQ)
• SQS integrates natively with Lambda as event source — Lambda polls SQS and auto-scales
• Short vs Long Polling: Short = returns immediately (even if empty, costs money). Long = waits up to 20s for a message (fewer API calls, lower cost).

💰 Block 3 — Cost Management & Optimization

⏱️ Time: 01:05 – 01:45 AWS Budgets · Cost & Usage Report · Cost Explorer · Savings Plans

Cost Optimization is a first-class pillar in the AWS Well-Architected Framework and a growing part of SAA-C03. AWS gives you a suite of tools to see what you're spending, predict costs, alert on overruns, and commit to discounts. Know which tool does which job — the exam loves to test this.

AWS Budgets
Set Cost & Usage Thresholds · Alerts & Actions · Proactive Control
Cost Management MEDIUM EXAM WEIGHT

AWS Budgets lets you set custom cost and usage budgets and receive alerts when you approach or exceed thresholds. Think of it as a financial alarm system — you define the limit, AWS monitors spend in real time, and notifies you (email, SNS, Slack via SNS) before you blow past it. You can also set Budget Actions that automatically take corrective action (like applying an IAM policy to block new resource creation) when a threshold is crossed.

🧠 Mental Analogy
AWS Budgets = Your bank's spending alerts. You tell your bank: "alert me if I spend more than £500 this week." Your bank monitors every transaction and texts you at £400 (80% threshold) and again at £500 (100%). AWS Budgets works exactly the same — but for your cloud bill — and can even freeze your account card (Budget Actions) if you hit the limit.

💡 Budget Actions — Automated Cost Control:
When a budget threshold is crossed, Budget Actions can automatically:
• Apply an IAM policy (e.g., deny EC2 launch permissions to developers)
• Apply a Service Control Policy (SCP) to an AWS Organizations OU
• Stop EC2/RDS instances (target specific instances by tag)
This enables automated cost guardrails — no human intervention needed to stop runaway spending.
🌍 Real-World Use Case
Startup — Prevent Bill Shock During Rapid Growth

A startup gives each development team their own AWS account within AWS Organizations. Finance sets up AWS Budgets per account:

• $5,000/month cost budget per dev team account
• Alert at 80% ($4,000) → email the team lead
• Alert at 100% ($5,000) → email CTO + Finance
• Budget Action at 110% ($5,500) → automatically apply IAM policy denying launch of new EC2/RDS instances

One month, a developer accidentally leaves a fleet of GPU instances running over a weekend. At 80% threshold, the alert fires. Team lead investigates and terminates the instances. They never hit the action threshold. The automated guardrail prevented a $30,000 surprise bill.

🎯 SAA-C03 Exam Tips:
• Trigger words: "alert when spend exceeds," "notify before budget overrun," "automated cost control," "stop resources when budget exceeded"

Savings Plans
Flexible Commitment-Based Discounts · Up to 72% off On-Demand
SAVE UP TO 72% HIGH EXAM WEIGHT Cost Optimization

AWS Savings Plans are a flexible pricing model offering significant discounts (up to 72%) in exchange for a commitment to a consistent amount of compute usage (measured in $/hour) for 1 or 3 years. Unlike Reserved Instances, Savings Plans are much more flexible — they apply automatically to eligible usage without specifying instance types in advance (for Compute Savings Plans). The more you commit, the more you save.

🧠 Mental Analogy
Savings Plans = A gym membership vs pay-per-visit. On-demand = paying £20 each time you visit the gym. Reserved Instances = buying a yearly pass for a specific gym, a specific time slot (inflexible). Savings Plans = buying a yearly pass that works at any gym in the chain, at any time — same big discount, much more freedom. You just commit to "I'll spend at least £50/month on gym visits."

Question 01 — Integration
A company uses Salesforce as their CRM and wants to automatically sync Salesforce Opportunity data to Amazon Redshift every night for analytics. The team has no developers available and needs a no-code solution. Which AWS service should they use?
A AWS Glue with a custom Salesforce connector
B Amazon AppFlow with a scheduled flow from Salesforce to Redshift
C AWS Lambda triggered by CloudWatch Events to call Salesforce API
D Amazon Kinesis Data Firehose with a Salesforce source
✅ Correct Answer: B
Amazon AppFlow is the correct answer. It's specifically designed for no-code, managed data transfer between SaaS applications (like Salesforce) and AWS services (like Redshift). A scheduled trigger runs nightly automatically. AWS Glue (A) requires ETL coding. Lambda (C) requires custom development and Salesforce API handling. Kinesis Firehose (D) doesn't have a native Salesforce source connector.
Question 02 — AppSync
A mobile application needs to display real-time updates to all connected users when inventory levels change in an Amazon DynamoDB table. The team wants to minimise infrastructure management and avoid writing WebSocket server code. Which solution meets these requirements?
A Amazon API Gateway with WebSocket API and Lambda integration
B Amazon SNS with mobile push subscriptions
C AWS AppSync with GraphQL subscriptions and DynamoDB as data source
D Amazon SQS with long polling from mobile clients
✅ Correct Answer: C
AWS AppSync with GraphQL subscriptions is the correct answer. AppSync manages WebSocket connections automatically, connects directly to DynamoDB, and pushes real-time updates to all subscribed clients when data changes — no server code required. API Gateway WebSocket (A) requires more Lambda integration code. SNS (B) doesn't maintain persistent connections for real-time in-app updates. SQS (D) is a pull model — mobile clients polling SQS is inefficient and not real-time.
Question 03 — EventBridge
A company wants to automatically send a Slack notification and start an EC2 backup process whenever an EC2 instance in their account changes state to "stopped." The solution should require no custom polling code. What is the MOST suitable approach?
A Enable EC2 CloudWatch Metrics, create a CloudWatch Alarm, trigger SNS
B Create an Amazon EventBridge rule matching EC2 state-change events to "stopped," with Lambda targets for Slack and backup
C Use AWS Config to detect the state change and trigger a remediation Lambda
D Schedule a Lambda function every minute to describe EC2 instances and check state
✅ Correct Answer: B
EventBridge natively receives EC2 state-change events on the default event bus. You create a rule with an event pattern matching the "stopped" state, and configure multiple Lambda targets (one for Slack, one for backup). This is event-driven — no polling needed. CloudWatch Alarms (A) monitor metrics, not state changes. AWS Config (C) is for compliance evaluation, not event routing. Lambda polling every minute (D) has latency, costs money for empty polls, and is the wrong pattern.
Question 04 — Amazon MQ
A financial services company is migrating 150 Java applications from on-premises to AWS. These applications communicate using Apache ActiveMQ with AMQP protocol and cannot be rewritten within the migration timeline. Which AWS service should they use for the message broker component?
A Amazon SQS with AMQP protocol adapter
B Amazon MQ with Apache ActiveMQ engine
C Amazon SNS with AMQP endpoint subscriptions
D Amazon Kinesis Data Streams with Java consumer library
✅ Correct Answer: B
Amazon MQ with Apache ActiveMQ is purpose-built for this scenario: migrating existing applications that use ActiveMQ/RabbitMQ with standard protocols (AMQP, MQTT, STOMP, OpenWire). Applications only need to update the connection URL — no code changes. SQS (A) does not support AMQP and requires SDK changes. SNS (C) doesn't support AMQP connections. Kinesis (D) is a streaming service, not a message broker replacement for ActiveMQ.
Question 05 — SNS vs SQS
An e-commerce platform needs to process order events in three separate systems simultaneously: update inventory, send a confirmation email, and generate an invoice. Each system should process independently and failures in one should not affect others. What is the BEST architecture?
A One SQS queue; all three consumers poll the same queue
B Three separate SQS queues; the application sends to all three queues
C One SNS topic with three SQS queues subscribed — fan-out pattern
D EventBridge rule with three Lambda functions as direct targets
✅ Correct Answer: C
The SNS + SQS fan-out pattern is optimal. One order event → SNS publishes to all 3 subscribers simultaneously. Each SQS queue provides durability + independent retry for each system. If the email system fails, the inventory queue is unaffected. A (single queue) means only one system gets each message. B (app sends to 3 queues) adds coupling — the app must know about all systems and manage 3 sends. D (direct Lambda) lacks the durability/retry buffer that SQS provides.
Question 06 — SQS Deep Dive
A company's SQS queue processes image resizing jobs. They notice some images are being resized multiple times. Investigation shows processing takes 45 seconds but the visibility timeout is 30 seconds. What is the MOST appropriate fix?
A Increase the visibility timeout to at least 60 seconds (2x processing time)
B Switch from Standard Queue to FIFO Queue
C Enable long polling on the queue
D Add a Dead Letter Queue to capture failed messages
✅ Correct Answer: A
The visibility timeout (30s) is shorter than processing time (45s). When the first consumer hasn't finished and the timeout expires, the message becomes visible again and a second consumer picks it up — causing duplicate processing. Increasing the visibility timeout to ≥60 seconds (2x as a rule of thumb) fixes this. FIFO (B) prevents duplicate delivery, not timeout-caused re-delivery. Long polling (C) reduces empty receive calls — doesn't fix this issue. DLQ (D) handles repeatedly failing messages, not duplicate processing from timeout expiry.
Question 07 — SQS FIFO
A trading platform needs to process stock orders in the exact sequence they were received, with no possibility of processing the same order twice. Which SQS queue type is required?
A SQS Standard Queue with message deduplication logic in the consumer
B SQS Standard Queue with message group IDs
C SQS FIFO Queue with exactly-once processing
D Amazon Kinesis Data Streams with shard-level ordering
✅ Correct Answer: C
SQS FIFO Queue guarantees: (1) strict FIFO ordering — messages processed in the exact order sent; (2) exactly-once processing — built-in deduplication within a 5-minute window. This is the only SQS type with these guarantees. Standard Queue (A/B) has best-effort ordering and at-least-once delivery (duplicates possible). Kinesis (D) provides shard-level ordering but is designed for streaming analytics, not transactional order processing — more complex to implement correctly.
Question 08 — AWS Budgets
A startup CTO wants to ensure that if any AWS account in their AWS Organization exceeds $8,000 in a month, all developers automatically lose the ability to launch new EC2 instances — without any manual intervention. Which service combination achieves this?
A AWS Cost Explorer anomaly detection + SNS notification to CTO
B CloudWatch billing alarm + Lambda to modify IAM policies
C AWS Budgets with a Budget Action to apply an IAM policy denying EC2 launch
D AWS Cost and Usage Report + Athena query to identify the breach
✅ Correct Answer: C
AWS Budgets with Budget Actions is purpose-built for this. Create a cost budget at $8,000, configure a Budget Action at 100% to automatically apply a deny IAM policy that blocks EC2 RunInstances — no human needed. Cost Explorer (A) only sends notifications, no automated action. CloudWatch + Lambda (B) technically works but is more complex to build and maintain — Budget Actions is the managed, simpler solution. CUR + Athena (D) is for analysis, not real-time automated enforcement.
Question 09 — CUR vs Cost Explorer
A large enterprise with 50 AWS accounts wants to build a custom monthly chargeback report showing exact costs per team, broken down by individual resource and tagged environment. They need to query this data using SQL. Which service provides the MOST granular data for this purpose?
A AWS Cost Explorer API with filtering by tag
B AWS Cost and Usage Report delivered to S3, queried with Amazon Athena
C AWS Budgets reports exported to CSV
D AWS Trusted Advisor cost optimization checks
✅ Correct Answer: B
CUR + Athena is the correct answer. CUR is the most granular billing dataset AWS provides — line-item level, per resource, per hour, with all tags included. Delivering to S3 and querying with Athena enables custom SQL analysis with full flexibility. Cost Explorer API (A) provides aggregated data — not line-item granularity for individual resources. Budgets (C) provides summary threshold reports, not granular line-item data. Trusted Advisor (D) gives recommendations, not billing data.
Question 10 — Savings Plans
A company currently spends $50,000/month on EC2 instances across multiple regions and instance families. They are actively modernising and expect to change instance types and regions over the next year. They want the MAXIMUM discount while retaining flexibility. Which option should they choose?
A Standard Reserved Instances for 1 year, All Upfront
B EC2 Instance Savings Plans for current instance families
C Compute Savings Plans for 1 year, All Upfront
D On-Demand Capacity Reservations for all regions
✅ Correct Answer: C
Compute Savings Plans provide up to 66% discount on EC2, Lambda, and Fargate — and are the most flexible: they apply automatically to any instance family, size, OS, region, or tenancy. Since the company plans to change instance types and regions, this flexibility is critical. Standard RIs (A) are locked to specific instance type + region — no flexibility during modernisation. EC2 Instance Savings Plans (B) are locked to a specific instance family + region — still too rigid. On-Demand Capacity Reservations (D) reserve capacity but provide NO discount.
Question 11 — Savings Plans vs RIs
A company runs Amazon RDS MySQL and Amazon ElastiCache Redis instances 24/7. They want to reduce costs on these services. Which pricing option provides the BEST discount?
A Compute Savings Plans covering RDS and ElastiCache
B EC2 Instance Savings Plans — they cover all AWS compute services
C Reserved Instances for RDS and ElastiCache separately
D SageMaker Savings Plans extended to RDS workloads
✅ Correct Answer: C
Reserved Instances are the only commitment-based discount model for RDS and ElastiCache. Savings Plans (A, B) do NOT cover RDS or ElastiCache — they only cover EC2, Lambda, Fargate (Compute SP) or EC2 + specific families (EC2 Instance SP). SageMaker Savings Plans (D) only cover SageMaker workloads. This is a critical exam trap: always remember that Savings Plans don't replace RIs for database services.
Question 12 — Scenario Synthesis
A company receives Stripe payment webhooks and needs to: (1) immediately acknowledge Stripe's HTTP request, (2) process each payment exactly once in the correct order, (3) notify the finance team via email for every payment over £10,000, and (4) archive all payment events to S3. What combination of services BEST meets all requirements?
A API Gateway → Lambda → SQS Standard → Lambda processors for all three needs
B EventBridge Partner integration (Stripe) → SQS FIFO → SNS → Kinesis Firehose
C API Gateway → Lambda (acknowledge) → SQS FIFO (ordered processing) + SNS (email alert with filter policy) + S3 (via Kinesis Firehose or Lambda)
D Amazon MQ → SNS FIFO → S3 directly
✅ Correct Answer: C
Breaking down requirements: (1) API Gateway + Lambda acknowledges Stripe immediately (async pattern). (2) SQS FIFO guarantees exactly-once, ordered processing for each payment. (3) SNS with a filter policy (amount > 10000) sends email only for large payments — Lambda publishes to SNS after processing. (4) Another Lambda or Kinesis Firehose archives to S3. Option B is close but EventBridge Partner + SQS FIFO + SNS misses the immediate HTTP acknowledgement to Stripe and has an ordering issue between EventBridge and SQS. MQ (D) is for migrating existing on-prem brokers, not building new cloud-native flows.
