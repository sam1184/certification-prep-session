🔐 Security, Identity & Compliance
8 Services · Highest exam weight · Every architecture question has a security angle
⏱ 40 min
🏗️ The Big Picture First (Beginners: Read This):
AWS Security works in layers. Think of it like a medieval castle: outer walls (Network Firewall) → drawbridge (IAM) → guards (Inspector/Security Hub) → vault (KMS) → the key-keeper (Secrets Manager). Macie is the spy-catcher who finds your crown jewels left in the open. RAM lets you invite allies into specific rooms without giving them the whole castle.
🪪
AWS Identity and Access Management (IAM)
The foundation of ALL AWS security · Controls who can do what to which resource
CRITICAL — Always on Exam
🎯 Sticky Analogy
IAM is like a corporate badge system. Users = employees (long-term badges). Roles = visitor passes (temporary, assumed). Groups = departments (Marketing, Finance). Policies = the room access list printed on the badge. Never give a person a master key — give them exactly the rooms they need.
Users
People / Apps
Long-term credentials. Avoid in production — use Roles.
Groups
Collections of Users
Attach policies to groups, not individual users.
Roles
Temporary Identity
EC2, Lambda, Cross-account. No static credentials!
Policies
JSON Permissions
Allow/Deny. Effect + Action + Resource + Condition.

The Two Types of Policies — Know the Difference

📋 Identity-Based Policy
✔ Attached to a User, Group, or Role
✔ Controls what that identity CAN do
✔ Example: Allow EC2 role to read S3
✔ Managed (AWS/Customer) or Inline
🛡️ Resource-Based Policy
✔ Attached to a resource (S3, KMS, SQS…)
✔ Controls WHO can access that resource
✔ Enables cross-account access without assuming a role
✔ Always has a Principal field

IAM Permission Evaluation Logic — The Exam Loves This

1. Explicit Deny?
→
YES = DENY (always wins)
→
Explicit Allow?
→
YES = ALLOW
→
Default = DENY
🎯 SAA-C03 Exam Tips — IAM:
• EC2 needs to call S3? → Attach an IAM Role to EC2. Never store Access Keys in code/env vars.
• Need cross-account access? → IAM Role with Trust Policy in target account + assume-role in source.
• Keyword: "least privilege" = use IAM conditions, resource-level permissions.
• SCP (Service Control Policy) = guardrails for entire AWS Organizations accounts. Applied at OU/account level. Even the root user can be restricted.
• Permission Boundary = limits the maximum permissions an IAM entity can have. Exam loves this.
• IAM Access Analyzer = finds resources shared with external accounts/internet. Automated security audit.
🏗️ Architecture — IAM Cross-Account Role Assumption & Permission Evaluation
Cross-account role assumption via STS · IAM permission evaluation order: SCP → Explicit Deny → Perm Boundary → Resource Policy → Identity Policy → ALLOW

AWS Key Management Service (AWS KMS)
Managed encryption key service · Used by almost every AWS service
HIGH FREQUENCY
🎯 Sticky Analogy
KMS is the bank vault manager. You don't keep the vault key — the bank does. You just say "encrypt this with my key" and "decrypt this for me." KMS never gives you the actual key material — all crypto happens inside its hardware security module (HSM). It's like depositing a locked box and the bank keeps the key safe for you.
CMK
Customer Master Key
AWS-managed (free) or Customer-managed ($1/month). You control rotation policy.
4KB
Max Direct Encrypt
KMS encrypts data directly up to 4KB. For larger data, use Envelope Encryption.
DEK
Data Encryption Key
Envelope encryption: generate DEK with KMS, encrypt data locally, store encrypted DEK.

Envelope Encryption — The Exam Pattern

App requests DEK from KMS
→
KMS returns Plaintext DEK + Encrypted DEK
→
App encrypts data with Plaintext DEK locally
→
Store Encrypted DEK alongside encrypted data
→
Wipe Plaintext DEK from memory
🏗️ Architecture — KMS Envelope Encryption Flow (Encrypt Large Data)
Envelope Encryption: KMS generates the DEK · App encrypts data locally with DEK · Only encrypted DEK is stored · KMS never sees your data directly
🎯 SAA-C03 Exam Tips — KMS:
• S3 default encryption → SSE-S3 (AWS manages keys) vs SSE-KMS (you control keys, audit via CloudTrail) vs SSE-C (you provide keys).
• RDS encryption → enable at creation, uses KMS CMK. Cannot encrypt an existing unencrypted RDS directly — must snapshot → copy (enable encryption) → restore.
• Multi-Region Keys → replicate CMK to other regions for DR / global applications. Same key material, different ARN per region.
• CloudTrail logs every KMS API call → full audit trail for who used which key when.
• Keyword: "bring your own key" (BYOK) = Customer-managed CMK with imported key material.

AWS Secrets Manager
Stores, rotates, and retrieves secrets (passwords, API keys, DB credentials)
MEDIUM FREQUENCY
🎯 Sticky Analogy
Secrets Manager is the password manager for your cloud — like 1Password but for AWS services. It doesn't just store passwords; it automatically rotates them on a schedule and updates the app — so your DB password changes every 30 days with zero downtime. No more hardcoded credentials in environment variables!
🤫 Secrets Manager
✔ Stores secrets (DB passwords, API keys)
✔ Automatic rotation via Lambda
✔ Native RDS/Redshift/DocumentDB integration
✔ Cross-account sharing
✗ Costs $0.40/secret/month
⚙️ SSM Parameter Store
✔ Config data + secrets (Standard = free)
✗ No native auto-rotation
✔ Hierarchical naming (/prod/db/password)
✔ SecureString tier uses KMS
✔ Free for Standard tier
🎯 Exam Trigger Words:
• "Automatically rotate" or "no hardcoded credentials" → Secrets Manager
• "Store config" or "free" or "hierarchical parameters" → SSM Parameter Store
• RDS password rotation exam scenario → Secrets Manager + Lambda rotation function
• Both encrypt using KMS under the hood

AWS Network Firewall
Managed stateful/stateless network firewall at the VPC perimeter
MEDIUM FREQUENCY
🎯 Sticky Analogy
Network Firewall is the security checkpoint at the entrance of your data centre. Every packet is inspected before it enters the building. It can do deep packet inspection (reads the contents, not just the address), block specific domains, and stop malware signatures — far more powerful than a simple Security Group.

AWS Firewall Comparison — Know Which Layer Each Operates At

Service Layer Where It Lives What It Inspects Key Use Case
Security Groups L3/L4 EC2/ENI level IP + Port + Protocol Allow/deny by IP range to instances
Network ACLs L3/L4 Subnet level IP + Port + Protocol Stateless subnet-wide block rules
AWS WAF L7 CloudFront / ALB / API GW HTTP headers, body, URI Block SQL injection, XSS, IP reputation
AWS Network Firewall L3–L7 VPC (dedicated subnet) Stateful + Deep Packet Inspection Domain filtering, IDS/IPS, advanced VPC security
AWS Shield L3/L4 Global (Standard = always on) Network-level DDoS traffic DDoS protection for all AWS resources
🎯 Trigger Words:
• "Deep packet inspection" or "domain filtering" or "IDS/IPS" at VPC level → Network Firewall
• "SQL injection" or "web application attack" → WAF
• "DDoS" → Shield (Standard free, Advanced = $3,000/month with SRT team)
• Network Firewall requires a dedicated firewall subnet in your VPC + route table changes

Amazon Inspector
Automated vulnerability scanning for EC2, containers, and Lambda functions
KNOW FOR EXAM
🎯 Sticky Analogy
Inspector is the building safety inspector who comes and checks your servers for cracks in the walls — known CVEs, unpatched packages, exposed ports, and software vulnerabilities. It continuously scans and reports risk scores. Unlike GuardDuty (which detects active threats), Inspector finds vulnerabilities before attackers do.
🖥️ EC2 Scanning
SSM agent required. Scans OS packages, network reachability. Continuous, not scheduled.
📦 Container Scanning
Scans ECR images when pushed. Also scans running ECS/EKS workloads for vulnerabilities.
λ Lambda Scanning
Scans Lambda function code and layers for vulnerabilities in packages.
🎯 Inspector vs GuardDuty — Classic Confusion:
• Inspector = "What vulnerabilities do my resources HAVE?" (Proactive scanning)
• GuardDuty = "Is something malicious HAPPENING right now?" (Active threat detection — ML on CloudTrail/DNS/VPC Flow logs)
• Trigger: "CVE" or "vulnerability scan" or "patch compliance" → Inspector
• Trigger: "suspicious activity" or "compromised instance" or "crypto mining" → GuardDuty

Amazon Macie
ML-powered sensitive data discovery and classification in S3
KNOW FOR EXAM
🎯 Sticky Analogy
Macie is the data privacy detective on your S3 buckets. It uses machine learning to find PII (Personally Identifiable Information) — credit card numbers, SSNs, passports, health records — hiding in your data lake. Think GDPR/HIPAA compliance auditor, running 24/7 automatically on all your S3 data.
🔑 One-Line Memory Hook: Macie = Machine learning for Audit of PII in S3 → MA(S3)

What Macie finds: Credit card numbers · SSNs · Health records · Passport numbers · Email addresses · Custom patterns you define
Where it looks: Only S3 (not EBS, not RDS, not DynamoDB)
What happens: Findings → Security Hub → EventBridge → Auto-remediation (e.g., block public access)
🎯 Trigger Words:
• "Sensitive data" or "PII" or "GDPR" or "HIPAA" in S3 → Macie
• "Classify data" in S3 → Macie
• "Public bucket with sensitive data" → Macie discovers it; S3 Block Public Access prevents it

AWS Resource Access Manager (AWS RAM)
Share AWS resources across accounts or within AWS Organizations — without duplication
KNOW FOR EXAM
🎯 Sticky Analogy
RAM is like a shared company car pool. Instead of each department buying their own car (duplicate VPC subnets, Transit Gateways, etc.), you share one set of resources across all accounts. Account A creates a VPC subnet → RAM shares it with Accounts B, C, D → all accounts can launch resources into it without peering complexity.

What Can Be Shared with RAM?

🌐 VPC Subnets
Share subnets across accounts in same Organization. Most common exam scenario.
🔗 Transit Gateway
Share a single TGW across accounts — hub for inter-VPC and on-prem connectivity.
🔍 Route53 Resolver
Share DNS resolver rules across accounts for centralised DNS resolution.
🖼️ License Manager
Share software license configurations (Windows, Oracle, etc.) across accounts.
🎯 Trigger Words:
• "Share resources across accounts" → AWS RAM
• "Centralised VPC" or "shared services VPC" → RAM + VPC subnet sharing
• "Without VPC peering" and multiple accounts need subnet access → RAM
• RAM works within same Organization — no cross-org sharing

📊 Analytics, Data Engineering & BI
11 Services · Know when to use each tool in the data pipeline
⏱ 45 min
🏗️ The Big Picture First (Beginners: Read This):
Modern data systems have 4 stages: Ingest → Store → Process → Visualise. Each AWS analytics service has a specific role. The exam tests whether you can pick the right service for the right stage. Here's the mental map:

Ingest: Kinesis (streaming) · MSK (Kafka) · Data Exchange (3rd party) · Data Pipeline (batch)
Store: S3 (data lake) · Lake Formation (governed lake) · Redshift (warehouse)
Process: Glue (ETL) · EMR (big data frameworks) · Athena (query in place)
Visualise: QuickSight (BI) · OpenSearch (search/dashboards)
🔎
Amazon Athena
Serverless SQL query engine — query data directly in S3 without loading it anywhere
HIGH FREQUENCY
🎯 Sticky Analogy
Athena is like having a SQL terminal pointed directly at your S3 bucket. No servers. No ETL pipeline. No data movement. You write SELECT * FROM my_logs WHERE date > '2024-01-01' and Athena reads the files in S3 and returns results. You pay per TB scanned — so use Parquet + partitioning to save money.
Serverless
No Infrastructure
No clusters. No provisioning. Just SQL + S3.
$5/TB
Pay per Scan
Parquet/ORC format = 30–90% cost savings. Partition data = fewer scans.
Federated
Query Anywhere
Athena Federated Query: query DynamoDB, RDS, on-prem via Lambda connectors.
Athena Architecture — The Canonical Pattern
Athena + Glue Data Catalog + QuickSight = the serverless analytics trifecta on the exam
🎯 Trigger Words:
• "Query S3 with SQL" or "ad-hoc analysis" or "serverless query" → Athena
• "Cost Usage Report analysis" → Athena + CUR (classic exam scenario)
• "VPC Flow Logs analysis" → Athena queries Flow Logs stored in S3
• Athena needs Glue Data Catalog to know the schema of your S3 data
🔧
AWS Glue
Serverless ETL service + Data Catalog — discover, prepare, and transform data
HIGH FREQUENCY
🎯 Sticky Analogy
AWS Glue is the data janitor + librarian. The Glue Crawler crawls your data sources (S3, RDS, DynamoDB) and creates a Data Catalog (the library card index — who has what schema). Then Glue ETL Jobs are the janitor that cleans, transforms, and moves the data from one place to another using Apache Spark under the hood.
🕷️ Glue Crawlers
Automatically discover schema from S3, RDS, JDBC sources. Populates Data Catalog.
📚 Data Catalog
Central metadata repository. Shared with Athena, EMR, Redshift Spectrum. One source of truth.
⚙️ ETL Jobs
Serverless Apache Spark jobs. Python/Scala. Transform CSV → Parquet. Clean & enrich data.
🎯 Glue vs EMR — Classic Exam Trap:
• Glue = Serverless ETL. No servers to manage. Best for straightforward transformations. Python/Scala scripts.
• EMR = Managed Hadoop/Spark clusters. YOU manage cluster size. Best for complex big data (Hive, Presto, HBase, custom Spark). More control, more cost.
• Trigger: "no infrastructure management" + ETL → Glue
• Trigger: "custom Hadoop" or "Hive" or "HBase" or "large-scale Spark cluster" → EMR
🐘
Amazon EMR (Elastic MapReduce)
Managed big data platform — Hadoop, Spark, Hive, Presto, HBase on EC2/EKS/Serverless
MEDIUM FREQUENCY
🎯 Sticky Analogy
EMR is like renting a whole data centre computing cluster for as long as you need it. You tell AWS "give me 100 nodes running Apache Spark" and within minutes you have a petabyte-scale processing engine. When your job is done, you shut it all down. Perfect for ML training data prep, genomics analysis, log processing at massive scale.
EC2 Mode
Classic Clusters
Master + Core + Task nodes. Use Spot for task nodes = 60–70% savings.
EKS Mode
Kubernetes-based
Run Spark on EKS. Container-based. Share cluster with other EKS workloads.
Serverless
No cluster sizing
Auto-scales workers. No min/max capacity decisions. Pay per vCPU-second.
🎯 Trigger Words for EMR:
• "Apache Hadoop" / "Apache Spark" / "Apache Hive" / "HBase" / "Presto" → EMR
• "Petabyte-scale" processing → EMR
• "Custom big data frameworks" → EMR (not Glue)
• Storage: EMR data on S3 (EMRFS) survives cluster termination; HDFS is ephemeral on cluster
• Use Spot instances for Task nodes — if they terminate, in-flight tasks restart; Core/Master should be On-Demand
🌊
Amazon Kinesis
Real-time streaming data platform — 4 sub-services, know each one's job
CRITICAL — Always on Exam
🎯 Sticky Analogy
Kinesis is the river system for your data. Data Streams is the raw river (you control the flow). Firehose is a managed canal with a destination (just pour data in, it arrives at S3/Redshift). Data Analytics is the analysis station along the river doing real-time calculations. Video Streams is a separate stream just for video.
Kinesis Service What It Does Retention Consumers Exam Trigger
Kinesis Data Streams Ingest + replay streaming data. You manage shards. 1–365 days Multiple (Lambda, KCL, Analytics) "Custom processing" / "replay" / "multiple consumers"
Kinesis Firehose Managed delivery to S3/Redshift/OpenSearch. No code. No storage (buffer only) One destination "Load streaming data to S3" / "near real-time" / "no management"
Kinesis Data Analytics Real-time SQL/Flink on streaming data N/A (processes live) Reads from KDS or Firehose "Real-time aggregation" / "anomaly detection on stream"
Kinesis Video Streams Ingest + store + process video streams Configurable Rekognition, SageMaker "Video streaming" / "live camera feeds" / "ML on video"
⚠️ Kinesis vs SQS — The #1 Exam Trap:
• Kinesis: Multiple consumers, ordered per shard, replay data up to 365 days, streaming analytics → use for real-time analytics and multiple consumer apps
• SQS: One consumer per message, at-least-once, decouple services, retry logic → use for async task processing, decoupling microservices
• Both need "real-time"? Kinesis = many readers of same stream. SQS = one worker processes and deletes.
🏗️ Architecture — Amazon Kinesis: Streams vs Firehose — Know the Difference
KDS = raw river (multiple consumers, replay, you manage shards) · Firehose = managed canal (one destination, no replay, serverless) · Analytics = real-time SQL engine on top of either

Amazon Redshift
Petabyte-scale data warehouse — OLAP, not OLTP
HIGH FREQUENCY
🎯 Sticky Analogy
Redshift is like a giant corporate reporting database. Your OLTP database (RDS) handles today's transactions. Redshift handles the question "What were our global sales trends for the last 5 years, broken down by region, product, and customer segment?" — complex analytical queries across billions of rows that would crush an RDS instance.
Columnar
Storage Format
Reads only needed columns. Massive compression. Perfect for aggregations.
Spectrum
Query S3 Direct
Redshift Spectrum: query data in S3 without loading. Uses Glue Data Catalog.
Serverless
New Option
Auto-scales. No cluster management. Pay per compute second used.
🎯 Trigger Words:
• "Data warehouse" or "OLAP" or "business intelligence" → Redshift
• "Petabyte analytics" → Redshift
• RDS = OLTP (transactions, row-based, high concurrency) · Redshift = OLAP (analytics, columnar, complex queries)
• Redshift Spectrum = extend warehouse to query S3 data lake without ETL
• Enhanced VPC Routing: force all Redshift traffic through your VPC (not public internet)

AWS Lake Formation
Simplifies building, securing, and managing data lakes on S3
KNOW FOR EXAM
🎯 Sticky Analogy
AWS Lake Formation is the zoning board and permit office for your data lake. You could build your data lake manually (IAM policies + Glue + S3 bucket policies — 10 different controls). Or you use Lake Formation, which provides a single unified permission model — one place to say "the Data Science team can query this table, but only the Name column, not the SSN column."
🔑 Key Lake Formation Capability — Column/Row Level Security:
Lake Formation allows fine-grained access control at the column and row level for data in S3, queried via Athena, EMR, or Redshift. This is not possible with S3 bucket policies alone (which are object-level). This is a favourite exam differentiator.
🏗️ Architecture — End-to-End Data Lake: Glue + Lake Formation + Athena + QuickSight
Data Lake stack: raw S3 → Glue ETL (CSV→Parquet) → Lake Formation (column/row security) → Athena / Redshift Spectrum / EMR → QuickSight dashboards
🎯 Trigger Words:
• "Build a data lake" quickly → Lake Formation
• "Column-level security" on S3 data → Lake Formation (not raw S3 policies)
• "Centralised access control" for data lake → Lake Formation
• Lake Formation sits on top of S3 + Glue Data Catalog and adds fine-grained permissions

🚀 Migration, Transfer & Hybrid Connectivity
6 Services · The "7 Rs of Migration" in action · Know the right tool for each migration type
⏱ 35 min
🏗️ Migration Mental Model (Beginners: Start Here):
Think of migration as moving house. Migration Hub = the moving coordinator tracking everything. Application Discovery Service = the surveyor who maps what's in your current house. Application Migration Service = the van that lifts your servers to the cloud (lift-and-shift). DataSync = fast courier for files and NFS data. Snow Family = shipping containers for moving massive amounts of data physically. Transfer Family = keeping your old door locks (FTP/SFTP) on the new house.
🗺️
AWS Migration Hub
Central tracking dashboard for all migrations — single pane of glass for your migration programme
AWARENESS LEVEL
One-Liner Memory Hook: Migration Hub = project management dashboard for cloud migration. Track progress of servers being migrated across multiple AWS migration tools (MGN, DMS, SMS). See what's been migrated, what's in progress, and what's pending — across all regions and migration tools in one view.
🎯 Trigger Words:
• "Track migration progress" across tools → Migration Hub
• "Centralised migration dashboard" → Migration Hub
• Migration Hub integrates with: Application Discovery Service, Application Migration Service (MGN), Database Migration Service (DMS)
🔬
AWS Application Discovery Service
Discover and map on-premises infrastructure before migrating
KNOW FOR EXAM
🎯 Sticky Analogy
Application Discovery Service is the building surveyor before demolition. Before you migrate, you need to know what you have: which servers talk to which, what dependencies exist, what performance baselines look like. ADS gives you this map, so you don't accidentally cut a critical connection when migrating.
🔌 Agentless Discovery
✔ Uses VMware vCenter connector
✔ No agent on each server needed
✔ Collects VM-level data (CPU, RAM, disk)
✗ Less detail (no process-level data)
💉 Agent-Based Discovery
✔ Install AWS Discovery Agent on each server
✔ Detailed: processes, network connections, performance
✔ Works on physical servers + VMs
✗ Requires agent deployment effort
🎯 Trigger Words:
• "Discover" or "assess" on-premises servers before migration → Application Discovery Service
• "Server dependencies" or "application dependencies" → ADS
• Data from ADS feeds into Migration Hub for tracking

AWS Application Migration Service (MGN)
Automated lift-and-shift migration — replicate and launch servers on AWS
HIGH FREQUENCY
🎯 Sticky Analogy
MGN is the "beam me up" migration tool. It continuously replicates your on-premises servers to AWS using block-level replication (like a live copy). When you're ready to cut over, you flip the switch — the server launches on EC2 within minutes, with minimal downtime. No application changes required.
MGN Migration Flow
MGN: continuous block-level replication → test → cutover. No data loss, minimal downtime lift-and-shift.
🎯 Trigger Words:
• "Lift-and-shift" or "rehost" → Application Migration Service (MGN)
• "Minimal downtime" server migration → MGN (continuous replication)
• "Replace SMS" → MGN is the successor to Server Migration Service
• MGN vs DMS: MGN = server migration (OS + apps). DMS = database migration (data only).

Fast, automated data transfer — NFS/SMB/S3 on-premises to/from AWS at up to 10Gbps per task
HIGH FREQUENCY
🎯 Sticky Analogy
DataSync is the high-speed fibre courier. You need to move 50TB of NAS/NFS files from your data centre to S3. Instead of building a custom sync script, DataSync handles it — with automatic encryption, integrity checks, scheduling, and bandwidth throttling. It also syncs ongoing changes, not just one-time transfers.

✔ NFS, SMB, HDFS, S3-compatible, EFS, FSx
✔ Up to 10Gbps per task
✔ Scheduling, retry, integrity verification
✔ Online transfer over internet or Direct Connect
✔ Ongoing sync or one-time migration

✔ Offline physical data transfer
✔ Petabytes in days (vs months over internet)
✔ Works in disconnected / limited bandwidth locations
✗ Days to weeks for device logistics

• "NFS to S3" or "SMB to EFS" or "file server migration" → DataSync
• "Bandwidth throttling" or "scheduled transfer" → DataSync
• DataSync vs Storage Gateway: DataSync = data transfer/migration. Storage Gateway = ongoing hybrid access (on-prem app accesses cloud storage as if local).

Physical data transfer devices — Snowcone, Snowball Edge, Snowmobile
HIGH FREQUENCY
🎯 Sticky Analogy
The Snow Family answers the question: "What's faster — shipping a hard drive or uploading over the internet?" For large datasets, physics wins. At 1Gbps internet speed, 1 petabyte takes 33 years. With a Snowmobile (a literal shipping container on a truck), it takes days. Never underestimate the bandwidth of a truck full of hard drives.

🧪 Practice Questions — 12 SAA-C03 Style Scenarios

These questions mirror the style, difficulty, and trick patterns of the actual SAA-C03 exam. Click an answer, then reveal the explanation. Each explanation tells you WHY the wrong answers are wrong — that's where the real learning happens.

Question 1 — IAM Roles & EC2
A development team is deploying a Python application on EC2 that needs to read objects from an S3 bucket in the same account. A junior developer suggests adding the Access Key ID and Secret Access Key to the application's environment variables on the EC2 instance. What is the MOST secure recommended approach?
A Store the Access Key and Secret Key in AWS Secrets Manager and have the app retrieve them at startup
B Create an IAM Role with an S3 read policy, attach the role to the EC2 instance, and remove all hardcoded credentials
C Encrypt the Access Key and Secret Key using KMS and store them in the EC2 user data
D Store the credentials in SSM Parameter Store as SecureString and retrieve them via the AWS SDK at application start
✅ Correct Answer: B
IAM Roles eliminate the need for any credentials. When you attach a role to an EC2 instance, the instance metadata service (IMDS) provides temporary credentials automatically — rotated every hour. No credentials to manage, no credentials to leak. Option A (Secrets Manager) is better than hardcoding but still requires you to manage credentials — a role requires nothing. Option C (user data) is visible in plain text. Option D (SSM Parameter Store) is better than env vars but still manages long-term keys. The golden rule: EC2 accessing other AWS services = always an IAM Role.
Question 2 — Encryption at Rest
A financial company stores customer data in RDS MySQL. After a recent audit, the compliance team requires that all data be encrypted at rest with keys that the company controls and can rotate on a custom schedule. The current RDS instance is unencrypted. What should a solutions architect recommend?
A Enable RDS encryption in the AWS console by modifying the existing instance
B Use AWS managed keys (aws/rds) and enable encryption on the existing instance
C Take a snapshot of the unencrypted RDS instance, copy the snapshot with encryption enabled using a Customer Managed KMS CMK, then restore a new RDS instance from the encrypted snapshot
D Enable RDS encryption on the existing instance and select a Customer Managed KMS key
✅ Correct Answer: C
You cannot enable encryption on an existing unencrypted RDS instance. This is a hard AWS constraint. The only path is: Snapshot → Copy (encrypt during copy) → Restore. Option A and D are wrong because modifying an existing instance cannot enable encryption. Option B uses AWS-managed keys which AWS controls the rotation schedule — the question requires customer-controlled rotation. A Customer Managed CMK (option C) allows you to define your own rotation schedule (e.g., 90 days) and gives you full key policy control. This process has zero data loss — the new instance is identical to the original.
Question 3 — Security Services Differentiation
A security team receives an alert that one of their EC2 instances is communicating with a known command-and-control (C2) server associated with a crypto-mining botnet. Which AWS service MOST LIKELY generated this finding?
A Amazon Inspector
B AWS Security Hub
C Amazon GuardDuty
D AWS Config
✅ Correct Answer: C
GuardDuty detects active threats using ML on VPC Flow Logs, DNS logs, and CloudTrail. Communication with a known malicious IP (C2 server) is exactly what GuardDuty finds. Inspector scans for vulnerabilities in your software (CVEs, unpatched packages) — proactive, not reactive. Security Hub aggregates findings FROM GuardDuty and others but doesn't generate the finding itself. AWS Config tracks configuration compliance (e.g., "is MFA enabled?") not network behaviour. Memory hook: GuardDuty = threat detection (active) · Inspector = vulnerability scanning (proactive) · Security Hub = aggregator · Config = compliance checker.
Question 4 — S3 Data Classification
A healthcare company has hundreds of S3 buckets containing historical files. The legal team suspects some buckets contain patient health records (PHI) that were accidentally placed in public buckets years ago. Which AWS service should be used to identify and alert on this sensitive data?
A AWS Config with a custom rule that checks for PHI keywords in S3 object metadata
B Amazon Macie
C Amazon Inspector
D Amazon GuardDuty
✅ Correct Answer: B
Amazon Macie uses ML to discover PII and sensitive data inside S3 objects. It reads the actual file content (not just metadata) and can identify PHI, PII, credit card numbers, SSNs, and custom sensitive patterns. Inspector scans EC2/containers/Lambda for software vulnerabilities — not S3 data classification. GuardDuty analyses behavioural patterns and network traffic — not data content. AWS Config rules check configuration states (bucket public access settings) but cannot read and classify data contents. Macie is the only service that actually reads inside S3 objects to classify sensitive data.
Question 5 — Analytics Service Selection
A data engineering team stores 5 years of application logs in S3 in compressed JSON format. The team wants to run occasional ad-hoc SQL queries to investigate incidents but doesn't want to manage any infrastructure or move the data. Queries run less than 10 times per day. What is the MOST cost-effective solution?
A Load the logs into Amazon Redshift Serverless and query using standard SQL
B Use AWS Glue Crawlers to catalogue the JSON files, then query with Amazon Athena
C Launch an EMR cluster, load data from S3 to HDFS, and run Hive queries
D Create an OpenSearch domain and ingest all S3 logs for full-text search
✅ Correct Answer: B
Athena is purpose-built for ad-hoc SQL queries directly on S3 data — no infrastructure, no data movement, pay per TB scanned. Glue Crawlers automatically discover the JSON schema and populate the Data Catalog so Athena knows the table structure. Less than 10 queries/day = very low scan costs. Redshift Serverless is more cost-effective at high query frequency — with 10 queries/day, Athena's per-scan model wins. EMR requires cluster management and data loading overhead — overkill and expensive for ad-hoc. OpenSearch is for search, not SQL analytics, and requires loading all data into its index. Pro tip: convert logs to Parquet format to reduce scan costs by 70%+.
Question 6 — Real-time Streaming
A ride-sharing company needs to ingest GPS location data from 100,000 active vehicles in real time. Three separate systems must consume this stream simultaneously: (1) a fraud detection ML model, (2) a real-time map update service, and (3) a data lake pipeline writing to S3. The team needs the ability to replay up to 7 days of data. What should the architect recommend?
A Amazon SQS Standard Queue with fan-out to three separate queues via SNS
B Amazon Kinesis Data Streams with Enhanced Fan-Out, three separate consumer applications, retention set to 7 days
C Amazon Kinesis Firehose delivering to S3, with Lambda triggers for the other two consumers
D Amazon MSK (Managed Apache Kafka) with three consumer groups subscribed to the topic
✅ Correct Answer: B
Kinesis Data Streams with Enhanced Fan-Out is purpose-built for this pattern. Key requirements met: (1) Multiple simultaneous consumers — KDS allows multiple consumer applications reading the same stream independently. (2) Replay — KDS retains data for 1–365 days (7 days = standard setting). (3) Real-time — Enhanced Fan-Out provides 2MB/s per shard per consumer at millisecond latency. Option A (SQS + SNS): SQS messages are deleted after consumption — no replay capability. Option C (Firehose): Firehose has one destination; it can't fan-out to three consumers, and has no replay. Option D (MSK) would technically work but the scenario doesn't mention existing Kafka workloads — KDS is the AWS-native answer.
Question 7 — ETL Service Selection
A company receives daily CSV files from 50 retail partners in an S3 bucket. They need to transform, clean, and convert these files to Parquet format before loading them into Redshift. The operations team has no Hadoop or Spark expertise and wants minimal infrastructure to manage. What is the BEST solution?
A Launch an EMR cluster daily using a Step Function, run a Spark job, terminate the cluster
B Create AWS Glue ETL jobs (serverless) triggered on S3 event, transform CSV to Parquet, load to Redshift
C Use AWS Data Pipeline with a custom EC2 activity to process each file
D Write an AWS Lambda function to process each CSV file and load to Redshift directly
✅ Correct Answer: B
AWS Glue is serverless ETL — no cluster management, no Spark expertise required. Glue Studio provides a visual interface; scripts are auto-generated Python/Scala Spark code. It natively converts CSV to Parquet and has a direct Redshift connector. EMR (A) requires Spark knowledge and cluster management — the question specifically says no Hadoop/Spark expertise and minimal infrastructure. Data Pipeline (C) is legacy, uses EC2, not serverless. Lambda (D): Lambda has a 15-minute timeout and 10GB memory limit — not suitable for processing large CSV files from 50 partners. Lambda is stateless compute, not ETL. The key discriminator: "no Spark expertise" + "minimal infrastructure" = Glue every time.
Question 8 — Migration Strategy
A manufacturing company wants to migrate 200 physical servers from their on-premises data centre to AWS. They need to: (1) first understand what applications are running and their interdependencies, (2) track the migration progress centrally, and (3) perform the actual server migration with minimal downtime. Which combination of AWS services addresses ALL three requirements?
A AWS Config → AWS Migration Hub → AWS DataSync
B Amazon Inspector → AWS Migration Hub → AWS Transfer Family
C AWS Application Discovery Service → AWS Migration Hub → AWS Application Migration Service (MGN)
D AWS Trusted Advisor → AWS Migration Hub → AWS Snow Family
✅ Correct Answer: C
This is the canonical migration service trio. (1) Application Discovery Service: discovers on-premises servers, maps application dependencies, collects performance data. (2) Migration Hub: centralised dashboard tracking all migration progress — integrates with ADS and MGN. (3) Application Migration Service (MGN): continuous block-level replication of servers to AWS with minimal cutover downtime (minutes). Option A: DataSync moves files/NFS data, not entire server workloads. Option B: Inspector scans for vulnerabilities, not server discovery; Transfer Family handles SFTP, not server migration. Option D: Trusted Advisor gives cost/performance recommendations; Snow Family is for offline data transfer, not server lift-and-shift.
Question 9 — Data Transfer Selection
A genomics research lab has 800TB of sequencing data on-premises that must be uploaded to S3. Their internet connection is 1Gbps and is also used for active research operations (cannot be fully dedicated to the transfer). The compliance team requires the transfer to complete within 2 weeks. What is the MOST appropriate solution?
A Use AWS DataSync with bandwidth throttling, limiting to 500Mbps so research operations are not impacted
B Set up AWS Direct Connect with a 10Gbps dedicated circuit
C Order multiple AWS Snowball Edge Storage Optimised devices, load data locally, ship to AWS
D Use S3 Transfer Acceleration over the existing internet connection
✅ Correct Answer: C
The maths disqualify the internet options. 800TB at 500Mbps (shared internet) = 800TB ÷ 62.5GB/hour ≈ 37 days. The requirement is 2 weeks. Multiple Snowball Edge devices each hold 80TB usable storage — you'd order ~10 devices, load them in parallel (local network speed, not internet speed), and ship. Total time: device delivery (2–3 days) + loading time (local 10Gbps+ NFS speeds) + shipping back (2–3 days) = well within 2 weeks. Option A (DataSync at 500Mbps): 37 days — fails the deadline. Option B (Direct Connect): provisioning takes weeks minimum and 800TB at 10Gbps still takes 18 hours just for transfer; overkill for a one-time migration. Option D (S3 Transfer Acceleration): uses same internet connection — slightly faster than S3 direct but still constrained by 1Gbps internet.
Question 10 — Cross-Account Resource Sharing
A company uses AWS Organizations with a central networking account and 15 application accounts. The networking team wants to deploy a shared VPC with subnets so that all 15 application accounts can launch EC2 instances into the shared subnets, without each application account needing its own VPC. What should the solutions architect recommend?
A Create VPC peering connections between the networking account VPC and each of the 15 application account VPCs
B Deploy a Transit Gateway in the networking account and attach all 15 account VPCs
C Use AWS Resource Access Manager (RAM) to share the subnets from the networking account VPC with all 15 application accounts
D Create IAM cross-account roles that allow the 15 accounts to create VPCs in the networking account
✅ Correct Answer: C
AWS RAM subnet sharing is the exact solution for shared VPC architectures across accounts. The networking account creates ONE VPC with subnets, shares the subnets via RAM with all 15 accounts. Each application account can then launch EC2 instances directly into those shared subnets — they appear in the application account's console as available subnets. Option A (VPC Peering): each account would still need its own VPC; peering connects VPCs, it doesn't share them. 15 accounts = 15 VPCs = 15×15=225 peering connections for full mesh = management nightmare. Option B (Transit Gateway): connects VPCs together, doesn't allow applications to share one VPC. Option D: Cross-account IAM roles can't give another account the ability to use your VPC subnets directly.
Question 11 — Secret Rotation
A company's application connects to an Amazon RDS database using a hardcoded password in the application configuration file stored in a private S3 bucket. The security team requires that: (1) the password is never visible in plaintext anywhere, (2) the password is automatically rotated every 30 days, and (3) the application connects without any code changes or downtime during rotation. What is the BEST solution?
A Store the password in SSM Parameter Store as SecureString and update the application to retrieve it at startup. Set a CloudWatch Events rule to rotate the password monthly.
B Store the RDS credentials in AWS Secrets Manager with automatic rotation enabled (30-day schedule). Update the application to call Secrets Manager API at each DB connection. Secrets Manager handles Lambda-based rotation natively for RDS.
C Enable RDS IAM Database Authentication and remove the password entirely. Grant the application's IAM Role permission to connect to RDS.
D Use AWS KMS to encrypt the password and store the encrypted value in DynamoDB. Rotate the password manually every 30 days and update DynamoDB.
✅ Correct Answer: B
AWS Secrets Manager provides native automatic rotation for RDS with zero downtime. It uses a Lambda rotation function that creates a new password in RDS, stores it in Secrets Manager, and the application's SDK call always retrieves the current secret — no code change needed, no config file, no plaintext. Option A (SSM Parameter Store): no native auto-rotation for RDS; requires custom Lambda + CloudWatch Events. The question requires native automatic rotation. Option C (IAM Auth): excellent choice and eliminates passwords entirely — BUT it requires code changes to generate and use IAM auth tokens instead of passwords. The question says no code changes. However, if the question said "eliminate passwords entirely," option C would be even better than B! Option D: manual rotation fails the automatic requirement.
Question 12 — Architecture Synthesis
A retail company is building a real-time analytics platform. Requirements: (1) Ingest 50,000 clickstream events/second from their website, (2) A fraud detection service and a personalisation engine must both process each event in real time, (3) Long-term storage of all events in S3 for future ML training (keep 1 year), (4) Business analysts must query event data with SQL without any infrastructure management, (5) Marketing team needs charts and dashboards. What combination BEST meets ALL requirements?
A Kinesis Firehose → S3 · Lambda for fraud and personalisation · Redshift for SQL · QuickSight for dashboards
B Kinesis Data Streams (multi-consumer) → fraud Lambda + personalisation Lambda + Kinesis Firehose to S3 · Glue Crawler on S3 · Athena for SQL · QuickSight for dashboards
C Amazon MSK → fraud service + personalisation service + DataSync to S3 · EMR for SQL · QuickSight for dashboards
D SQS FIFO → fraud Lambda (reads, deletes) → second SQS → personalisation Lambda → S3 · Athena for SQL · QuickSight for dashboards
✅ Correct Answer: B
Kinesis Data Streams is the only service that allows multiple independent consumers on the same stream simultaneously with replay capability. Here's why each requirement maps to option B: (1) 50K events/sec → KDS scales to handle this with sufficient shards. (2) Two consumers on same stream → KDS Enhanced Fan-Out allows fraud Lambda and personalisation Lambda to read independently — without one blocking the other. (3) 1-year S3 storage → Kinesis Firehose as a third consumer delivers to S3 automatically. (4) SQL on S3, no infrastructure → Glue Crawler detects schema + Athena queries. (5) Dashboards → QuickSight on Athena. Option A fails: Firehose has ONE destination; you can't also have Lambda consumers on Firehose. Option D fails: SQS messages are deleted after consumption — the first Lambda deletes the message before personalisation can read it. FIFO with fan-out doesn't solve this.
