🔐 Section 1 — Identity, Secrets & Security Posture

🔑IAM · KMS · Secrets Manager· Complexity: ★★★★☆
Q 01
IAM KMS Secrets Manager Multi-account

A financial institution runs a multi-account AWS Organization. The central security team (account A) manages a KMS Customer Managed Key (CMK) used to encrypt an RDS database in a workload account (account B). A new compliance requirement states that database credentials must be rotated every 30 days automatically, and the secret must be encrypted with the same CMK managed in account A. A developer suggests four approaches. Which is correct?

  • A Store the RDS credentials in AWS Systems Manager Parameter Store (SecureString) in account B, referencing the CMK ARN from account A. Enable Parameter Store native rotation every 30 days.
  • B Store the RDS credentials in AWS Secrets Manager in account B. In the KMS key policy in account A, add account B as an allowed principal. In Secrets Manager, reference the cross-account CMK ARN and enable automatic rotation with a Lambda rotation function.
  • C Store the credentials in DynamoDB in account A, encrypted with the CMK. Write a scheduled Lambda that reads from DynamoDB and updates the RDS password every 30 days.
  • D Create a replica of the CMK in account B using KMS multi-region keys. Store credentials in Secrets Manager in account B using the replicated key and enable native rotation.
Correct Answer: B
B is correct. AWS Secrets Manager supports cross-account KMS keys natively. You grant account B the ability to use account A's CMK by updating the key policy in account A (adding account B as a principal for kms:GenerateDataKey and kms:Decrypt) and creating an IAM role in account B with permission to call those KMS actions. Secrets Manager in account B can then encrypt secrets with the cross-account CMK ARN, and the built-in Lambda rotation function handles 30-day rotation automatically.
Why not A? SSM Parameter Store SecureString does not support native automatic rotation — you must write your own automation. It also cannot reference cross-account CMK ARNs directly without additional complexity.

Why not C? DynamoDB is not a secrets management service. This is a custom bespoke solution — the exam always prefers managed services over DIY for rotation.

Why not D? Multi-region KMS key replication solves geographic redundancy — not cross-account access. Using a replicated key still lives in account B, defeating the "managed centrally in account A" requirement.
The #1 distractor here is D. "Multi-region key" sounds like "multi-account" but these are different concepts. Multi-region keys sync between regions in the SAME account. Cross-account access requires key policy changes, not replication.
Keyword decode: "automatic rotation" → Secrets Manager (not SSM). "Cross-account CMK" → update KMS key policy in the key owner account + IAM in consumer account. When you see both terms together, the answer involves Secrets Manager + KMS key policy principal grant.

🕵️Macie · Security Hub · Inspector· Complexity: ★★★★☆
Q 02
Macie Security Hub Inspector Scenario: Differentiation

A healthcare company needs to fulfil three separate compliance mandates simultaneously:

Mandate 1: Automatically scan all S3 buckets for protected health information (PHI) and credit card numbers stored unintentionally.
Mandate 2: Continuously assess EC2 instances for known CVEs (Common Vulnerabilities and Exposures) and unintended network exposure.
Mandate 3: Aggregate and normalize security findings from all AWS security services into a single pane of glass with a compliance score against HIPAA.

Which combination of services maps to Mandate 1, Mandate 2, and Mandate 3 respectively?

  • A Mandate 1 → GuardDuty | Mandate 2 → Inspector | Mandate 3 → Security Hub
  • B Mandate 1 → Macie | Mandate 2 → GuardDuty | Mandate 3 → Security Hub
  • C Mandate 1 → Macie | Mandate 2 → Inspector | Mandate 3 → Security Hub
  • D Mandate 1 → Macie | Mandate 2 → Inspector | Mandate 3 → Audit Manager
Correct Answer: C
C is correct — the precise service-to-mandate mapping is:
  • Macie → Mandate 1: Macie uses ML to automatically discover, classify, and protect sensitive data in S3 — specifically PII, PHI, financial data. This is its sole purpose.
  • Inspector → Mandate 2: Amazon Inspector continuously scans EC2 instances AND container images for software vulnerabilities (CVEs) and unintended network reachability. It is the CVE scanner.
  • Security Hub → Mandate 3: Security Hub is the aggregation layer. It collects findings from Macie, Inspector, GuardDuty, Config, Firewall Manager, etc. and maps them against compliance standards including HIPAA, CIS, PCI-DSS.
Option A replaces Macie with GuardDuty for S3 scanning. GuardDuty detects threats and anomalous behaviour (e.g. unusual API calls, crypto-mining) — it does NOT identify PHI data content. Option D swaps Security Hub for Audit Manager. Audit Manager collects evidence for audits but is NOT the aggregation dashboard — Security Hub is.
The "three Musketeers" of AWS security services — memorize their one-liner: Macie = sensitive DATA in S3 | Inspector = software VULNERABILITIES on compute | GuardDuty = threat BEHAVIOUR across accounts. Security Hub = the AGGREGATOR. When you see "compliance dashboard" or "single pane of glass" → Security Hub. When you see "scan S3 for PII/PHI" → Macie, never GuardDuty.

🌐Network Firewall · RAM · IAM· Complexity: ★★★★★
Q 03
Network Firewall RAM Multi-account

A company has a hub-and-spoke AWS Organization with 12 spoke accounts and a shared inspection VPC in the central networking account. The security team wants to deploy AWS Network Firewall in the central inspection VPC and enforce deep-packet inspection for all east-west traffic between spoke accounts. The firewall ruleset must be centrally managed and cannot be modified by spoke account teams.

Which combination of features achieves this with the least operational overhead?

  • A Deploy Network Firewall in each spoke VPC independently. Use AWS Config to enforce that all rulesets match a central template stored in S3. Spoke teams must request changes via a ticketing system.
  • B Deploy Network Firewall in the central inspection VPC. Share the firewall policy (not the firewall endpoints) with spoke accounts using AWS RAM. Spoke teams can attach their VPCs to the shared policy but cannot edit rules.
  • C Deploy Network Firewall centrally. Use AWS Firewall Manager with a Network Firewall policy scoped to the Organization. Firewall Manager automatically deploys and enforces the policy across all accounts and VPCs without spoke team involvement.
  • D Deploy Network Firewall in the central VPC. Use AWS RAM to share the firewall endpoints with all spoke accounts. Each spoke account routes egress traffic via Transit Gateway to the shared firewall endpoints.
Correct Answer: C
C is correct. AWS Firewall Manager is purpose-built for this exact scenario. It integrates with AWS Organizations, allowing you to define a Firewall Manager security policy (wrapping a Network Firewall policy) at the Organization or OU level. Firewall Manager then automatically deploys the Network Firewall across all in-scope accounts and VPCs, remediates non-compliant resources automatically, and prevents spoke teams from editing the centrally-managed rules. This is the "least operational overhead" answer because there is no manual deployment per account.
Option D is a very attractive distractor — sharing firewall endpoints via RAM is a real capability, but it requires each spoke team to manually configure routing, which adds operational overhead and doesn't prevent rules from being modified at the firewall level. The key constraint was "cannot be modified by spoke teams" + "least operational overhead" — only Firewall Manager delivers both.
Keyword pattern: "Organization-wide enforcement" + "cannot be overridden by member accounts" = AWS Firewall Manager (not RAM, not per-account Config rules). RAM shares RESOURCES. Firewall Manager enforces POLICY. These are different. RAM = sharing. Firewall Manager = governance at scale.

📊 Section 2 — Data Analytics & Streaming

🌊Kinesis · MSK · Redshift· Complexity: ★★★★☆
Q 04
Kinesis Data Streams MSK Redshift

A logistics company ingests 500,000 GPS tracking events per second from IoT sensors on delivery vehicles. The data engineering team has two requirements:

Req 1: Real-time anomaly detection on the raw stream (flagging vehicles that deviate from routes) within 200ms of ingestion using custom ML inference code.

Req 2: The raw stream must also be loaded into Amazon Redshift within 60 seconds of arrival for operational dashboards used by dispatchers.

The existing engineering team is expert in Apache Kafka. Which architecture meets both requirements?

  • A Use Amazon Kinesis Data Streams with Enhanced Fan-Out for low-latency reads. Attach a Lambda function consumer for anomaly detection (Req 1). Use Kinesis Data Firehose to load into Redshift via S3 COPY (Req 2).
  • B Use Amazon MSK (managed Kafka) since the team is Kafka-expert. Deploy a Kafka Streams application on EKS for anomaly detection (Req 1). Use the MSK Redshift connector via Kafka Connect to stream data directly into Redshift within 60 seconds (Req 2).
  • C Use Amazon Kinesis Data Streams. Attach an AWS Glue Streaming job for anomaly detection (Req 1). Use Amazon Redshift Streaming Ingestion with Kinesis as the source (Req 2).
  • D Use Amazon SQS FIFO queues for ordering. Attach a Lambda for anomaly detection. Use a scheduled AWS Glue job every minute to read from SQS and load into Redshift.
Correct Answer: C
C is correct — and it's the most direct path using native AWS-managed services:
  • Kinesis Data Streams handles 500K events/sec with horizontal sharding.
  • Glue Streaming runs continuously against the Kinesis stream for anomaly detection within the 200ms window (Glue Streaming uses micro-batches that can be tuned to sub-second).
  • Redshift Streaming Ingestion is a native Redshift feature that reads directly from Kinesis Data Streams via a materialized view with near real-time latency well under 60 seconds — no S3 staging required.
Option B is the second-best answer — MSK + Kafka Connect to Redshift is technically valid, and the scenario mentions Kafka expertise. BUT the exam favours managed serverless when available, and MSK Redshift connector (Kafka Connect) has more operational overhead than Redshift Streaming Ingestion. Option A is wrong: Kinesis Firehose to Redshift via S3 COPY has a minimum buffer time — the fastest Firehose S3 interval is 60 seconds + S3 COPY time, which likely misses the 60-second SLA. Option D: SQS is NOT a streaming service — it cannot handle 500K events/sec at this volume without extreme cost and FIFO queues are limited to 3000 msg/sec.
Key decode: "existing Kafka expertise" is a red herring when the question asks for the SAA-recommended architecture. The exam tests AWS-native solutions. When you see "load into Redshift within seconds" + Kinesis as source → Redshift Streaming Ingestion (not Firehose → S3 → COPY which is slower). When you see "Kinesis + real-time ML" → think Lambda or Glue Streaming, NOT Firehose (Firehose = delivery/batch, not processing).
📐 Kinesis → Glue Streaming + Redshift Streaming Ingestion
🗄️Athena · Glue · Lake Formation· Complexity: ★★★★★
Q 05
Athena Glue Lake Formation

An enterprise data lake stores raw, curated, and consumption-layer data in S3 using the AWS Glue Data Catalog. Different business units require column-level access control — the HR department should be able to query the employees table but must NOT see the salary or ssn columns. The Finance department should see all columns. Analysts use Amazon Athena to query the lake.

What is the correct implementation using AWS-native services with the least custom code?

  • A Create two separate S3 prefixes — one with salary/ssn redacted for HR and one complete for Finance. Attach IAM policies to each group restricting S3 prefix access. Use Athena named queries scoped to each prefix.
  • B Use AWS Lake Formation column-level permissions. Register the S3 location with Lake Formation. Grant the HR IAM role SELECT permission on the employees table but exclude the salary and ssn columns. Grant Finance SELECT on all columns. Athena queries enforce these permissions automatically via the Glue Data Catalog.
  • C Use AWS Glue DataBrew to create two job recipes — one that masks salary/ssn for HR output and one that passes all columns for Finance. Store outputs in separate S3 locations and create separate Athena workgroups per team.
  • D Use Athena workgroup query result encryption and set a per-column KMS key. HR's workgroup uses a key that cannot decrypt salary/ssn columns; Finance uses a master key. Configure Glue Data Catalog column metadata to reference the appropriate key.
Correct Answer: B
B is correct. AWS Lake Formation natively supports column-level security (and row-level filtering). When Lake Formation is integrated with the Glue Data Catalog, you can grant/revoke permissions on specific columns of a table to specific IAM principals. When Athena queries that table, Lake Formation enforces the column restrictions transparently — no custom code, no duplicate data, no separate S3 prefixes required. This is the canonical AWS design for fine-grained data lake access control.
Option A is the old pre-Lake Formation pattern — creating duplicate data with redacted columns. This wastes storage, creates consistency problems, and doesn't scale to 100 columns or 50 teams. Option D is fiction — Athena workgroups use KMS for encrypting query results, NOT for column-level access control of source data. Per-column KMS is not a real Athena feature. Option C (DataBrew) is a data preparation tool, not an access control mechanism.
Pattern: "column-level" OR "row-level" access control on a data lake = AWS Lake Formation (not IAM S3 policies, not Glue, not Athena workgroups alone). Lake Formation is the governance layer that sits on top of the Glue Data Catalog. Athena and EMR query through Lake Formation, which enforces fine-grained permissions. If you see "data lake" + "fine-grained access" → Lake Formation is always in the answer.

📈QuickSight · OpenSearch · Redshift· Complexity: ★★★☆☆
Q 06
QuickSight OpenSearch Redshift

A retail company needs to build a customer-facing search experience where shoppers can type a free-text query like "red running shoes size 10" and see relevant results with typo tolerance and relevance ranking. Separately, the BI team wants to run complex SQL aggregations on the same product catalogue — grouping by category, calculating revenue per SKU, and producing interactive dashboards accessible in a web browser without any server management.

Which pair of services is the right fit?

  • A Amazon OpenSearch Service for customer search (full-text, fuzzy, relevance-ranked) · Amazon QuickSight for the BI team dashboards (serverless, SQL on connected data sources including Redshift)
  • B Amazon Redshift for customer search (SQL LIKE queries) · Amazon QuickSight for BI dashboards
  • C Amazon OpenSearch Service for both search AND BI dashboards using OpenSearch Dashboards (Kibana)
  • D Amazon Kendra for customer search (intelligent enterprise search) · Amazon Redshift for SQL aggregations with custom front-end dashboards built on EC2
Correct Answer: A
A is correct. OpenSearch Service is the purpose-built full-text search engine — it natively supports fuzzy matching, relevance scoring, faceted search, and handles unstructured text queries like "red running shoes." QuickSight is AWS's serverless BI service — it connects to Redshift, S3, Athena, RDS etc., renders interactive dashboards in the browser with no server management required. This is the textbook SAA pairing.
Option B: Redshift uses SQL — running full-text product search via SQL LIKE queries is not fuzzy, not relevance-ranked, and will be extremely slow. Redshift is for analytical SQL, not search. Option C: OpenSearch Dashboards (formerly Kibana) is for operational analytics of OpenSearch data — it's a log/metric exploration tool, NOT a governed BI dashboard tool for business users. Option D: Kendra is an intelligent enterprise search service for internal document search (SharePoint, wikis, etc.) — not product catalogue search.
Quick service disambiguation — search type matters: "Free-text, fuzzy, product search" → OpenSearch Service. "Enterprise document search, Q&A on PDFs/wikis" → Amazon Kendra. "SQL aggregation BI dashboards" → QuickSight (+ Redshift as source). Never use Redshift for full-text search. Never use OpenSearch as a BI dashboard for business users.
⚡EMR · Glue · Data Pipeline· Complexity: ★★★★☆
Q 07
EMR Glue Data Pipeline

A data engineering team needs to run Apache Spark jobs that process 500TB of raw clickstream data stored in S3. The jobs run nightly and take 6–8 hours. The team wants to use custom Spark configuration tuning (executor memory, parallelism, shuffle partitions) and requires access to the Spark History Server UI for debugging. Cost is a priority and the infrastructure should terminate after each job run.

Which service is the most appropriate choice?

  • A AWS Glue ETL jobs — Glue runs Apache Spark under the hood, auto-scales workers, and terminates automatically after job completion. No server management required.
  • B AWS Data Pipeline — schedule the data transformation pipeline nightly. It handles dependencies, retries, and terminates EC2 resources after each run.
  • C Amazon EMR with transient clusters using Spot Instances — launch on a schedule (EventBridge Scheduler), apply custom spark-defaults.conf, access Spark History Server via EMR persistent application UI, and terminate cluster after job completion to minimize cost.
  • D Amazon EMR Serverless — submit Spark jobs without managing clusters. Automatically scales and terminates workers. No access to Spark configuration files or History Server UI.
Correct Answer: C
C is correct. The key constraints are: (1) custom Spark configuration tuning — this requires direct access to Spark config, only available on full EMR clusters; (2) Spark History Server — only accessible with full EMR cluster; (3) cost-optimized nightly run → transient clusters with Spot Instances on a schedule. Transient EMR clusters launch, run, and self-terminate — you only pay for the 6–8 hour window, and Spot Instances reduce EC2 cost by 60–90%.
Option A (Glue): Glue runs Spark but hides the Spark engine — you cannot tune spark-defaults.conf directly, and there's no native Spark History Server. Glue is the right answer when you want NO cluster management. Option D (EMR Serverless): Same problem — no direct Spark configuration tuning and limited access to History Server. Option B (Data Pipeline): AWS Data Pipeline is a legacy orchestration service, largely superseded by AWS Glue workflows and Step Functions. It also doesn't natively optimize for Spark performance at 500TB scale.
Decision tree for Spark on AWS: Do you need custom Spark tuning OR Spark History Server → EMR (full cluster). Do you want fully managed, no config, serverless → Glue ETL or EMR Serverless. Spot Instances on EMR = cost optimization for batch jobs. The phrase "custom configuration" is always the signal that you need the full EMR cluster experience.
📦Data Exchange · MSK · Kinesis· Complexity: ★★★★☆
Q 08
Data Exchange MSK Kinesis

A financial data company wants to monetize its proprietary market sentiment datasets by selling access to hedge funds and banks. The data is updated daily and needs to be delivered as S3 files with governed entitlements (only paying subscribers can access). The company also wants to discover and subscribe to third-party weather data from an external provider to enrich its models.

SELECT TWO that together meet both requirements Which two AWS services best address both use cases?

  • A Publish the sentiment dataset to AWS Data Exchange as a data product, setting subscription rules and entitlements. Subscribers access data via governed S3 exports.
  • B Subscribe to the weather data provider's published dataset on AWS Data Exchange. AWS manages the data delivery to S3 automatically when updates are published.
  • C Use Amazon MSK to stream the sentiment data via Kafka topics, granting each subscriber an MSK IAM role with access to specific topic partitions.
  • D Build a custom API Gateway + S3 Pre-signed URL system where subscribers call the API to receive time-limited links to the dataset files.
Correct Answers: A and B
A + B are correct. AWS Data Exchange is purpose-built for this exact two-sided marketplace: data providers publish datasets and manage entitlements; data subscribers discover and subscribe via the AWS Marketplace catalogue. For selling (Requirement 1), option A is correct — the company publishes its dataset as a product on Data Exchange. For buying (Requirement 2), option B is correct — they subscribe to the weather provider on Data Exchange, and delivery to S3 is automated. Both sides of the Data Exchange marketplace are addressed.
Option C (MSK) is over-engineered — MSK is for real-time event streaming applications with internal consumers. Distributing B2B data to external subscribers via Kafka topic ACLs is operationally complex and NOT what MSK is designed for. Option D (custom API Gateway) is the pre-AWS Data Exchange pattern — it works but adds significant undifferentiated engineering effort when Data Exchange exists specifically to eliminate that work.
AWS Data Exchange is often a "new service" trap — students don't know it and default to custom solutions. The exam keywords: "monetize data", "sell datasets to third parties", "subscribe to third-party data", "data marketplace" → AWS Data Exchange. It's essentially AWS Marketplace but specifically for data products (files, APIs, Redshift shares).

🚚 Section 3 — Migration, Transfer & Discovery

🔍Application Discovery · Migration Hub· Complexity: ★★★☆☆
Q 09
Application Discovery Migration Hub MGN

A company wants to migrate 400 on-premises servers to AWS. Before migrating, they need to:

Phase 1: Automatically discover all running processes, network dependencies, and resource utilization across all 400 servers — without manually installing agents on Windows servers (the ops team is unavailable).

Phase 2: After migrating a subset, track migration progress of all servers across multiple AWS migration tools (Database Migration Service, Application Migration Service) in a single consolidated view.

Which combination of features satisfies both phases?

  • A Phase 1 → AWS Application Discovery Service Agentless Collector (vCenter-based) for VMware environments | Phase 2 → AWS Migration Hub for centralized tracking
  • B Phase 1 → Install AWS Discovery Agent on each of 400 servers | Phase 2 → AWS Migration Hub
  • C Phase 1 → AWS Systems Manager Inventory to collect server data | Phase 2 → AWS Config to track EC2 deployment progress
  • D Phase 1 → AWS Application Discovery Service Agentless Collector | Phase 2 → AWS CloudTrail for audit trail of migration API calls
Correct Answer: A
A is correct. AWS Application Discovery Service offers two modes: (1) Discovery Agent — install on each server, works on bare metal/VMs, requires access to each box; (2) Agentless Collector — integrates with VMware vCenter, discovers VMs without touching individual servers. Since the ops team cannot install agents on 400 Windows servers, the agentless collector (via vCenter) is the practical path. For Phase 2, AWS Migration Hub is explicitly designed to aggregate and track migration status across DMS, MGN, and other migration tools in one dashboard.
Option B technically works for Phase 1 but violates the constraint "without manually installing agents." The question says ops team is unavailable — the agentless path via vCenter is the right choice. Option C: Systems Manager Inventory can collect data, but it requires SSM Agent on servers — which is also an agent install. AWS Config tracks configuration changes, not migration workflow progress. Option D: CloudTrail logs API calls for compliance/audit — it is not a migration tracking dashboard.
Application Discovery Service decision: "Can you install agents?" → YES = Discovery Agent (deeper data: processes, network connections). NO = Agentless Collector (VMware vCenter required, less granular). Migration Hub keyword: "single dashboard across multiple migration tools" → always Migration Hub. It doesn't migrate anything itself — it just aggregates tracking.
❄️Snow Family · DataSync · Transfer Family· Complexity: ★★★★★
Q 10
Snow Family DataSync Transfer Family

A media production company has three distinct data movement scenarios:

Scenario X: 2PB of archive footage stored on on-premises NAS. The company has a 1Gbps internet link. IT estimates online transfer would take over 6 months. They need the data in S3 within 45 days.

Scenario Y: Nightly sync of 500GB of edited video files from on-premises NFS servers to Amazon S3, with bandwidth throttling to not saturate the production network, and automatic retry on failures.

Scenario Z: External post-production partners upload final deliverables (MP4, ProRes) using their existing SFTP clients directly into S3 without any code changes on the partner side.

Match the correct AWS service for each scenario.

  • A X → AWS Snowball Edge Storage Optimized (multiple devices) | Y → AWS DataSync | Z → AWS Transfer Family (SFTP endpoint backed by S3)
  • B X → AWS DataSync with dedicated 10Gbps Direct Connect | Y → AWS Storage Gateway | Z → Amazon S3 Transfer Acceleration with presigned URLs
  • C X → AWS Snowmobile | Y → AWS DataSync | Z → AWS Transfer Family (SFTP)
  • D X → AWS Snowball Edge | Y → AWS Transfer Family (FTP) | Z → S3 multipart upload API
Correct Answer: A
A is correct — exact reasoning per scenario:
  • Scenario X → Snowball Edge: At 1Gbps, 2PB would take ~185 days online. Multiple Snowball Edge Storage Optimized devices (80TB usable each) can move 2PB in ~45 days with parallel shipping. Snowmobile (option C) handles exabyte-scale (≥10PB); 2PB is Snowball territory.
  • Scenario Y → DataSync: AWS DataSync is purpose-built for recurring, automated data transfers from NFS/SMB to S3/EFS/FSx. It has built-in bandwidth throttling, scheduling, automatic retry, and delta sync. It uses a DataSync agent deployed on-prem.
  • Scenario Z → Transfer Family: AWS Transfer Family provides managed SFTP/FTPS/FTP/AS2 endpoints backed by S3 or EFS. External partners use their existing SFTP clients with zero code changes — exactly what the scenario requires.
Snowmobile confusion: Snowmobile is for petabyte to exabyte (typically ≥10PB) migration — it's a literal 45-foot shipping container. 2PB is achievable with multiple Snowball Edge devices, which is logistically simpler. Option B adds Direct Connect which doesn't solve the 6-month timeline — DC increases bandwidth but you'd need ~10Gbps sustained to hit 45 days for 2PB, and provisioning DC takes months itself.
Snow Family sizing guide: <10TB = Snowcone | 10TB–80TB (one device) or up to multiple PB = Snowball Edge | ≥10PB (exabyte scale) = Snowmobile. DataSync vs Storage Gateway: DataSync = pure data migration/sync (file transfers). Storage Gateway = hybrid cloud storage (ongoing application integration). Transfer Family keyword trigger: "SFTP/FTPS/FTP clients" + "no code changes on partner side" → Transfer Family, every time.

🚀Application Migration Service (MGN)· Complexity: ★★★★☆
Q 11
MGN Migration Hub Lift & Shift

A company is performing a lift-and-shift migration of 200 on-premises physical servers (running a mix of Windows Server 2016 and RHEL 8) to Amazon EC2. The key requirements are:

• Zero downtime — production servers must not be interrupted during replication
• Continuous block-level replication of data until the cutover window
• The ability to do a test cutover in a staging VPC before the production cutover
• Automatic conversion of physical server boot configurations to EC2-compatible format

Which AWS service meets all four requirements?

  • A AWS Database Migration Service (DMS) — continuously replicates server data, supports Windows and Linux, and converts boot configurations.
  • B AWS DataSync — schedules continuous file-level sync from on-premises servers to EC2 EBS volumes, then performs a final sync at cutover.
  • C AWS Application Migration Service (MGN) — installs a lightweight replication agent that performs continuous block-level replication to a staging area. Supports test cutovers in a test VPC, automatic boot configuration conversion, and zero-downtime production cutovers.
  • D AWS Elastic Disaster Recovery (DRS) — designed for on-premises to AWS replication with continuous block-level sync and point-in-time recovery, with test drills supported.
Correct Answer: C
C is correct. AWS Application Migration Service (MGN) is the official AWS recommended service for lift-and-shift migrations. It: (1) installs a lightweight agent that performs continuous block-level replication without interrupting production; (2) supports test cutovers in an isolated VPC; (3) automatically converts physical-to-virtual boot configurations (BIOS/UEFI to EC2 Nitro); (4) is OS-agnostic across Windows and Linux versions.
Option A (DMS): DMS migrates DATABASES — it moves database schemas and data between database engines (Oracle → Aurora, etc.). It is NOT a server migration service. Option B (DataSync): DataSync is for file-level transfers (NFS/SMB to S3/EFS). It does NOT perform block-level replication, cannot handle boot volumes, and is not designed for server cutovers. Option D (DRS): Elastic Disaster Recovery is for ongoing DR of applications that are ALREADY running on AWS or on-premises — it's for recovery, not initial migration. Though technically similar to MGN under the hood, the exam will never select DRS for a migration use case.
DMS vs MGN: The most common confusion on the exam. DMS = Database Migration (data in databases). MGN = Server Migration (entire servers, OS included). If you see "migrate Oracle to Aurora" → DMS. If you see "migrate 200 servers to EC2" → MGN. Elastic DRS keyword: "disaster recovery" or "RPO/RTO" or "failback" → DRS. "Migration" + "lift and shift" + "servers" → MGN every time.
🔐KMS · CloudHSM · Secrets Manager· Complexity: ★★★★★
Q 12
KMS CloudHSM Compliance

A government contractor must comply with a regulation that requires:

Requirement 1: Cryptographic keys must be stored in hardware that is exclusively controlled by the customer — AWS must have no ability to ever access the key material.
Requirement 2: The hardware must be FIPS 140-2 Level 3 validated.
Requirement 3: The solution must integrate with a custom C++ application running on EC2 using the PKCS#11 standard for cryptographic operations.

Which service satisfies all three requirements?

  • A AWS KMS with Customer Managed Keys (CMK) and key material imported by the customer. AWS KMS hardware is FIPS 140-2 Level 2 validated. The KMS API is used for all cryptographic operations.
  • B AWS KMS with Custom Key Store (CloudHSM-backed). The customer controls the CloudHSM cluster, FIPS 140-2 Level 3 validated, but cryptographic operations go through the KMS API — not PKCS#11 directly.
  • C AWS CloudHSM — customer-controlled dedicated HSM hardware, FIPS 140-2 Level 3 validated, supports PKCS#11, JCE, and CNG libraries directly. AWS has no access to keys; they're managed entirely by the customer.
  • D AWS Secrets Manager with KMS integration — secrets are encrypted with a CMK and rotated automatically. FIPS 140-2 Level 3 hardware is used for KMS backing hardware in GovCloud.
Correct Answer: C
C is correct. AWS CloudHSM is the only AWS service where: (1) you have exclusive control of the key material — AWS literally cannot access your keys; (2) uses hardware validated at FIPS 140-2 Level 3 (KMS standard is Level 2); (3) exposes PKCS#11, JCE (Java), and Microsoft CNG libraries directly to your application.
Option A (KMS): Standard KMS is FIPS 140-2 Level 2, not Level 3. Also, AWS can theoretically access KMS operations under legal compulsion — you don't have "exclusive" hardware control. Option B (KMS with Custom Key Store): Technically Level 3 via CloudHSM backing, but the PKCS#11 interface is NOT exposed to your application — only KMS API is available. The requirement to use PKCS#11 from a C++ application eliminates this option. Option D: Secrets Manager stores secrets but is not an HSM — it doesn't expose PKCS#11 and doesn't provide Level 3 hardware regardless of region.
KMS vs CloudHSM decision matrix: Use KMS when: AWS-managed convenience, Level 2 compliance is sufficient, using AWS-native integrations (S3, RDS, EBS). Use CloudHSM when: FIPS Level 3 required, exclusive key control required (no AWS access), PKCS#11 / JCE / CNG required for custom application. The phrase "exclusive control" or "no AWS access to keys" is the definitive CloudHSM signal.

Q 13
IAM RAM Cross-account

A company has two AWS accounts — Account A (Producer) and Account B (Consumer). They need to share an Amazon VPC subnet so that EC2 instances in Account B can launch into the shared subnet managed in Account A's VPC.

Additionally, an application in Account B's EC2 must be able to read objects from an S3 bucket in Account A without the objects leaving the AWS network and without using S3 public endpoints.

Which two mechanisms solve each requirement with the least complexity?

  • A VPC Subnet Share → Use AWS Resource Access Manager (RAM) to share the subnet from Account A to Account B | S3 Private Access → Use a VPC Endpoint (Gateway type) for S3 in the shared VPC, and update Account A's S3 bucket policy to allow Account B's IAM role
  • B VPC Subnet Share → Set up VPC Peering between Account A and Account B | S3 Private Access → Use a NAT Gateway in Account A with routes from Account B
  • C VPC Subnet Share → Create a Transit Gateway and attach both VPCs | S3 Private Access → Use S3 Transfer Acceleration to minimize data transfer across regions
  • D VPC Subnet Share → RAM subnet sharing | S3 Private Access → VPC Peering between the EC2's VPC and S3's VPC (S3 doesn't have a VPC)
Correct Answer: A
A is correct on both counts:
  • RAM Subnet Sharing: AWS Resource Access Manager (RAM) can share VPC subnets from a VPC owner account to participant accounts within the same Organization. Account B can then launch EC2 into Account A's shared subnet — no VPC Peering needed because they're literally in the same VPC.
  • S3 VPC Endpoint: An S3 Gateway Endpoint in the shared VPC routes S3 traffic through AWS's private network (not the public internet). Account A's S3 bucket policy is updated to include Account B's IAM role as an allowed principal. The EC2 in Account B (running in Account A's shared subnet) reaches S3 privately via the endpoint.
Option B: VPC Peering creates a private connection between two separate VPCs — you can't "share" a subnet via peering. EC2 in Account B would still be in Account B's VPC. Also NAT Gateway doesn't solve the "without leaving AWS network" requirement. Option D: This is a trick — the second half is nonsense. S3 is a global service, not in a VPC. You cannot peer with S3's "VPC" because there isn't one. The correct private access method is a VPC Gateway Endpoint for S3.
RAM vs VPC Peering vs TGW — the decision: Share a subnet so another account's EC2 can run IN your VPC → RAM subnet sharing (same VPC, simplest). Connect two separate VPCs (same or different accounts) with non-overlapping CIDRs → VPC Peering. Connect many VPCs/on-prem in hub-and-spoke → Transit Gateway. S3 private access from EC2: Gateway Endpoint (free, no data charges) → S3 traffic stays on AWS backbone.
🔄Glue · EMR · Redshift · OpenSearch· Complexity: ★★★★★
Q 14
Glue Redshift OpenSearch Lake Formation Architecture

A large e-commerce company is designing an end-to-end analytics platform. Their requirements are:

• Raw order, clickstream, and product data lands in S3 (data lake).
• Data must be catalogued and transformed with no server management.
• The BI team runs complex multi-table SQL joins on TB-scale datasets for weekly business reports.
• The customer service team needs to search customer records using free-text name/email search with sub-second response.
• All data access must be governed with table and column-level permissions per team.

SELECT THREE Which combination of services forms the correct architecture?

  • A AWS Glue (ETL + Data Catalog) — serverless cataloguing and transformation of raw S3 data, no server management required
  • B Amazon Redshift — columnar MPP warehouse for complex SQL analytics by BI team on TB-scale data
  • C Amazon OpenSearch Service — free-text search for customer service records with sub-second response
  • D Amazon EMR — distributed Spark processing for the BI team's weekly reports (no server management option selected)
Correct Answers: A, B, and C
A + B + C are correct:
  • A — Glue: Serverless ETL + Glue Data Catalog + Glue Crawlers address the "no server management" cataloguing and transformation requirement. Glue also integrates natively with Lake Formation for governance.
  • B — Redshift: Columnar MPP database optimized for complex multi-table SQL joins across TB-scale datasets. This is the canonical analytics warehouse for BI workloads.
  • C — OpenSearch: Customer record free-text search (name, email) with sub-second response — this is OpenSearch's primary use case.
Why not D (EMR)? EMR requires cluster management even with managed scaling — it is NOT "no server management." The BI team needs SQL joins on TB-scale → Redshift is the correct service. EMR would be correct if the requirement was custom Spark code or petabyte-scale processing with cluster control. Also, weekly batch BI reports don't justify a continuously-running EMR cluster.

Lake Formation (implied): The governance requirement (table/column-level permissions) would be implemented via Lake Formation overlaid on the Glue Data Catalog — it wasn't offered as an option but would be part of the complete architecture.
D is the classic "sounds right" distractor. EMR can do what Redshift does but with far more operational overhead. The question says "no server management" — this eliminates EMR. The phrase "complex SQL joins" combined with "no server management" = Redshift. EMR is for when you need Hadoop ecosystem tools (Spark, Hive, HBase) with custom configuration control.
Analytics service quick selector — commit this to memory: Serverless ETL → Glue. Complex SQL BI → Redshift. Full-text search → OpenSearch. Ad-hoc SQL on S3 → Athena. Custom Spark with control → EMR. Stream processing → Kinesis Data Analytics / Glue Streaming. BI dashboards → QuickSight. Data governance → Lake Formation. If a question names "no server management" AND "SQL joins at scale" → it's Redshift, not EMR. If it says "petabyte" AND "custom Spark" → EMR.
